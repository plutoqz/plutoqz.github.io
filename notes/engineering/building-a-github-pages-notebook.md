# 把 GitHub Pages 变成可持续更新的学习笔记站

很多个人主页的问题不在于“有没有搭起来”，而在于搭完之后没有持续写下去。  
如果主页只有简介和项目链接，它很容易变成一次性装修的门面；但当它开始接住 Markdown 笔记，站点就会逐渐变成持续更新的知识容器。

## 为什么笔记系统值得和主页放在一起

项目和笔记天然互相补足。

- 项目回答“你做过什么”
- 笔记回答“你是怎么理解这些问题的”
- 两者放在一起，访客才能快速判断你的技术兴趣和思考方式

对个人站点来说，笔记比博客更容易坚持，因为它不要求每一篇都写成完整文章。

## 结构设计的关键点

把笔记系统做成静态站时，我更在意下面三个点：

### 1. 内容来源要简单

最好只维护 Markdown 文件，不再额外维护一份文章列表。  
一旦需要手工同步目录，写作流就会变重。

### 2. 分类要来自目录

当 `notes/webgis/xxx.md` 这样的结构天然存在时，分类信息就不应该再重复维护。  
目录本身就是信息架构的一部分。

### 3. 单篇文章要有目录

学习笔记通常不是短帖，而是带层次的长文。  
如果文章侧边没有目录，读者会很难判断篇章结构。

## 自动索引为什么重要

自动索引可以把“上传 Markdown”和“站点出现文章”之间的中间环节压缩到最少。

一个可接受的流程应该是：

1. 把文件放进 `notes/分类/文件名.md`
2. 提交并推送到 GitHub
3. 站点自动更新笔记清单

只要做到这一步，主页和技术博客之间的边界就会开始变薄。

## 我会怎么继续迭代

后续可以继续加三类能力：

- 全文搜索
- 标签页
- 系列文章之间的上下篇跳转

但第一阶段最重要的，仍然是把上传内容和发布展示的链路打通。
