# 评测、可靠性、安全与成本

## 本章目标

-   学会从“能跑”走向“可上线”的评价方式。
    
-   掌握 Agent 在可靠性、安全性和成本上的主要风险。
    
-   建立一套面向生产的 guardrail 与 eval 组合思路。
    

## 核心问题

-   Agent 应该怎么测，测什么？
    
-   为什么安全问题在 Agent 里比普通问答更尖锐？
    
-   如何在能力、成本、时延、风险之间做平衡？
    

## 正文

### 1\. 为什么 Agent 比普通 LLM 应用更难评测

因为它不是一个静态输出问题，而是一个“路径问题”：

-   最终答案对不对？
    
-   中间路径合不合理？
    
-   工具调得对不对？
    
-   有没有多余动作？
    
-   有没有越权或危险行为？
    
-   成本和时延是否可接受？
    

OpenAI 的 Agent evals 文档明确提到，除了最终效果，还推荐用 trace grading 去识别 workflow 级错误。这一点非常关键，因为 Agent 常见失败不是“最后一句话说错”，而是“前面走错了路”。

### 2\. 评测至少要看四层

#### 2.1 结果层

最终任务完成了吗？

例子：

-   是否正确回答问题
    
-   是否成功修复 bug
    
-   是否正确创建工单
    

#### 2.2 路径层

中间步骤是否合理？

例子：

-   是否选了正确工具
    
-   是否少走弯路
    
-   是否进入无意义循环
    

#### 2.3 安全层

有没有危险行为？

例子：

-   是否泄露敏感数据
    
-   是否响应越权请求
    
-   是否被 prompt injection 诱导
    

#### 2.4 运营层

系统代价是否可接受？

例子：

-   平均轮数
    
-   工具调用次数
    
-   token 成本
    
-   P95 延迟
    

### 3\. 一套实用的 eval 方法

#### 3.1 固定样本集

准备一批代表性任务：

-   正常样本
    
-   边界样本
    
-   错误输入
    
-   工具失败样本
    
-   安全攻击样本
    

#### 3.2 回放式评测

记录真实 trace，对比版本升级前后的行为变化。

#### 3.3 结构化打分

不要只问“好不好”。要拆成指标：

-   `task_success`
    
-   `tool_selection_accuracy`
    
-   `unsafe_action_rate`
    
-   `mean_turns`
    
-   `cost_per_run`
    

#### 3.4 人工评审

高风险场景不能完全自动评分，尤其是：

-   法务
    
-   金融
    
-   医疗
    
-   客服赔付
    

### 4\. 可靠性：最常见的四类故障

#### 4.1 工具层故障

-   参数错误
    
-   接口超时
    
-   权限不足
    
-   外部系统返回异常
    

#### 4.2 决策层故障

-   选错工具
    
-   漏掉关键步骤
    
-   过度规划
    
-   错把中间结果当最终结论
    

#### 4.3 状态层故障

-   重试导致重复副作用
    
-   中断恢复后状态不一致
    
-   多 Agent 之间共享状态错乱
    

#### 4.4 交互层故障

-   对用户意图理解偏差
    
-   守护规则过严导致可用性差
    
-   守护规则过松导致风险外溢
    

### 5\. Prompt Injection：为什么 Agent 更危险

OpenAI 的安全指南把 prompt injection 列为 Agent 的核心风险之一。原因在于：

-   Agent 会读取不可信文本
    
-   Agent 可能基于这些文本调用真实工具
    
-   工具结果和外部文本又会回流进后续决策
    

于是攻击者不只是影响一段回答，而是可能影响：

-   数据泄露
    
-   越权动作
    
-   错误工作流跳转
    
-   误导 Agent 操作敏感系统
    

### 6\. 几条非常重要的安全原则

#### 6.1 不可信输入不要塞进高优先级指令位

OpenAI 安全指南明确提醒：不要把不可信变量直接注入 developer messages。因为高优先级提示一旦被污染，攻击者几乎是站在“系统层”发号施令。

#### 6.2 用结构化输出约束数据流

官方指南明确建议：在节点之间用枚举、固定 schema、必填字段等结构化输出，减少自由文本在节点间传播，从而降低注入与数据走私风险。

#### 6.3 高风险 MCP/工具调用要保留审批

文档明确建议对 MCP 工具保持审批开启，让用户或人工确认读写操作。这个思路可以推广到所有高风险 action tool。

#### 6.4 Guardrails 是多层组合，不是单点魔法

OpenAI 公开指南强调 guardrails 应和认证、授权、访问控制、常规软件安全措施一起使用。也就是说：

-   moderation 不能替代 RBAC
    
-   relevance classifier 不能替代审批
    
-   prompt rule 不能替代沙箱
    

### 7\. 成本与时延的本质

Agent 的总成本通常来自：

-   模型调用轮数
    
-   单轮输入输出 token
    
-   工具调用成本
    
-   并行 worker 数量
    
-   重试与回滚成本
    

Agent 的总时延通常近似为：

`总时延 = 多轮模型耗时 + 多次工具耗时 + 重试耗时 + 守护/评测耗时`

所以当你优化时，不要只盯模型推理速度，还要看：

-   是否减少不必要轮数
    
-   是否合并工具请求
    
-   是否把只读检索并行化
    
-   是否用更强 schema 降低重试
    

### 8\. 一个面向生产的防线分层

| 层级  | 典型措施 |
| --- | --- |
| 输入层 | 敏感信息脱敏、恶意输入检测、长度限制 |
| 决策层 | 明确 instructions、结构化输出、最大轮数限制 |
| 工具层 | schema 校验、权限校验、审批、幂等键 |
| 状态层 | checkpoint、审计日志、恢复策略 |
| 输出层 | 格式校验、敏感信息过滤、结果复核 |
| 运营层 | eval、trace grading、告警、人工兜底 |

### 9\. 何时必须引入人工

下面几类动作建议默认人工确认或至少可人工打断：

-   大额支付和退款
    
-   删除数据
    
-   改写核心业务记录
    
-   对外承诺、合规结论、法律表述
    
-   高风险系统配置变更
    

公开指南也强调：敏感、不可逆、高风险动作在系统还没充分验证可靠性前，应触发人工监督。

## 工程细节与取舍

### 1\. “更安全”常常意味着“更慢”

因为更多校验、审批和规则会引入额外步骤。真正要做的是按风险分级，而不是一刀切。

### 2\. 结果正确不代表路径正确

一个 Agent 可能答对了，但中间进行了越权查询或调用了不必要的危险工具。上线时不能只看最终正确率。

### 3\. 不要把所有问题都推给模型

大量可靠性问题更适合通过：

-   更清晰工具
    
-   更强 schema
    
-   更稳状态机
    
-   更严格权限模型
    

来解决，而不是反复改 prompt。

## 常见误区

### 误区 1：做几个 benchmark 就算评测完成

不对。生产 Agent 的问题很多发生在工具、路径、副作用和安全边界，而不是静态问答题。

### 误区 2：Guardrail 开得越多越好

不对。过严会严重影响可用性，导致系统几乎什么都不敢做。

### 误区 3：只要输出没泄密就安全

不对。Agent 可能在工具调用链路里就已经把敏感数据暴露出去了。

## 面试与实战问答

### 问：Agent 应该怎么评测？

短答： 同时看结果、路径、安全和运营指标。

深答： 至少包含固定样本集、回放式 trace、结构化评分和高风险场景的人审。

### 问：为什么 prompt injection 在 Agent 中更危险？

短答： 因为 Agent 不只是回答，它还可能调用工具并执行动作。

深答： 一旦不可信文本影响后续工具调用，攻击面就从“错误回答”升级成“错误执行”。

### 问：如何提升 Agent 可靠性？

短答： 减少自由度，增加结构化、显式状态、幂等、回放和评测。

深答： 可靠性不是靠模型“更聪明”，而是靠运行时和治理层把不确定性包住。
