# 性能优化、高并发与弹性伸缩

## 本章目标

- 回答 Agent 系统为什么慢、哪里慢、怎么拆、怎么扩、怎么降。
- 建立从串行链路优化到高并发治理的完整思路。
- 能把“慢”说成可量化、可治理的分层问题。

## 关键问题

- 串行工具链为什么会拖垮 Agent 体验？
- 高并发场景下，伸缩和限流应该放在哪里？
- 降级与熔断在 Agent 系统里怎么设计才合理？

## Q9：Agent 推理链路包含多个串行工具，响应延迟高，该如何优化？

### 一句话回答

先做链路拆解，区分 `模型思考时间、检索时间、工具时间、排队时间、重试时间`，然后通过并行化、预取、缓存、少轮次规划和模型路由来减少串行瀑布。

### 详细展开

优化顺序通常是：

1. 画出 waterfall，找最大延迟贡献者
2. 把无依赖工具并行化
3. 对可预测查询做预取
4. 缩短上下文，减少模型耗时
5. 减少不必要的反思轮次
6. 对低价值步骤换轻量模型或规则

典型例子：

- 先查订单再查物流再查退款规则，如果三者彼此独立，就不该串行
- 如果用户一上来就给了订单号，可以提前并行拉订单和物流
- 如果检索是稳定 FAQ，可直接命中缓存或语义缓存

### 落地要点

- 优化前先量化：P50、P95、P99 分别慢在哪。
- 对工具链采用 DAG，而不是默认链表式执行。
- 对强依赖链路，要考虑提前终止和部分回答。

## Q10：高并发场景下，如何设计 Agent 服务的弹性伸缩策略？

### 一句话回答

Agent 服务应按层伸缩：入口网关、编排器、工具网关、检索层、模型代理层分别扩缩容，并以 `QPS、队列长度、并发数、token 吞吐、P95 延迟` 做联合驱动。

### 详细展开

推荐拆层：

- `API Gateway`：按请求数和连接数扩
- `Runner / Orchestrator`：按活跃 run 数和等待队列扩
- `Retrieval / Tool layer`：按外部依赖负载扩
- `Model proxy`：按 token/s、并发会话数、供应商限额扩

高并发下要注意：

- 状态必须外部化，否则扩容后会话漂移
- worker 尽量无状态，便于水平扩展
- 预热模型连接、HTTP 连接池和向量库连接池

### 落地要点

- 不要只按 CPU 扩容，Agent 很多时候卡的是队列和外部依赖。
- 为不同业务线配置不同伸缩和优先级策略。
- 对模型供应商限额要单独建流控层，不然本地扩再多也没用。

## Q31：多工具编排时如何做任务拆解、并行化与结果聚合？

### 一句话回答

把任务拆成有依赖关系的 DAG，能并行的并行，必须串行的串行，最后用结构化聚合器汇总结果，而不是让模型自由拼接一堆子结果。

### 详细展开

建议步骤：

1. 先识别依赖关系
2. 把互不依赖的工具分成 fan-out 阶段
3. 每个子任务返回统一结构
4. 在 fan-in 节点做合并、去重、冲突解决

适合并行的任务：

- 多源检索
- 多文档摘要
- 多系统读取

不适合并行的任务：

- 明确前置依赖的审批流
- 需要前一步结果决定下一步参数的动作

### 落地要点

- 聚合节点最好显式定义字段优先级和冲突规则。
- 子任务过多会导致聚合开销过大，要有 fan-out 上限。
- 并行只是减少 wall clock，不一定减少总成本。

## Q36：如何优化首 token 延迟、整体响应时间和长尾请求？

### 一句话回答

首 token 延迟重点看路由、上下文构建和模型首包；整体响应时间看总轮数和工具链；长尾请求要靠超时、hedging、熔断、降级和热点隔离。

### 详细展开

首 token 优化：

- 预热连接
- 减少 prompt 拼装成本
- 控制首轮上下文大小
- 用更快的首轮模型做路由或意图判定

整体时长优化：

- 减少工具链串行
- 缩短模型轮数
- 命中缓存
- 只对必要请求加 rerank 或二次评估

长尾优化：

- 设置工具超时
- 对高抖动依赖使用 hedge request
- 对低优先级任务排队而不是拖慢全部请求
- 热租户与冷租户隔离资源池

### 落地要点

- 一定要分别看 TTFT、E2E latency、tool latency、queue latency。
- 长尾常常不是模型慢，而是某个外部依赖拖垮了整个链路。
- 流式返回能显著改善体验，但不能掩盖真实处理延迟问题。

## Q37：高并发下模型调用如何做限流、排队和优先级控制？

### 一句话回答

核心是把“请求量”换算成“资源量”，按 `并发、token、成本、SLA` 做多维限流和优先级队列，而不是所有请求一把梭哈进模型。

### 详细展开

常见控制方式：

- `token bucket`：限制瞬时速率
- `concurrency limit`：限制同时活跃请求数
- `priority queue`：高优先级业务优先
- `weighted fair queue`：不同租户公平分享资源
- `deadline aware scheduling`：离超时近的优先执行

高优先级常见包括：

- 售后在线会话
- 商家工作台核心操作
- 高价值客户服务

低优先级包括：

- 离线总结
- 非实时推荐生成
- 批量分析

### 落地要点

- 配额要区分请求数和 token 数。
- 限流后要有明确返回策略：排队、降级还是拒绝。
- 面试时如果能提“以 token 维度做流控”，会比只说 QPS 更专业。

## Q38：Agent 服务如何做降级与熔断？

### 一句话回答

Agent 的降级不是“挂了就报错”，而是分层退化：从多步 Agent 退到单步问答、从自动执行退到建议模式、从实时工具退到缓存或模板回复。

### 详细展开

可以设计几级降级：

- `Level 1`：关闭非关键工具，如 rerank、反思器
- `Level 2`：从多 Agent 退回单 Agent
- `Level 3`：从 Agent 退回 workflow / FAQ 检索
- `Level 4`：直接转人工或返回排队中

熔断条件可以是：

- 下游错误率过高
- 工具超时率飙升
- 成本或 token 消耗超阈值
- 相同错误在短时间集中爆发

### 落地要点

- 熔断要按工具、租户、业务线细粒度做，不要全局一起熔。
- 降级路径要提前演练，不能等故障时才想。
- 返回用户的话术也要设计，避免“系统忙，请稍后再试”成为唯一答案。

## Q39：多模态 Agent 架构需要额外考虑什么？

### 一句话回答

多模态 Agent 比纯文本 Agent 多了 `媒体预处理、特征抽取、异步管线、模态对齐、存储与带宽成本、审核与安全` 这些问题。

### 详细展开

额外考虑点包括：

- 图片、音频、视频的预处理和压缩
- 不同模态特征向量的索引与检索
- 大对象存储和 CDN
- 模态间对齐，如图文商品卡、视频商品讲解
- 更严格的内容安全与版权审核

多模态链路通常比文本更适合异步化，因为：

- 特征抽取耗时
- 数据体积大
- 下游工具和审核更复杂

### 落地要点

- 模态原始数据不要直接塞进对话上下文，优先抽取结构化特征。
- 对视频和语音任务要额外关注成本与尾延迟。
- 多模态缓存策略要考虑对象版本与特征版本。

## Q40：在淘天场景下，Agent 如何与商品、店铺、订单、营销系统协同？

### 一句话回答

不要让 Agent 直接跨系统乱调，而要通过领域工具层协同商品、店铺、订单、营销系统，并把实时事实查询、规则判断、可写动作和风控审批分开。

### 详细展开

典型协同方式：

- 商品域：查商品详情、卖点、属性、库存、上下架状态
- 店铺域：查店铺资质、服务承诺、活动参与情况
- 订单域：查订单状态、物流、退款、支付异常
- 营销域：查优惠券、活动规则、报名条件、投放结果

推荐架构：

- 统一 Tool Gateway 暴露领域能力
- 每个领域保留自己的 source of truth
- Agent 只做编排、解释、总结和动作提案
- 真正执行由领域服务 + 风控 + 审批完成

### 落地要点

- 事实查询优先结构化接口，文档检索负责解释和补充。
- 营销、退款、改价等动作必须带风控和审批。
- 跨域协同时，主键映射和事件时序一致性很重要。
