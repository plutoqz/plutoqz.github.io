# Function Calling、工具设计与系统集成

## 本章目标

- 回答 Agent 从“会调工具”到“稳定调工具”中间缺的所有工程环节。
- 讲清工具描述、Function Calling、结构化输出、重试补偿、老系统接入和租户隔离。
- 把“模型调用工具”这件事，讲成一个真正可靠的系统。

## 关键问题

- 为什么 Function Calling 很容易在 demo 好用、线上不稳？
- description 到底该怎么写？
- 老系统能力接入 Agent 后，怎么避免工具层失控？

## Q5：Function Calling 的可靠性如何保障？如果模型一直调用同一个错误工具怎么办？

### 一句话回答

Function Calling 的稳定性不是靠模型“自己悟”，而是靠 `强 schema、运行时校验、错误分型、循环检测、降级策略、人工兜底` 一起保证。

### 详细展开

可靠性常见问题有：

- 选错工具
- 参数缺失或字段类型错误
- 工具成功返回但业务语义不对
- 工具失败后模型反复重试同一个错误动作

推荐的防线是：

- 工具 schema 严格类型化，必填字段、枚举、范围都收紧
- tool result 做结构化标准化，不把一大段自由文本直接回喂
- error code 分型，如 `INVALID_ARGUMENT`、`PERMISSION_DENIED`、`TIMEOUT`
- runner 做重复调用检测，比如 `tool_name + args_hash + error_code`
- 连续同类失败达到阈值就强制 replan、切备用工具或转人工

如果模型一直调用同一个错误工具，可这样处理：

1. 运行时识别重复模式
2. 在下一轮上下文中明确反馈“该工具不适用，原因是什么”
3. 暂时屏蔽该工具或降低其候选优先级
4. 必要时切到确定性路由器或人工接手

### 落地要点

- 不要把工具失败只返回成一句“failed”。
- 反复错误工具调用，本质上是状态机和工具设计问题，不只是模型问题。
- 面试里如果能提 `loop breaker`、`args hash`、`tool cooldown`，会很加分。

## Q6：工具的 description 怎么写才能让模型准确调用？给出淘天场景的示例

### 一句话回答

好的 description 不是“介绍工具是什么”，而是明确告诉模型 `什么时候用、什么时候别用、参数怎么填、结果代表什么、风险在哪里`。

### 详细展开

一个高质量 description 至少包含：

- 工具用途
- 适用场景
- 不适用场景
- 必填参数和取值约束
- 返回字段说明
- 风险提示和副作用

错误示例：

`查询订单信息`

这个描述过短，模型不知道：

- 查订单状态还是查退款资格
- 是否支持模糊查询
- 必须传订单 id 还是可传手机号
- 是否有权限限制

更好的淘天示例：

```text
工具名：get_order_snapshot
用途：查询单个订单的最新结构化快照，包括订单状态、支付状态、退款状态、物流状态和金额信息。
适用场景：当用户明确提供订单号，或上游已完成订单定位时使用。
不适用场景：不要用它做订单搜索；不要在缺少订单号时猜测调用。
必填参数：
- order_id: 字符串，必须是完整订单号
返回：
- order_status
- pay_status
- refund_status
- logistics_status
- total_amount
风险：
- 只读工具，无副作用
- 如果订单不存在或无权限，返回明确错误码，不要重试超过1次
```

再比如高风险工具：

```text
工具名：submit_refund_application
用途：为已确认满足条件的订单提交退款申请草稿。
适用场景：仅在已拿到订单号、退款原因、退款金额，并通过资格校验后使用。
不适用场景：不要用它代替资格校验；不要在用户表达不明确时调用。
副作用：会创建退款申请记录，需要审批。
```

### 落地要点

- description 要写“不要在什么情况下调用”。
- 工具名、参数名、description 要语义一致。
- 能举一两个正反例，效果通常比长篇泛描述更好。

## Q26：如何评估 RAG/Agent 的效果？离线评测和在线指标怎么结合？

### 一句话回答

评估要分组件层和任务层，离线看召回、排序、工具选择和安全边界，在线看解决率、人工转接率、用户满意度、成本和时延。

### 详细展开

离线指标可以看：

- 检索层：Recall@K、MRR、nDCG
- 生成层：faithfulness、citation accuracy、format accuracy
- 工具层：tool selection accuracy、argument accuracy
- Agent 层：task success、mean turns、unsafe action rate

在线指标可以看：

- 问题解决率
- 人工转接率
- 一次解决率
- 首响应时间
- 平均会话时长
- token 成本 / 每次解决成本

### 落地要点

- 离线评测负责“能不能”，在线指标负责“值不值”。
- 评估集要覆盖正常、边界、异常、恶意输入。
- 不能只看最终答案，Agent 还要看路径质量。

## Q27：Agent 输出经常看起来合理但事实不对，如何降低幻觉？

### 一句话回答

降低幻觉的核心是让模型少靠猜、多靠证据，并对“没有证据时的行为”做显式约束。

### 详细展开

常见做法：

- 明确要求基于检索或工具结果回答
- 引用证据 id 或来源片段
- 对实时事实强制先查工具再答
- 引入 answer verifier 或 fact checker
- 对低置信度场景允许“不确定 / 需人工确认”

还要减少上下文噪声，因为很多幻觉不是“模型胡编”，而是被错误上下文误导。

### 落地要点

- 对规则、价格、库存、订单状态这类事实，尽量 tool-first。
- 给模型一个明确兜底：没证据时宁可澄清，不要补全。
- 通过离线样本专门测“看起来很像真的错答案”。

## Q28：如何做工具结果校验与结构化输出约束？

### 一句话回答

结构化输出不是为了好看，而是为了把自然语言不确定性压缩到可验证的接口里。

### 详细展开

推荐三层校验：

1. `Schema 校验`：类型、必填、枚举、范围
2. `业务校验`：金额是否为正、时间区间是否合法、订单状态是否允许该动作
3. `一致性校验`：字段间是否冲突，如退款金额不能超过订单实付金额

工具结果也应标准化成固定结构：

- status
- error_code
- retryable
- payload
- source
- timestamp

### 落地要点

- 校验失败时要把失败类型显式返回给模型。
- 强副作用工具前最好再做一次服务端业务校验。
- 输出约束和工具约束要同时做，不能只约束模型最终文本。

## Q29：如果外部工具超时、失败或返回脏数据，Agent 怎么降级？

### 一句话回答

降级原则是“先保安全，再保正确，最后保体验”，常见策略包括超时熔断、读缓存、切备用工具、降低任务目标、转人工。

### 详细展开

超时或失败时常见处理链：

- 短重试：仅对幂等、可重试错误生效
- 切备用：如主检索挂了切只读缓存或次级索引
- 降目标：从“自动执行”退到“生成建议”
- 转人工：高风险动作不继续自动化

返回脏数据时要先做清洗和校验，不要直接回喂给模型。

### 落地要点

- 每个工具都要定义 `retryable / non-retryable / dirty-data` 三种失败类型。
- 对外回答可明确告知“当前查询到的数据可能不完整”。
- 降级路径本身也要评测，不能只在纸面上存在。

## Q30：如何设计重试、幂等与补偿机制，避免重复副作用？

### 一句话回答

Agent 场景里很难做到真正 exactly-once，工程上通常是 `幂等 key + 显式状态 + 有界重试 + 补偿动作`。

### 详细展开

推荐实践：

- 给每个可写动作生成 idempotency key
- 执行前先查是否已成功过
- 失败后只对明确可重试错误重试
- 多步流程用 saga / compensation 思路设计回滚

例如发券流程：

1. 先生成申请单
2. 审批通过后发券
3. 发券成功写审计
4. 若后续步骤失败，则补偿为撤回或冻结

### 落地要点

- 补偿动作不一定等于回滚成功，要明确最终一致性策略。
- 写操作前要落状态或事件日志，便于恢复。
- 幂等设计要覆盖“网络重试”和“模型重复调用”两种情况。

## Q33：如何把已有搜索、推荐、客服、风控系统能力接入 Agent？

### 一句话回答

不要让 Agent 直接侵入老系统内部实现，而是通过 `工具网关 + 领域封装服务 + 统一协议` 的方式接入现有能力。

### 详细展开

推荐做法：

- 搜索系统：暴露结构化检索工具，如 `search_products`
- 推荐系统：暴露候选生成或 rerank 工具，而不是直接开放底层特征
- 客服系统：暴露工单查询、转人工、会话摘要写入工具
- 风控系统：暴露风险评分或可执行性判定工具

这样做的好处是：

- 保护原系统边界
- 统一鉴权、审计、限流
- 方便把复杂接口收敛成模型容易调用的 schema

### 落地要点

- 工具层是“领域 API”，不是老系统内部 RPC 的直接镜像。
- 对推荐、风控这类复杂系统，最好只开放有限职责能力。
- 所有工具都应标准化返回，避免每个系统风格不同。

## Q34：Agent 服务如何做租户隔离与资源配额控制？

### 一句话回答

租户隔离要覆盖 `身份、数据、缓存、向量空间、工具权限、模型预算、QPS 配额`，否则很容易出现串数、抢资源和成本失控。

### 详细展开

隔离维度通常包括：

- 身份隔离：tenant_id、shop_id、user_id
- 数据隔离：memory namespace、vector collection、session store
- 缓存隔离：cache key 带 tenant 和 permission scope
- 工具隔离：不同租户可用工具集合不同
- 成本隔离：tokens、QPS、并发数、日预算

配额控制建议：

- 每租户配置并发上限
- 设置 token budget 和 cost budget
- 高等级租户享受更高 SLA
- 爆量时做优先级排队和降级

### 落地要点

- 不要只做逻辑隔离，至少要在 key、namespace、trace 里全带租户标识。
- 多租户场景里，缓存命中前必须再次校验权限。
- 预算控制要在网关和 runner 双层都做。
