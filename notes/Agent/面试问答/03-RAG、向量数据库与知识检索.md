# RAG、向量数据库与知识检索

## 本章目标

- 回答 RAG 系统里最常见的检索、索引、向量库和知识工程问题。
- 建立“召回不是只看向量相似度”的工程认知。
- 能在淘天场景下说清楚为什么这么切 chunk、为什么这么选库、为什么这样接高时效知识。

## 关键问题

- 向量数据库到底怎么选？
- chunk、embedding、hybrid search、rerank 分别解决什么问题？
- 电商实时知识和文档知识如何共存？

## Q4：向量数据库选型对比（Milvus/Pinecone/Chroma），淘天场景怎么选？

### 一句话回答

Milvus 更适合大规模、强控制、自建或私有化部署； Pinecone 更适合托管式、快速上线和低运维负担； Chroma 更适合本地开发、原型验证和轻量场景。淘天这类高并发、强时效、强治理场景，主选通常更偏 Milvus，这个结论属于工程推断。

### 详细展开

可以先给面试官一张对比表：

| 维度 | Milvus | Pinecone | Chroma |
| --- | --- | --- | --- |
| 形态 | 开源 + 云服务 | 托管式服务为主 | 开源开发体验强，云能力也在演进 |
| 规模能力 | 强分布式、适合超大规模 | 托管扩展方便 | 更适合轻量到中等规模与开发场景 |
| 自主可控 | 高 | 相对低 | 中 |
| 运维负担 | 较高 | 低 | 低到中 |
| 检索形态 | Dense / Sparse / Hybrid / 多向量 | Dense / Sparse / Hybrid / Rerank | Dense / Sparse / Filter / Search API |
| 典型优势 | 云原生分布式、计算存储分离、可深度定制 | 快速接入、托管省心、配套推理能力 | 本地原型快、API 简洁、开发门槛低 |

结合官方文档能抓住几个关键点：

- Milvus 官方文档明确强调分布式、计算存储分离、面向十亿乃至更大规模以及 dense + sparse + BM25 混合检索能力。
- Pinecone 官方文档强调 fully managed database、快速开始、集成 embedding/rerank、dense/sparse 分离索引的 hybrid 检索。
- Chroma 官方文档强调 retrieval 一体化体验、dense + sparse、metadata filtering、从 local 到 single-node 再到 distributed 的架构。

在淘天场景里，常见约束是：

- 数据规模大，商品、店铺、活动、规则、知识卡片都很多
- 更新频繁，活动规则和商品状态有明显时效性
- 多租户和权限边界复杂
- 成本和基础设施可控性要求高

基于这些约束，工程上往往更偏：

- `生产主库`：Milvus 或其他强分布式、可私有化、可深度定制方案
- `快速试点或海外独立业务`：Pinecone
- `本地原型、离线工具、单机验证`：Chroma

### 落地要点

- 不要把“向量库选型”只看成检索速度问题，还要看治理、时效、部署方式、成本和团队掌控力。
- 淘天场景里，热点实时事实不一定全部放向量库，很多应直接走结构化实时服务。
- 面试时最好把“为什么选 Milvus”说成工程推断，而不是说成阿里官方公开结论。

## Q7：RAG 系统中，chunk 大小怎么确定？不同业务场景的策略差异？

### 一句话回答

chunk 大小没有固定标准，核心是看“用户问题粒度、文档结构、检索方式、模型上下文预算、答案所需证据跨度”，本质上是在做召回率和噪声率的平衡。

### 详细展开

常见经验：

- chunk 太小：语义不完整，召回到片段但回答不全
- chunk 太大：噪声太多，相关片段被稀释，占用上下文

可以按场景分：

| 场景 | 建议 chunk 策略 |
| --- | --- |
| FAQ / 客服知识库 | 200 到 500 tokens，按问答对或语义段落切 |
| 规则/制度文档 | 400 到 800 tokens，保留标题层级与条款边界 |
| 商品详情/卖点 | 以单商品、单模块为单位，不盲目定长切 |
| 长报告/研究材料 | 500 到 1200 tokens，配标题与章节路径 |
| 代码/配置 | 以函数、类、模块边界切，不按纯 token 长度 |

overlap 通常建议 10% 到 20%，但不是越大越好。重叠过大容易让重复块挤满 top-k。

### 落地要点

- 优先按结构切，再用 token 上限兜底。
- chunk 里要保留标题路径、文档 id、段落位置信息。
- 不同知识源不要共用一套切分策略。
- 最后用离线评测验证，而不是凭感觉定 512 或 1024。

## Q8：检索结果相关性不高，怎么优化召回？

### 一句话回答

召回差一般不是单点问题，而是 `query 表达、embedding、chunking、索引、过滤、融合、rerank、知识质量` 的综合问题，要分层定位。

### 详细展开

建议按下面顺序排查：

1. `Query 是否对`：用户问法口语化，是否需要 rewrite、query expansion、多路 query。
2. `Chunk 是否对`：切得过碎或过粗都会影响召回。
3. `Embedding 是否对`：中文、电商术语、品牌型号词是否表达充分。
4. `Filter 是否对`：类目、时间、租户、语言过滤是否过严或缺失。
5. `Retrieval strategy 是否对`：是否该上 hybrid search 或 rerank。
6. `Knowledge 本身是否脏`：重复、过期、模板噪声、解析错误。

常用优化手段：

- query rewrite 和 multi-query
- dense + BM25 hybrid
- metadata pre-filter
- cross-encoder rerank
- hard negative mining
- 对失败 query 建专项词典和意图模板

### 落地要点

- 先看召回不到，还是召回到但排序错。
- 对电商问题，品牌词、型号词、活动词、类目词通常对 BM25 很重要。
- rerank 很有效，但成本和延迟也明显更高，应控制候选集规模。

## Q21：RAG 里的 embedding 模型怎么选？通用模型和领域模型如何权衡？

### 一句话回答

先用通用高质量 embedding 建基线，再看领域术语、语言分布、成本和时延是否驱动你上领域模型或蒸馏模型。

### 详细展开

选 embedding 一般看：

- 语言覆盖：中文、电商术语、英文混合是否都好
- 领域适配：品牌、属性、SKU、售后、营销活动词汇是否表达准确
- 维度与存储成本：向量维度越高，存储和检索成本通常越高
- 推理时延：在线 query embedding 延迟是否可接受
- 可迁移性：后续换模型时重建索引成本有多大

通用模型适合：

- 数据域比较广
- 需要快速起步
- 先建立 baseline

领域模型适合：

- 行业术语强
- query/文档分布明显偏垂类
- baseline 在关键业务 query 上不够好

### 落地要点

- 不要只看公开榜单，要用自己的 query-doc 对评测 Recall@K、MRR、nDCG。
- 淘天场景要重点看中文商品词、属性词、活动规则表达。
- 可先双写双检索，在同一批样本上做 A/B 再切换。

## Q22：向量检索、BM25、混合检索、Rerank 怎么组合？

### 一句话回答

生产常见组合是 `过滤 -> 多路召回 -> 结果融合 -> Rerank -> 上下文打包`，而不是把某一种检索当万能解。

### 详细展开

推荐的分层方式：

1. `Pre-filter`：先按租户、类目、语言、时间、生效状态过滤。
2. `Candidate generation`：向量检索负责语义，BM25 负责词面匹配。
3. `Merge`：用 RRF 或加权融合合并候选。
4. `Rerank`：对 top 20 到 top 100 做更精细排序。
5. `Context packing`：去重、压缩、按证据强度排序后喂给模型。

适用经验：

- 搜索词很短、关键词很强时，BM25 很重要。
- 用户问法口语化、语义泛化强时，向量检索更重要。
- 高精度问答场景几乎都建议加 rerank。

### 落地要点

- Hybrid 不是“把两种结果拼一下”这么简单，融合权重和去重策略很关键。
- Rerank 只适合小候选集，否则延迟高。
- 检索链路要能单独评测每一层效果，不然不好优化。

## Q23：知识库文档怎么清洗、切分、打标和版本化？

### 一句话回答

知识库建设不是“把 PDF 扔进向量库”，而是要做解析、清洗、结构重建、切分、打标签、版本管理和删除回收。

### 详细展开

标准流程可以是：

1. `解析`：把 PDF、网页、表格、工单知识转成结构化中间格式。
2. `清洗`：去页眉页脚、模板噪声、重复段、乱码。
3. `结构重建`：恢复标题层级、表格关系、列表语义。
4. `切分`：按业务策略切 chunk。
5. `打标`：文档类型、业务域、类目、语言、发布时间、生效时间、租户、敏感等级。
6. `版本化`：文档版本号、chunk 版本号、向量版本号、索引版本号。

版本化尤其重要，因为：

- 文档会变更
- 规则会失效
- 向量可能要重建
- 缓存和检索结果需要可追溯

### 落地要点

- 每个 chunk 都要保留 source_doc_id 和 source_span。
- 删除或失效时，要能从源文档反向定位所有 chunk 和向量。
- 不同来源的可信度要进 metadata，供 rerank 或生成时参考。

## Q24：电商场景中商品、类目、活动、规则等高时效知识如何接入 RAG？

### 一句话回答

高时效知识不能一股脑全塞向量库，应该做“文档知识 + 结构化实时知识 + 规则引擎 + 检索编排”的分层接入。

### 详细展开

可以分三层：

- `慢知识`：客服 SOP、售后规则说明、运营手册，适合 RAG 文档检索
- `中速知识`：活动规则、类目口径、营销说明，适合带版本和生效时间的检索
- `快事实`：库存、价格、优惠、订单状态，应该优先走实时服务查询

典型做法：

- 文档类问题先走 RAG
- 涉及实时字段的问题先调用结构化工具
- 最后让模型融合“规则说明 + 实时事实”给出回答

### 落地要点

- 把“是否需要实时事实”做成路由条件。
- 活动规则必须带生效时间和适用范围过滤。
- 对价格、库存、优惠等字段，向量检索只能做辅助解释，不能作为最终事实源。

## Q25：向量库写入延迟、索引构建、热更新怎么设计？

### 一句话回答

向量库更新要区分“实时可查”和“全量最优索引”两条路径，通常采用流式写入 + 增量可见 + 异步索引构建 + 周期性合并压缩。

### 详细展开

常见策略是：

- 新数据先进入写路径，尽快变成“可查询”
- 索引构建在后台异步完成
- 查询时兼顾“已构建索引的数据”和“最新增量数据”
- 周期性做 compaction、merge、删改回收

如果业务更新频繁，还要考虑：

- 版本切换时如何避免旧规则被召回
- 删文档或下线知识后如何快速失效
- 索引重建时如何灰度切流

### 落地要点

- 用版本号管理 collection / index / embedding model。
- 热更新优先保证“正确可见”，再追求“索引最优”。
- 大规模重建索引时，可以双索引并行，切流后再回收旧索引。
