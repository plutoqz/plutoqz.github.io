# 监控、Trace、评测、安全与发布治理

## 本章目标

- 回答 Agent 为什么必须从 Day 1 就把观测、评测、安全和发布治理设计进去。
- 建立“业务指标 + 链路指标 + 安全指标 + 发布指标”的完整监控思路。
- 能把 Trace、Replay、灰度、回滚讲成一套成熟的工程体系。

## 关键问题

- Agent 上线后到底该盯什么？
- 如何用 Trace 把一次请求完整串起来？
- 为什么 Agent 的安全和评测都比普通聊天机器人更难？

## Q11：Agent 系统的核心监控指标有哪些？怎么设计告警策略？

### 一句话回答

Agent 监控要分成 `业务效果、模型链路、工具链路、系统资源、安全风险、成本治理` 六层，告警则要按 `即时故障、趋势异常、SLO burn rate` 分级。

### 详细展开

推荐指标体系：

| 维度 | 关键指标 |
| --- | --- |
| 业务效果 | 解决率、人工转接率、一次解决率、用户满意度、投诉率 |
| 模型链路 | mean turns、token in/out、TTFT、E2E latency、模型错误率 |
| 工具链路 | tool success rate、tool timeout rate、tool retry rate、重复调用率 |
| 系统资源 | QPS、队列长度、CPU、内存、连接池、线程池占用 |
| 安全风险 | prompt injection 命中率、越权调用拦截率、DLP 命中率 |
| 成本治理 | cost per request、cost per resolved case、异常 token 消耗 |

告警策略建议：

- `P1`：解决率暴跌、核心工具全挂、模型不可用、越权调用激增
- `P2`：P95 延迟飙升、人工转接率异常、成本异常上涨
- `P3`：单租户异常、局部工具质量下滑、召回率下降

### 落地要点

- 用阈值告警 + 异常检测 + burn-rate 告警结合，不要只设死阈值。
- 指标要能下钻到 tenant、tool、model、prompt version。
- 监控要和回放系统联动，告警后能快速定位具体 trace。

## Q12：如何追踪 Agent 的单次请求完整链路？跨多个服务的 Trace 怎么设计？

### 一句话回答

把一次用户请求视为一个顶层 trace，并为 `网关、状态加载、检索、模型调用、工具调用、Guardrail、审批、输出` 分别打 span，跨服务使用统一 trace id 和上下文传播标准。

### 详细展开

推荐的 span 结构：

- `gateway_span`
- `memory_load_span`
- `retrieval_span`
- `generation_span`
- `tool_span`
- `guardrail_span`
- `approval_span`
- `response_span`

跨服务 Trace 设计要点：

- 入口生成 `trace_id`
- `session_id / run_id / tenant_id` 作为 metadata 或 baggage
- 服务间使用统一传播协议，比如 W3C Trace Context / OpenTelemetry
- 异步任务通过 span links 或父子关系继续串联

记录内容建议包括：

- 输入长度和敏感级别标签
- 模型版本、prompt 版本、工具版本
- 每步开始结束时间
- 状态变化
- 错误类型和重试次数

### 落地要点

- Trace 中要做敏感字段脱敏，不要把全部正文原样打进日志。
- Agent 场景里 group_id 很重要，可把同一会话的多个 trace 关联起来。
- 面试时提到 OpenTelemetry 的 trace / span / context propagation，会更完整。

## Q41：Prompt Injection 在 Agent 系统里为什么更危险？如何防？

### 一句话回答

因为 Agent 不只是“回答”，还会“执行”，一旦不可信内容进入决策链，就可能诱导工具调用、越权访问或错误工作流跳转。

### 详细展开

普通聊天系统里，注入通常导致错误回答；在 Agent 里，注入还可能导致：

- 错误地调用工具
- 绕过业务约束
- 泄露隐私信息
- 触发高风险动作

核心防护思路是“信任分区”：

- 高优先级指令区：system / developer
- 中优先级区：结构化状态、受信工具结果
- 低优先级区：用户输入、网页、文档片段、检索内容

对应措施：

- 显式标记不可信内容来源
- 低信任内容不能写进高优先级 prompt 区
- 对高风险工具启用审批和策略校验
- 用结构化输出而不是自由文本在节点间传递

### 落地要点

- 对来自网页、商品描述、外部工单文本的内容尤其要警惕。
- 防注入不能只靠规则提示，还要靠权限模型和审批。
- 检索前做内容清洗、检索后做证据筛选，也能减轻注入面。

## Q42：工具越权、数据泄露、敏感信息外发怎么防控？

### 一句话回答

要把工具看成一层独立安全边界，做 `最小权限凭证、参数级鉴权、DLP 检测、出网控制、租户隔离、审计追踪`。

### 详细展开

防控重点包括：

- 每个工具使用独立、最小权限凭证
- 工具调用前验证调用者角色、租户、对象归属
- 对敏感字段做脱敏、tokenization 或字段级授权
- 出网统一走代理层，过滤敏感数据外发
- 对高风险字段做 DLP 扫描，如手机号、身份证、地址、订单金额

可进一步增加：

- 工具 allowlist / denylist
- 每工具单独 rate limit
- 风险级动作审批
- 结果二次审计

### 落地要点

- 不要把一个高权限服务账号共享给所有工具。
- 数据泄露风险不只在最终输出，也在工具请求和回传链路。
- 对跨系统调用，要清楚谁是 source of truth，谁只是只读镜像。

## Q43：如何设计 Agent 的评测集、回放系统和回归测试机制？

### 一句话回答

生产 Agent 要有 `题库 + 真实 trace 回放 + 安全攻击样本 + 版本对比` 四件套，才能在改 prompt、换模型、加工具后不把线上搞炸。

### 详细展开

评测集建议分层：

- 常规样本：高频真实问题
- 边界样本：长上下文、模糊需求、缺字段输入
- 故障样本：工具超时、脏数据、权限拒绝
- 安全样本：prompt injection、越权请求、数据外发诱导

回放系统应支持：

- 重放历史 trace
- 固定或 mock 工具结果
- 对比不同模型 / prompt 版本
- 输出差异报告，如工具选择变化、轮数变化、成本变化

### 落地要点

- 回归测试不能只测最终答案，要测路径和副作用。
- 高风险工具尽量在沙箱或 dry-run 下回放。
- 最有价值的样本常来自线上事故和差评工单。

## Q44：Agent 上线后如何做灰度发布、A/B 实验和版本回滚？

### 一句话回答

Agent 发布要做版本化和可切换设计，把 `模型版本、prompt 版本、工具 schema 版本、索引版本、路由策略版本` 分开管理，并通过 canary、shadow、A/B 和一键回滚控制风险。

### 详细展开

推荐发布流程：

1. 线下 replay 和回归通过
2. shadow traffic 观察，不影响真实结果
3. canary 到少量租户或流量分片
4. A/B 比较业务指标
5. 分阶段扩大

回滚要有：

- 模型开关
- prompt 模板开关
- 工具版本开关
- 索引版本回切
- 缓存隔离或失效机制

### 落地要点

- 发布单元尽量按租户或业务域切，不要全量一次发。
- A/B 不只看满意度，还要看人工转接率、成本和安全拦截。
- 版本切换时，老 trace 和新 trace 要能并存分析。

## Q45：如何识别和治理低价值推理、无效工具调用和成本失控？

### 一句话回答

要把“思考”和“行动”变成可计量对象，识别那些没有提升成功率、却显著增加轮数、工具调用和 token 成本的步骤。

### 详细展开

建议监控这些指标：

- 平均轮数 / 成功请求
- 平均工具调用次数 / 成功请求
- 重复工具调用率
- 无状态变化轮次占比
- cost per successful task
- 工具失败后仍继续调用的比例

治理手段：

- 设置预算管理器，超过预算强制收敛
- 对低收益反思节点降频
- 对重复失败工具设置 cooldown
- 将高频简单问题改成 workflow 或 FAQ

### 落地要点

- 成本优化不是一味降模型，先去掉无价值步骤往往收益更大。
- 以“单位成功任务成本”而不是“单次请求成本”更能说明问题。
- 对不同业务线单独建成本画像，避免平均数掩盖异常。

## Q47：如何设计面向排障的日志结构和事件模型？

### 一句话回答

日志不要只记一坨文本，而要围绕事件模型设计：`一次 run 发生了哪些关键事件、状态怎么变、工具怎么调、错误怎么演化`。

### 详细展开

推荐事件类型：

- `request_received`
- `memory_loaded`
- `retrieval_done`
- `generation_done`
- `tool_called`
- `tool_failed`
- `guardrail_blocked`
- `human_handoff`
- `response_sent`

每条事件建议带：

- trace_id / span_id / run_id / session_id
- tenant_id / user_id / shop_id
- model_version / prompt_version / tool_version
- state_before / state_after
- latency_ms
- token_usage
- error_code
- args_hash

### 落地要点

- 事件日志适合排障和回放，文本日志适合补充上下文。
- 先定义事件 schema，再让各服务按 schema 打点。
- 日志平台要支持按 trace、tenant、tool、model 维度检索聚合。
