# 记忆、上下文与缓存工程

## 本章目标

- 回答多轮对话怎么记、怎么控、怎么删、怎么省。
- 理清 prompt 分层、上下文预算、摘要压缩、缓存命中与合规治理。
- 让“记忆”从概念变成一套可落地的存储和生命周期设计。

## 关键问题

- 多轮记忆到底怎么分层？
- 缓存怎么设计才不会引入污染、泄露和陈旧结果？
- 模型升级、提示词改版后，怎么保证记忆系统还能工作？

## Q3：多轮对话中，记忆如何分级管理？缓存策略怎么设计？

### 一句话回答

多轮对话里的记忆应至少分成 `请求级上下文、会话级短期记忆、长期用户画像、外部知识层、执行状态层`，缓存则要分成 `检索缓存、工具结果缓存、Prompt/语义缓存、最终结果缓存`。

### 详细展开

建议这样分层：

| 层级 | 作用 | 示例 |
| --- | --- | --- |
| 请求级上下文 | 只服务当前一轮 | 当前工具返回值、当前页面内容 |
| 会话级短期记忆 | 维持连续对话 | 最近 N 轮消息、当前目标、已确认约束 |
| 长期画像记忆 | 维持稳定偏好 | 用户语言、店铺属性、偏好模板 |
| 外部知识层 | 按需检索，不直接常驻 | 规则库、商品库、活动配置 |
| 执行状态层 | 控制 run 进度 | 当前步骤、失败次数、审批状态 |

缓存策略可以再分四类：

- `retrieval cache`：相同查询与过滤条件的检索结果缓存
- `tool result cache`：只读工具的查询结果缓存
- `prompt / semantic cache`：高度相似问题直接复用结果
- `final answer cache`：高重复 FAQ 场景缓存最终答案

### 落地要点

- 执行状态不要缓存成最终答案，要独立持久化。
- 高时效数据如库存、价格、活动状态应设置极短 TTL 或不缓存。
- 语义缓存必须按租户、用户、权限和时间窗做隔离。
- 对写操作绝不能直接走结果缓存。

## Q18：Agent 的 system、developer、user prompt 如何分层设计？

### 一句话回答

system 负责平台级不可变规则，developer 负责当前 Agent 的角色和流程策略，user 负责本次任务意图；检索内容和工具输出应被当成低信任上下文，而不是高优先级指令。

### 详细展开

一个好用的分层方式是：

- `system`：平台级安全规则、法律合规、不可突破的边界
- `developer`：该 Agent 的目标、工具使用规则、输出格式、错误处理策略
- `user`：当前任务和个性化需求
- `retrieval/tool outputs`：事实材料或观测结果，不具备高优先级指令权

这种分层有三个收益：

- 能防止业务规则与平台规则混在一起
- 更方便做 A/B 与版本回滚
- 便于做 prompt 注入隔离

### 落地要点

- 不可信文本不要拼进 developer 指令位。
- developer prompt 应尽量模块化，如“角色定义”“工具选择规则”“终止条件”“输出 schema”分段。
- 用户输入和检索内容进入前最好加标签，如 `USER_INPUT:`、`RETRIEVED_CONTEXT:`。

## Q19：如何控制 prompt 长度，避免上下文膨胀？

### 一句话回答

控制 prompt 长度的核心不是“少放点字”，而是做 `上下文预算管理`：只保留当前决策必需的信息，把状态结构化、把历史摘要化、把知识外部化。

### 详细展开

常见手段有：

- 给不同区块设 token budget，如 system 10%、memory 20%、retrieval 40%、tool results 20%、response reserve 10%
- 历史消息只保留最近窗口 + 稳定摘要
- 工具返回做字段裁剪，不回注整段原始 JSON
- 把“当前执行到哪”放到显式 state，而不是散在历史里
- 对检索只保留 top-k 且做去重、去噪、去模板化

### 落地要点

- 先问“下一轮决策真正需要什么”，而不是“之前所有东西都重要吗”。
- 预算超限时要有降级策略：缩短历史、压缩摘要、缩小检索范围、换轻量模型。
- 对大对象如页面 DOM、长文档全文，要先做抽取再注入。

## Q20：会话摘要何时触发？如何避免摘要污染后续判断？

### 一句话回答

摘要应在“历史过长但仍需连续性”时触发，而且摘要必须区分 `已确认事实、待验证假设、用户偏好、未完成任务`，否则会把猜测固化成事实。

### 详细展开

推荐的触发条件：

- 历史 token 超过阈值
- 对话轮数超过阈值
- 任务从一个阶段切到另一个阶段
- 会话要长期挂起并等待后续恢复

为了避免摘要污染，可以把摘要拆成结构化字段：

- `confirmed_facts`
- `open_questions`
- `user_preferences`
- `pending_actions`
- `do_not_assume`

还可以给摘要增加：

- 时间戳
- 来源轮次范围
- 版本号
- 置信度或“是否需要二次验证”标记

### 落地要点

- 摘要不要由一次模型自由生成后永久覆盖原始历史，最好保留原始归档。
- 高风险任务恢复时，优先回读结构化状态和关键证据，而不是只看摘要。
- 当摘要和当前输入冲突时，应优先相信新证据，并可触发重摘要。

## Q35：如何设计 Agent 的缓存体系？有哪些缓存命中风险？

### 一句话回答

缓存体系应按 `数据来源、时效要求、副作用等级、租户边界` 分层设计；最大的风险不是“没命中”，而是“命中过时、越权、错误上下文的结果”。

### 详细展开

常见缓存分层：

- `LLM prompt cache`：相同或高度相似输入的模型结果
- `semantic cache`：意图相似问题的答案复用
- `retrieval cache`：检索结果缓存
- `tool cache`：只读工具结果缓存
- `feature cache`：用户画像、租户配置、路由规则等中间特征缓存

缓存命中风险主要有：

- `stale`：价格、库存、活动、规则已变更
- `cross-tenant leak`：不同商家或不同用户数据串了
- `permission mismatch`：同问题不同用户权限不同，不能复用
- `context mismatch`：问题文本相似，但上下文目标不同
- `poisoning`：错误答案被长时间复用

### 落地要点

- 缓存 key 不能只用 query，要带上 tenant、role、tool version、time bucket、permission scope。
- 强时效业务要用短 TTL 或事件驱动失效。
- 命中缓存后仍可追加轻量校验，如库存、订单状态二次确认。

## Q46：如何做 Agent 会话存储、归档、TTL 和合规删除？

### 一句话回答

会话数据要按热数据、冷归档、审计数据分层存储，并针对不同数据类型设置不同 TTL、加密策略、删除索引和审计保留规则。

### 详细展开

建议分三层：

- `hot session store`：近期会话、显式状态、最近摘要，支持快速恢复
- `cold archive`：历史消息和 trace，主要用于审计和回放
- `compliance store`：必要的最小审计证据，满足监管或风控要求

数据分类建议：

- 用户可见会话内容
- 工具调用参数与返回
- 系统内部状态
- 敏感信息与 PII
- 审批与风控记录

删除策略建议：

- 普通对话按 TTL 自动清理
- 敏感字段单独加密并支持定向删除
- 向量化后的内容也要建立反向索引，支持同步删除
- 删除要留下不可逆的审计事件，而不是把一切都“物理消失”

### 落地要点

- 删除 API 要做到“会话正文、摘要、向量索引、缓存、副本”一致清理。
- 不同业务线 TTL 不一样，售后、风控、客服聊天不能一刀切。
- 加密后检索能力会下降，要提前设计冷热路径。

## Q48：Agent 里是否需要保存 Chain-of-Thought？怎么做可观测而不泄露敏感推理？

### 一句话回答

通常不建议长期保存完整原始 Chain-of-Thought，更好的做法是保存 `决策摘要、工具调用理由、关键状态变化、评测标签`，满足可观测性而不暴露过多敏感推理。

### 详细展开

原因有三类：

- 安全与隐私：原始推理可能包含敏感信息、策略细节或误导性内容
- 产品风险：一旦泄露，可能放大提示注入和越权利用
- 工程价值有限：排障更需要结构化事件，不一定需要完整自由文本思维链

更实用的可观测方案是记录：

- 本轮选择了哪个工具
- 触发该选择的结构化理由标签
- 状态从什么变成什么
- 失败类型、重试次数、终止原因
- 评测器对这轮的判定

### 落地要点

- 对外不可见，对内也应按最小权限访问。
- 需要调试时优先看 trace spans 和状态 diff，再看必要的推理摘要。
- 如果必须短时保留原始思考内容，应设置极短 TTL 和严格权限。

## Q49：当模型升级或提示词改版后，怎么做兼容性验证？

### 一句话回答

模型和 prompt 的升级不能靠线上撞运气，必须做 `基线样本回放 + trace diff + 金丝雀流量 + 指标守门 + 快速回滚`。

### 详细展开

建议分五步：

1. 建立 golden set，覆盖常规、边界、异常、安全样本。
2. 对历史 trace 做 replay，对比工具选择、终止原因、成本和错误率。
3. 小流量 canary，观察线上延迟、成功率、人工转接率、投诉率。
4. 对高风险工具设 stricter guardrail，防止新模型出现越权动作。
5. 保留模型版本和 prompt 版本双维度开关，便于快速回滚。

需要重点比较的指标：

- task success
- tool selection accuracy
- unsafe action rate
- mean turns
- token cost
- human handoff rate

### 落地要点

- 不要只看最终答案对不对，还要看链路有没有变化。
- prompt 改版时要同步检查缓存 key、summary 模板、评测器规则是否兼容。
- 最稳妥的做法是“先模型升级，后 prompt 升级”，不要同时大改。
