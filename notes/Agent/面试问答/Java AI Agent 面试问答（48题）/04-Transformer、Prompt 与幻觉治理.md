# Transformer、Prompt 与幻觉治理

## 本章目标

- 把 Transformer、Prompt、角色消息和 hallucination 讲成一套连贯体系。
- 回答“为什么大模型会错”“怎么让它少错”“怎么把 prompt 管起来”。

## 关键问题

- Transformer 为什么成了大模型的基础架构？
- hallucination 到底是什么，为什么仅靠“加一句请准确回答”解决不了？
- Prompt 工程为什么是系统设计问题，而不只是写几句提示词？

| 幻觉成因 | 典型表现 | 常见治理手段 |
| --- | --- | --- |
| 训练数据缺口 | 瞎编事实 | RAG、工具查询、引用约束 |
| 指令不清 | 回答跑偏 | Prompt 结构化、角色分层 |
| 上下文不全 | 补全错误细节 | memory、检索、澄清问题 |
| 任务过难 | 逻辑跳步 | 分解任务、ReAct、planning |
| 输出约束太弱 | JSON 格式乱 | schema、结构化输出、校验 |

## Q4：什么是 hallucination（幻觉）？为什么会发生？

### 一句话回答

hallucination 是模型生成了“看起来合理、实际上不可靠或不真实”的内容，本质上是概率生成和真实世界约束之间的偏差。

### 详细展开

模型生成下一个 token 的目标，是让序列在训练分布下更合理，不是天然保证事实为真。因此会出现三类常见幻觉：

1. `事实幻觉`：编造不存在的人、文档、功能、指标。
2. `引用幻觉`：给出看似专业的来源，但来源并不存在。
3. `推理幻觉`：中间步骤看起来顺，但前提或结论不成立。

发生原因通常包括：

- 模型参数中没有相关知识，或知识过时。
- prompt 让模型“必须回答”，但没有足够证据。
- 检索结果不完整，模型自行脑补。
- 输出格式要求太强，模型为了满足格式而瞎填字段。

### 落地要点

- 把 hallucination 看成系统问题，而不是模型“人品问题”。
- 高风险场景里，应该允许模型说“不知道”“证据不足”，而不是强迫输出。
- 要区分“没知识”和“有知识但误用”，治理方式不同。

### 高频追问

- 温度调低能完全解决幻觉吗？
  - 不能。低温只会让错误更稳定，不会自动让事实更真。

## Q11：什么是 Prompt Engineering？

### 一句话回答

Prompt Engineering 是把任务目标、边界、上下文、工具和输出格式组织成稳定输入的工程过程，而不只是“写几句提示词”。

### 详细展开

OpenAI 官方文档把 prompt engineering 定义为“编写有效指令，使模型持续产出满足要求内容的过程”。在工程里，它至少包含：

- 角色分层：系统 / developer / user
- 上下文编排：history、memory、retrieval、tool schema
- 输出约束：JSON、schema、citation、禁止项
- 评测与迭代：prompt 改版必须有回归测试

一个好的 prompt 应满足：

- 目标明确
- 输入边界明确
- 输出格式明确
- 错误处理明确
- 与工具、记忆、检索协同

### 落地要点

- Prompt 不是静态文案，而是系统组件。
- 同一业务需要把“稳定规则”和“动态变量”分开。
- Prompt 版本要可回滚、可灰度、可评测。

### 高频追问

- Prompt Engineering 会不会被更强模型淘汰？
  - 不会。模型越强，系统输入越复杂，prompt 设计只会更像一门接口工程。

## Q26：Transformer 架构核心原理是什么？

### 一句话回答

Transformer 的核心是自注意力机制，它让模型在并行计算中直接建模序列中任意位置之间的关系。

### 详细展开

可以把 Transformer 的关键点概括成五个词：

1. `Embedding`：把 token 映射成向量。
2. `Positional Encoding`：告诉模型顺序信息。
3. `Self-Attention`：计算每个 token 对其他 token 的关注权重。
4. `Feed Forward Network`：做非线性变换。
5. `Residual + LayerNorm`：让深层训练更稳定。

自注意力的价值是：

- 可以并行处理整段序列。
- 能捕捉长距离依赖。
- 相比 RNN 更适合大规模训练。

### 落地要点

- 面试里不要只背 Q、K、V，最好能讲出“为什么 Transformer 能扩展到大模型”。
- 还可以补一句：上下文窗口成本高，正是因为 attention 对长序列计算和内存都很敏感。

### 高频追问

- Transformer 为什么推理会贵？
  - 因为上下文越长，attention 和 KV cache 的代价越明显；这一点和 Q40 可以连起来答。

## Q27：如何减少大模型 hallucination？

### 一句话回答

减少 hallucination 不能只靠一句“请不要编造”，而要同时治理知识来源、任务拆解、输出约束和结果校验。

### 详细展开

常见治理手段有：

- `RAG`：让回答基于外部证据。
- `Tool Calling`：对实时数据走工具，而不是靠模型猜。
- `Prompt 约束`：要求给出证据、允许拒答、禁止无依据扩展。
- `结构化输出`：对关键字段使用 schema。
- `分步执行`：复杂任务先拆解再做。
- `后验校验`：规则检查、事实检查、引用检查、LLM-as-judge。

### 落地要点

- 对高风险场景优先使用“先查再答”，而不是“直接答”。
- 输出中最好区分 `事实`、`推断`、`建议` 三类内容。
- 引用片段最好附 source_id 或文档路径，方便回放。

### 高频追问

- 什么时候应该直接拒答？
  - 没证据、权限不足、风险过高、问题超出范围时，拒答比编造好。

## Q39：ChatGPT 的 system / user / assistant role 有什么作用？

### 一句话回答

这些 role 用来区分不同来源的指令和上下文，从而帮助模型区分“规则”“用户输入”和“历史输出”。

### 详细展开

在经典 Chat Completions 风格中：

- `system`：设定全局行为边界，例如身份、原则、风格。
- `user`：用户输入的需求或问题。
- `assistant`：模型历史输出，帮助形成多轮上下文。

而在较新的 OpenAI 文档中，很多接口强调的是 `instructions` 或 `developer` 消息，其优先级在应用开发者控制层面更明确。工程上可以这样理解：

- `system / developer`：业务规则、角色、禁止项。
- `user`：业务变量和用户诉求。
- `assistant`：历史回答和工具结果带来的上下文延续。

### 落地要点

- 稳定规则放 system / developer，不要混进 user message。
- 用户原话不要被拼接成 system，否则容易带来 prompt injection 风险放大。
- assistant 历史要裁剪，否则上下文膨胀很快。

### 高频追问

- 为什么不把所有说明都放 user 里？
  - 因为系统规则和用户输入语义不同，混在一起会弱化边界，也不利于治理。

## Q40：LLM 为什么推理成本高？

### 一句话回答

LLM 推理成本高，主要因为模型参数大、上下文长、生成是逐 token 进行、并且高质量任务常常需要更复杂的链路。

### 详细展开

成本通常来自四部分：

1. `模型本身`：参数规模越大，计算越重。
2. `上下文`：输入 token 越多，prefill 成本越高。
3. `输出长度`：生成是逐 token 的，输出越长越贵。
4. `系统链路`：RAG、工具调用、rerank、审查器都会叠加成本。

还有一个常被忽略的点：

- 推理模型通常更会规划和多步思考，但往往更慢、更贵。

### 落地要点

- 控制成本最有效的手段，往往不是“换更便宜模型”而是减少无效上下文和无效调用。
- 把不同任务分层到不同模型，是最实用的成本优化方式之一。

### 高频追问

- 为什么首 token 延迟高？
  - 因为模型要先处理完整输入上下文，才能开始输出。

## Q44：如何设计 Prompt 管理系统？

### 一句话回答

Prompt 管理系统的目标是让 prompt 像代码一样可版本化、可测试、可灰度、可回滚。

### 详细展开

一个实用的 Prompt 管理系统至少要支持：

- `模板化`：静态规则和动态变量分离。
- `版本管理`：每次变更有版本号、变更说明和负责人。
- `环境隔离`：dev / test / prod 可区分。
- `灰度发布`：按租户、流量、实验组切换。
- `评测回归`：prompt 改动后自动跑评测集。
- `观测`：记录版本号、命中率、失败率、满意度。

一个常见数据模型是：

- `prompt_key`
- `version`
- `channel`
- `template`
- `variables_schema`
- `output_schema`
- `owner`
- `status`

### 落地要点

- Prompt 不要散落在代码里硬编码。
- Prompt 版本要写入请求日志和 trace，方便定位回归。
- 最好支持“模板预览 + 样例渲染 + 历史 diff”。

### 高频追问

- Prompt 管理系统和配置中心有何区别？
  - Prompt 管理系统除了存模板，还要支持评测、版本、灰度和效果分析，这比普通配置中心要求更高。
