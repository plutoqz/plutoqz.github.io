# 淘天 Agent 面试问答

## 学习目标

-   用一套接近一线工程面试的方式，系统回答 Agent、RAG、Function Calling、缓存、弹性、监控、安全与发布治理问题。
    
-   不只会背概念，还能把问题拆成：架构、数据流、状态机、缓存、权限、降级、评测、告警、发布等工程细节。
    
-   面对淘天这类高并发、高时效、强业务约束的场景，能给出有取舍的方案，而不是“上一个大模型就行”。
    

## 适合人群

-   准备阿里淘天、字节、美团、腾讯等 AI 应用、搜索推荐、平台工程、智能体方向面试的人。
    
-   已经做过 Agent 或 RAG Demo，但需要把答案升级成“生产级系统设计语言”的人。
    
-   想把 Agent 主笔记进一步压缩成可面试表达的人。
    

## 范围边界

-   重点回答软件 Agent、任务型 Agent、RAG Agent、工具调用 Agent、多 Agent 编排系统。
    
-   回答偏工程落地，不展开机器人控制、强化学习、具身智能。
    
-   关于“淘天场景怎么选”的结论，部分是基于公开文档和电商系统约束做的工程推断，不是官方公开架构披露。
    

## 前置知识

-   LLM 与 Function Calling 基础
    
-   RAG 与向量检索基础
    
-   分布式系统、缓存、监控、链路追踪基础
    
-   常见服务治理概念：限流、熔断、灰度、回滚、幂等
    

## 使用方式

每道题尽量按三层来写：

1.  `一句话回答`：适合先抛结论。
    
2.  `详细展开`：适合 2 到 5 分钟口述。
    
3.  `落地要点`：适合面试官继续追问“那你在生产里怎么做”。
    

建议你练习时也按这三层来讲，这样既能先给框架，又能随时下钻。

## 建议阅读顺序

1.  先读 `01`、`02`，把架构、状态机、记忆、缓存和 prompt 分层讲顺。
    
2.  再读 `03`、`04`，把 RAG、向量库、Function Calling、工具集成讲实。
    
3.  然后读 `05`、`06`，把性能、弹性、监控、Trace、安全、评测、发布治理补齐。
    
4.  最后练 `07`，把前面内容整合成一套完整系统设计答案。
    

## 面试表达建议

-   先讲边界：先说任务目标、风险和是否真的需要 Agent。
    
-   再讲主链路：输入、状态、模型决策、工具调用、结果回写、终止条件。
    
-   再讲治理层：权限、审批、缓存、监控、Trace、评测、灰度、回滚。
    
-   最后讲取舍：为什么不用 workflow、为什么不拆多 Agent、为什么向量库这么选。
    

## 学完后的达标标准

如果你能做到下面这些事，说明这套问答已经真正内化了：

-   不看资料，口头答清楚 Q1、Q3、Q4、Q5、Q9、Q11、Q12、Q50。
    
-   对“为什么不用普通 workflow”“为什么不直接多 Agent”“为什么不用 Pinecone”这类追问，能给出场景化取舍。
    
-   能画出一套 Agent 数据流，并指出状态、缓存、权限、Trace、评测放在哪里。
    
-   能把“看起来很聪明”的方案，修正成“生产可控”的方案。
