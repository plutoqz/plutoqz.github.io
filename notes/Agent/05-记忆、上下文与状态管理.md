# 记忆、上下文与状态管理

## 本章目标

-   理解 Agent 为什么离不开显式状态和分层记忆。
    
-   掌握短期记忆、长期记忆、检索知识、执行状态之间的区别。
    
-   学会围绕成本、隐私、恢复能力来设计 memory。
    

## 核心问题

-   记忆到底在解决什么问题？
    
-   上下文、状态、记忆、知识库有什么差别？
    
-   什么时候该总结、压缩、过期、加密？
    

## 正文

### 1\. 先把四个概念分开

这是 Agent 工程里最容易混淆的一组词。

| 概念  | 主要作用 | 例子  |
| --- | --- | --- |
| 上下文 | 当前这次模型调用能看到的信息 | 当前消息、系统提示、工具返回 |
| 状态  | 当前 run 的显式执行进度 | 已完成步骤、当前任务 ID、重试计数 |
| 记忆  | 跨轮次保留的信息 | 用户偏好、过去对话摘要、历史任务结果 |
| 知识库 | 独立存储的外部知识 | 文档库、向量库、数据库 |

一句话理解：

-   上下文是“这一轮我看到了什么”。
    
-   状态是“我现在进行到哪了”。
    
-   记忆是“我以后还想记住什么”。
    
-   知识库是“我随时可以去查什么”。
    

### 2\. 为什么 Agent 特别需要记忆

因为 Agent 不是一次问答，而是持续执行。没有记忆和状态，它会出现三类问题：

-   反复询问已经知道的信息
    
-   忘记之前做过的步骤
    
-   把长任务做成一堆彼此割裂的短任务
    

### 3\. OpenAI Sessions 给我们的启发

OpenAI Agents SDK 的 session 文档给出一个很清晰的心智模型：

-   每次 run 前，系统会自动取回会话历史并拼到输入前面。
    
-   每次 run 后，新产生的用户消息、模型输出、工具调用等会被自动写入 session。
    
-   后续 run 继续使用同一 session，就能延续上下文。
    

这说明 session memory 首先解决的是“跨 run 的连续性”，而不是复杂的智能记忆。

### 4\. 记忆的分层设计

工程上最好把记忆至少分成四层。

#### 4.1 请求级上下文

只为当前一步服务，用完即弃。

例如：

-   当前工具返回值
    
-   当前文件内容
    
-   当前网页片段
    

#### 4.2 会话级短期记忆

在一个会话或任务内持续保留，帮助 Agent 接续对话和任务。

例如：

-   最近几轮对话
    
-   当前任务目标
    
-   用户刚提供的约束
    

#### 4.3 长期用户/系统记忆

跨会话保留的相对稳定信息。

例如：

-   用户偏好
    
-   团队写作风格
    
-   常见项目约束
    
-   历史成功方案模板
    

#### 4.4 外部知识检索层

不直接塞进上下文，而是在需要时检索。

例如：

-   产品文档
    
-   代码库索引
    
-   CRM 数据
    
-   监控系统事件
    

这层很多时候更像 RAG 或数据库，而不该全部称为“memory”。

### 5\. LangGraph 给出的另一个视角

LangGraph 官方文档把 memory 和 long-running workflow 放在一起讲，并明确强调：

-   短期 working memory 用于当前推理和执行。
    
-   长期 memory 用于跨 session 维持状态化行为。
    
-   durable execution 让系统在失败后继续从检查点恢复。
    

这说明在复杂 Agent 里，memory 不是孤立功能，而是和执行恢复强耦合的。

### 6\. 常见记忆策略

#### 6.1 Sliding Window

只保留最近 N 轮消息。

优点：

-   简单
    
-   延迟低
    
-   成本可控
    

缺点：

-   容易忘掉远处但重要的信息
    

#### 6.2 Summary Memory

定期把历史压缩成摘要，再替换原始长历史。

优点：

-   节省上下文
    
-   保留长期主线
    

缺点：

-   摘要会丢失细节
    
-   摘要错误会持续污染后续决策
    

#### 6.3 Retrieval Memory

把历史或知识切分存储，通过检索按需回注。

优点：

-   适合大量信息
    
-   可扩展
    

缺点：

-   依赖检索质量
    
-   容易召回无关内容
    

#### 6.4 Structured Profile Memory

把稳定信息抽成结构化字段，而不是自然语言长文。

例如：

```json
{
  "user_locale": "zh-CN",
  "prefers_brief_answers": true,
  "default_repo_branch_prefix": "codex/"
}
```

这种方式比“用一大段文字描述用户偏好”稳定得多。

### 7\. 记忆压缩与清理

OpenAI Sessions 文档提到 compaction session，可以对历史进行压缩，避免上下文无限增长。这个能力背后的工程问题是：

-   什么信息应该保留原文？
    
-   什么信息可以摘要？
    
-   什么信息应该过期？
    
-   什么信息应该迁移到结构化存储？
    

一个实用原则：

-   决策会再次依赖的信息，优先结构化保留。
    
-   只用于审计回放的信息，可以原样存档但不一定喂回模型。
    
-   高风险、敏感数据应该设置 TTL 或更严格访问控制。
    

### 8\. 安全与隐私不是附属问题

OpenAI Agents SDK 还提供 `EncryptedSession`，支持加密存储和 TTL 过期。这个设计很有代表性，因为它提醒我们：

-   memory 不是只考虑“记得住”
    
-   还要考虑“谁能看”“保留多久”“泄露后后果如何”
    

常见风险包括：

-   把敏感原文长期保存在会话里
    
-   让模型把不该共享的数据传给外部工具
    
-   把过期记忆继续作为事实注入
    

### 9\. State 与 Memory 的最佳实践

下面是一套很实用的分工：

| 信息类型 | 推荐存放位置 |
| --- | --- |
| 当前步骤和执行进度 | 显式 state |
| 最近几轮对话 | session memory |
| 用户稳定偏好 | 结构化 profile memory |
| 海量外部资料 | 检索层 |
| 写操作结果和审计轨迹 | 业务数据库 / 事件日志 |

一句话概括：

> 能结构化的尽量结构化，能外部化的尽量外部化，不要让所有信息都堆在消息历史里。

## 工程细节与取舍

### 1\. 历史越长，不代表效果越好

长历史会带来：

-   token 成本上升
    
-   延迟上升
    
-   噪声增加
    
-   注入面扩大
    

### 2\. 对有副作用的 Agent，要把“执行状态”独立出来

比如：

-   是否已经创建工单
    
-   是否已经发过邮件
    
-   是否已通过审批
    

这些信息不能只存在模型看到的自然语言里，必须能被运行时精确判断。

### 3\. 过期机制很重要

用户偏好可能长期有效，但任务临时约束、一次性令牌、短时文件上下文不应该永久保留。TTL 是非常实际的手段。

### 4\. 加密会影响检索能力

OpenAI `EncryptedSession` 文档也提醒，内容加密后，基于内容的搜索能力会受限。这是典型工程 trade-off：安全性和可检索性往往需要平衡。

## 常见误区

### 误区 1：RAG 就是记忆

不完全对。RAG 更像按需查外部知识；记忆更多是系统主动保留对未来有用的信息。

### 误区 2：把所有历史都喂回模型最保险

不对。上下文污染、成本失控、注入风险都会随之上升。

### 误区 3：只要有 session 就够了

不对。session 解决连续性，不自动解决结构化状态、幂等、副作用恢复和长期知识组织。

## 面试与实战问答

### 问：Agent 的 memory 应该怎么分层？

短答： 至少分为请求级上下文、会话级短期记忆、长期结构化记忆、外部知识检索层。

深答： 分层的核心目的，是分别优化连续性、成本、检索质量、安全性和恢复能力。

### 问：状态和消息历史为什么要分开？

短答： 因为历史适合回放，状态适合控制执行。

深答： 如果所有执行进度都藏在消息文本里，系统就难以精确恢复，也难以避免重复副作用。

### 问：什么时候要做记忆压缩？

短答： 当历史增长开始影响成本、延迟或准确率时。

深答： 压缩不是简单截断，而是把“未来决策真正需要的信息”提取出来，用摘要、结构化 profile 或检索索引继续保留。
