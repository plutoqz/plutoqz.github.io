# 智能 OnCall Agent：知识库、上下文与评测

> 说明：
> - `【代码可证】` 表示可以直接从仓库代码、配置或文档中落到实现细节。
> - `【需按实际补充】` 表示仓库里没有团队人数、上线规模、业务指标等信息，面试前需要替换成你的真实经历。

106. 知识库由什么组成  
答：【代码可证】当前核心知识库由内部文档向量库构成，底层是 Milvus；运维场景里的实时事实在设计上来自 Prometheus 告警和日志工具，而不是提前全部塞进向量库。按现有代码成熟度看，这条“静态知识库 + 实时工具数据”的混合结构在 Java 版体现得更完整，Go 版则保留了同样的接入方向。

107. 历史工单怎么用  
答：【代码可证】当前仓库没有看到专门的工单表和工单检索实现，所以我不会硬说已经做了工单案例推理。更稳妥的回答是：当前主知识源是内部文档，后续如果要接历史工单，我会把它做成结构化案例库或检索增强样本，而不是简单拼进上下文。

108. 告警和日志很长时怎么裁剪  
答：我会按“和当前告警是否相关”来裁剪，而不是全量喂给模型。比如日志只取关联服务、关联时间窗和错误级别，告警只取告警名、级别、活跃时间和关键描述，这样既省 token，也更能减少噪声。

109. 多轮对话保留哪些上下文  
答：【代码可证】Java 版会按 session 记录用户和助手消息对，最多保留 6 轮消息对；Go 版用 `SimpleMemory` 保存消息，窗口大小也是围绕最近几轮。也就是说，当前保留的是最近多轮问答上下文，而不是全量长会话。

110. 如何避免长对话污染  
答：【代码可证】做法很直接，就是滑动窗口裁剪。Java 版会成对删除最旧消息，Go 版也会在消息超限后按偶数条丢弃，尽量保持“问题-回答”配对关系。

111. 有没有记忆机制  
答：【代码可证】有短期记忆，但没有完整长期记忆。短期记忆体现在会话消息窗口；长期可复用知识更多还是依赖文档向量库，而不是专门的 user memory 或 incident memory 服务。

112. 重复问同一告警怎么复用结论  
答：【代码可证】当前主要复用的是会话内记忆，也就是同一个 `sessionId/client_id` 下最近几轮消息。更长期的复用目前没有做成 Incident Cache，所以我会把这题回答成“当前先做会话内复用，后续可以加告警摘要缓存和历史 case 复用”。

113. 如何处理知识过期  
答：我会把它拆成两块。静态文档通过重新上传和重新向量化覆盖旧数据，Java 版文件上传后会先删除同源 `_source` 的旧向量再重建；实时信息则尽量走工具查询，这样不会因为向量库里的旧文档直接覆盖当前事实。

114. 知识权限控制  
答：【代码可证】底座层用了 Sa-Token，说明接口鉴权是有的。  
答：但当前知识库内部更细粒度的“文档级权限过滤”没有做得特别重，所以面试时我会说：当前重点先在可用性，权限隔离会在知识库分区、检索过滤和工具白名单层继续补。

115. 有没有评测体系  
答：【需按实际补充】仓库里没有完整的离线评测集或 dashboard，所以我不会硬说已经建立完善评测。更合适的回答是：当前主要通过关键 case 回放和人工验证来做阶段性评估，下一步要补离线样本集和结果打分。

116. 是否做过人工标注、回放或 A/B  
答：【需按实际补充】代码里看不到完整 A/B 或标注平台。你如果实际做过小规模 case 对比、人工抽样验收或内测反馈，可以按真实经历补进去；如果没有，就实话实说目前以人工回放为主。

117. 如何判断“建议有用”  
答：对 OnCall 来说，“有用”至少要满足三点：建议能指向真实根因、能给出下一步可执行动作、不会误导用户走错方向。也就是说，它不是单看语言像不像，而是看它能不能帮助值班同学更快缩小排查范围。

118. false positive 和 false negative 怎么看  
答：在运维场景里，误判和漏判的代价不同。错误地给出一个看似确定的结论，风险往往比承认“不确定”更高，所以系统设计上宁可保守一点，也不要在证据不足时过度自信。

119. 检索结果和实时工具冲突怎么办  
答：我会优先信实时工具结果，再用知识库解释“为什么会出现这个现象”。知识库是经验和背景，日志与监控是当前现场，这个优先级不能反过来。

120. 如何表达不确定性  
答：我会要求系统把“不确定”明确说出来，并说明原因是“检索不到、工具失败还是证据冲突”。这比强行给出一个完整答案更专业，也更符合 OnCall 场景对可信度的要求。
