# 后端工程化、稳定性与可观测性

> 说明：
> - `【代码可证】` 表示可以直接从仓库代码、配置或文档中落到实现细节。
> - `【需按实际补充】` 表示仓库里没有团队人数、上线规模、业务指标等信息，面试前需要替换成你的真实经历。

131. 两个项目的核心后端服务  
答：【代码可证】自然资源平台的核心后端边界很清晰：`RagController/RagServiceImpl` 负责 RAG 代理，`FileParsingService` 负责 SHP 解析和空间关系计算，`KgGenerationService` 负责图谱生成，`Neo4jService` 负责图谱落库，Python `LightRAG` 负责抽取和查询；OnCall 项目则可以拆成对话服务、知识入库服务、向量检索服务、工具层和告警分析工作流。

132. 数据结构和索引设计  
答：【代码可证】自然资源平台在 Neo4j 侧以 `uniqueId` 做节点唯一约束，并按节点 Label 分组批量导入；OnCall 的 Milvus 侧固定了 `id/content/vector/metadata` 四类字段，上传文件会以 `_source + chunkIndex` 生成稳定主键，便于覆盖更新。也就是说，两边都在围绕“可去重、可批量导入、可追溯”来做索引设计。

133. MyBatis Plus 的作用  
答：【代码可证】RuoYi 底座使用 MyBatis Plus，适合处理系统基础表和常规业务 CRUD，能减少很多样板代码。它不适合替代 Neo4j 或向量库，因为图关系查询和向量相似度查询本来就不是关系 ORM 的强项。

134. MySQL 和 Neo4j 的边界  
答：【代码可证】MySQL 主要承载系统基座能力，比如用户、权限、配置、业务表；Neo4j 专门承载实体、关系和图谱查询。我的划分原则是：强事务、结构化业务数据进 MySQL；强关系、可视化和图遍历需求进 Neo4j。

135. Redis、MQ、搜索/向量库怎么回答  
答：【代码可证】自然资源平台底座配置了 Redis，但知识图谱主链路的缓存目前更多是进程内状态和 RuoYi 系统能力；向量库侧由 LightRAG 承担，可按部署环境切换实现。OnCall 项目则直接接了 Milvus 做知识检索。  
答：当前主链路没有明显引入 MQ，所以这题我会实话实说：还没把重任务完全队列化。

136. 什么时候一定要补 Redis 或 MQ  
答：当你要支撑更多并发构图任务、长时间文档处理任务或多实例部署时，就不能只靠本地内存和 `@Async` 了。这时候至少要把任务状态放到 Redis 或数据库里，再用 MQ 做削峰和重试，才能把系统做稳。

137. 如何做限流、隔离和降级  
答：我会从三层回答。接口层做上传大小、请求速率和会话数控制；任务层把构图和大模型调用拆成异步或独立线程池；模型/工具层做超时、重试和 fallback。  
答：【代码可证】当前已经有上传大小限制、模型超时和重试配置，但完整限流和隔离能力还有继续补的空间。

138. 大模型不稳定时如何兜底  
答：【代码可证】Java OnCall 配置了 DashScope 调用超时和重试，LightRAG 查询链路也有“检索不到就返回 fail_response”的分支。我的回答会是：当前先做超时和失败回传，后续可以再补缓存结果、简化模式和人工兜底。

139. 哪些接口要保证幂等  
答：文件上传入库、图谱构建提交、知识库重建、Agent 执行这类接口都应该考虑幂等。  
答：【代码可证】OnCall 文件上传已经通过 `_source` 删旧再重建实现了“同源覆盖更新”，自然资源平台 commit 流程则需要以 `taskId` 保证不会重复提交同一批图谱。

140. 如何避免重复执行  
答：【代码可证】OnCall Java 版上传时会先删同源数据再索引，Go 版也会按 `_source` 查 Milvus 再删除旧记录；自然资源平台则以 `taskId` 管理 build/preview/commit 流程。  
答：如果要更稳，我会再补任务状态机和幂等键，防止重复点击导致双写。

141. 慢查询、线程池、连接池打满怎么排查  
答：我一般按“症状 -> 日志 -> 指标 -> 依赖”四步排。先看是接口慢、任务慢还是模型慢，再看对应阶段日志和异常，再看数据库/Neo4j/Milvus/模型接口是否卡住，最后再判断是线程池、连接池还是外部依赖的问题。

142. 日志设计怎么做  
答：【代码可证】自然资源平台已经把空间任务拆成阶段化日志，OnCall 也会记录请求入口、工具调用、流式过程和最终完成。更好的做法是统一给每个请求和任务挂 `requestId/taskId/sessionId`，让“用户请求 -> 工具调用 -> 最终输出”能串成一条链。

143. 有没有 tracing、metrics 或告警平台  
答：【代码可证】当前仓库里主要体现的是结构化日志和 Prometheus 告警查询工具，并没有成体系接入 OpenTelemetry 之类的 tracing。我的回答会是：现阶段先靠日志和基础监控定位问题，后续会补 trace、调用耗时分布和错误码看板。

144. 如何串模型调用、检索调用和工具调用日志  
答：最好的方式是统一一个 trace id，再给每一段子调用打 span。  
答：【代码可证】当前代码已经有 sessionId、taskId、client_id 等关键上下文，只是还没有完全标准化成分布式 tracing，可以作为下一步工程化升级点。

145. 如何统计 token、耗时和失败原因  
答：我会把统计拆三层。模型层记录 token、prompt 长度和响应时延；检索层记录召回条数、向量搜索耗时和命中率；工具层记录工具名、参数、返回状态和失败原因。  
答：【需按实际补充】当前仓库没有看到完整的 token 计量看板，所以这部分建议你按真实观察值或后续方案来答。

146. 如果做多租户怎么设计  
答：我会从四层做隔离：鉴权层区分租户身份，数据层增加 tenant_id 或独立命名空间，索引层按知识库/租户拆分 collection 或 filter，图谱层按租户拆图库或标签。  
答：【代码可证】RuoYi 配置里 `tenant.enable=false`，说明底座预留了多租户，但当前知识图谱链路还没有完整启用。

147. 主要安全风险  
答：这类项目最典型的风险有四类。第一类是 Prompt Injection，把恶意文本带进模型；第二类是越权查询，把不该看的日志或文档查出来；第三类是敏感信息泄露，比如日志里带 token 或账号；第四类是高风险工具调用，比如把读工具误扩成写工具。  
答：【代码可证】OnCall Go 版里甚至预留了 `mysql_crud`，所以更要强调权限和审计。

148. 工具权限和审计措施  
答：【代码可证】当前主要还是靠工具白名单和接口层限制来控风险，读工具远多于写工具，Java 版也通过 MCP 配置显式注册工具。更完整的生产设计里，我会补“工具级权限、参数校验、调用审计和人工确认”四件套。

149. 如果正式上生产，最担心什么  
答：我最担心的是“结果不稳但用户误以为它很稳”。对这类系统来说，模型幻觉、检索偏差、工具超时和权限风险叠加起来，比单纯接口报错更危险，所以我会优先补评测、审计和降级，而不是只追求功能更多。

150. 当前最缺的一块工程化能力  
答：如果实话实说，我觉得最缺的是“稳定性工程”和“评测工程”。也就是任务状态机、失败重试、统一 tracing、效果评测和线上指标看板这些能力；功能原型已经有了，但要变成稳态生产系统，这一层还得继续补。
