31.  RAG 主流程<br>答：【代码可证】Java 端 `/system/rag/stream-query` 收到问题后，会通过 `RagServiceImpl` 把请求转发到 Python LightRAG 服务，再把结果以 `SseEmitter` 流式转回前端。LightRAG 内部会先做 query mode 判断，再走关键词提取、向量/图谱检索、上下文拼接和回答生成。
    
32.  文档切分策略<br>答：【代码可证】自然资源平台的 LightRAG 默认按 token 分块，`chunk_token_size=1200`、`chunk_overlap_token_size=100`；如果指定字符分割，也会在必要时再次按 token 拆分。这个策略比固定字符切分更稳，因为它直接对齐模型上下文。
    
33.  为什么这样切分<br>答：这个切分策略本质上是在“上下文完整性”和“召回颗粒度”之间取平衡。块太小会把专业语义拆散，块太大又会拉高噪声和 token 成本，所以我会说当前实现先采用 LightRAG 的稳妥默认值，再根据领域文档结构做微调。
    
34.  检索方式<br>答：【代码可证】LightRAG 支持 `local/global/hybrid/naive/mix` 多种模式，不是单一向量检索。`local` 更偏局部上下文，`global` 更偏全局关系，`hybrid` 同时考虑两者，这就天然支持文本检索和图谱检索的结合。
    
35.  召回与重排怎么讲<br>答：【代码可证】从 LightRAG 的实现看，先做关键词抽取和相关实体/关系/文本块召回，再按 query mode 组装上下文交给模型生成，严格意义上不是单独挂一个 reranker 服务。面试时我会实话实说：当前更像“多源召回 + 上下文控制”，如果追求更高精度，后续可以再接显式重排模型。
    
36.  专业术语和别名问题<br>答：这个问题我会从两层回答。第一层靠向量检索缓解术语变体问题，第二层靠图谱里的实体节点和关系增强语义对齐；如果某个专业术语存在简称、别名或多义，我会再补规则词典或实体规范化表，不完全依赖 LLM 自己猜。
    
37.  图谱在问答中的作用<br>答：【代码可证】图谱不是摆设，它承担了结构化关系补充的角色。纯文档检索能找到“某段话里提到了什么”，而图谱能补“这个实体和哪些实体有关系、关系类型是什么”，所以它更适合做关系扩展和语义约束。
    
38.  文本结果和图谱结果如何融合<br>答：我会把融合回答成“三步法”：先从向量检索拿到相关文本块，再从这些块或关键词映射到相关实体和关系，最后把文本证据和图谱关系共同拼成 prompt 上下文。这样回答既贴代码，又符合 LightRAG 的检索-上下文-生成思路。
    
39.  什么情况下图谱更有优势<br>答：当问题本身就是关系型的，比如“某类地块和某类规划对象是什么关系”“一个对象被哪些对象包含/邻接/相交”，图谱比纯文本检索更有优势。因为这类问题答案不是一句原文，而是结构化关系的组合。
    
40.  图谱什么时候会带来噪声<br>答：如果图谱 schema 不稳、实体抽取不准或者关系边过多，图谱会把错误结构放大。此时图谱不但不能提升问答，还会拉长检索链路，所以图谱适合在结构明确、关系可靠的子域里用，而不是所有问题都强行上图。
    
41.  实体和关系怎么抽取<br>答：【代码可证】LightRAG 里明确有实体关系抽取逻辑和 Prompt，抽取后再做合并、去重和摘要压缩。也就是说，当前主流程是“LLM 抽取为主，规则/结构校验为辅”，不是纯手写规则。
    
42.  图谱 schema 怎么设计<br>答：【代码可证】空间数据建图部分是通过 `NodeDefinition` 和 `RelationshipDefinitionDto` 配出来的，节点定义里有 `name/field/labelField/filterField/properties`，关系定义里有 `method/source/target/type/foreignKey` 等字段。换句话说，schema 不是写死的，而是可配置的。
    
43.  如何保证实体消歧和关系准确性<br>答：当前版本主要靠两层手段。第一层是在图谱生成时用业务定义约束节点类型、显示字段和过滤条件；第二层是在持久化前做去重和唯一约束，比如 Neo4j 侧按 `uniqueId` 建约束。<br>答：【需按实际补充】如果你线上还有人工校验或抽样验收流程，也可以补上。
    
44.  增量更新如何处理<br>答：【代码可证】严格说当前自然资源图谱主链路偏全量构建，因为 `KgGenerationService` 里会先 `clearDatabase()` 再 `saveGraph()`。所以我会诚实回答：当前版本先保证构建正确性，增量更新是后续要补的方向。
    
45.  文本索引和图谱索引一致性<br>答：现在的一致性是通过任务流来保证的，而不是靠分布式事务。也就是先在 Python 侧完成文本处理和图谱预览，再由用户触发 commit，把最终图谱同步到 Neo4j；这保证了“同一批知识”的文本和图谱不会乱写，但还不是强事务式的一致性模型。
    
46.  如何评估融合效果<br>答：如果面试官追这个，我会答成两层。离线看召回质量、关系覆盖度和答案命中率，线上看问答可用率、人工反馈和引用有效性；如果没有完整评测系统，也要能说清你至少做过 case 对比，比较过纯向量检索和图谱增强后的回答差异。
    
47.  回答错误时如何定位问题<br>答：我一般按“切分 -> 召回 -> 关系扩展 -> prompt -> 模型输出”的顺序排。先看相关 chunk 有没有召回，再看相关实体和边有没有进上下文，如果上下文没问题再排 prompt 和模型本身，这样定位会更高效。
    
48.  ROI 最高的优化点<br>答：对这个项目来说，我认为 ROI 最高的通常不是换更大的模型，而是做更稳的 schema、别名归一化和检索策略调优。因为自然资源场景知识相对专业，数据质量和检索链路往往比单纯换模型更有效。
    
49.  是否做了引用溯源<br>答：【代码可证】系统在存储里保留了文档、chunk、file path 等元信息，具备做溯源的基础。<br>答：【需按实际补充】但当前仓库没有看到非常完整的前端“引用展示”交互，所以面试时可以回答成“后端元数据链路已经具备，前端展示可以继续加强”。
    
50.  如何降低幻觉<br>答：【代码可证】第一，查询必须先走检索，不是直接裸问模型；第二，检索不到时服务会明确返回“知识库里没有相关信息”；第三，图谱和文本一起提供上下文，减少模型凭空补全。再往后优化的话，我会补引用展示、低置信度拒答和评测回放。
