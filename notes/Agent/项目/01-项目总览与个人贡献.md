# 项目总览与个人贡献

> 说明：
> 
> -   `【代码可证】` 表示可以直接从仓库代码、配置或文档中落到实现细节。

> -   `【需按实际补充】` 表示仓库里没有团队人数、上线规模、业务指标等信息，面试前需要替换成你的真实经历。

1.  整体介绍<br>答：【代码可证】我主要做了两个方向的项目。自然资源知识服务平台是基于 RuoYi 的前后端分离系统，Java 后端负责文档接入、SHP 解析、空间关系计算、知识图谱生成与 Neo4j 持久化，Python LightRAG 负责分块、实体关系抽取和查询；智能 OnCall Agent 项目则是面向运维场景的智能体应用，第一版用 `Spring Boot + Spring AI + Milvus` 做 RAG 问答和多 Agent 运维，后续升级成 `Go + Eino + Plan-Execute-Replan`，重点强化了规划执行链路和流式交互。
    
2.  为什么做自然资源平台<br>答：【代码可证】这个项目明显是围绕专业领域知识服务和空间数据处理来的，不是通用聊天。仓库里既有 `RagController/RagServiceImpl` 这条文档问答链路，也有 `FileParsingService/KgGenerationService` 这条空间数据建图链路，说明目标是把自然资源行业里的文档资料、空间要素和关系统一成可检索、可问答、可分析的知识底座。
    
3.  为什么做 OnCall Agent<br>答：【代码可证】OnCall 项目针对的是真实的运维痛点，而不是只做一个聊天壳。按当前仓库成熟度看，Java 版已经把内部文档、Milvus 检索、Prometheus/日志工具、SSE 输出和多 Agent 编排串成了一条可运行链路；Go 版则进一步补上了 `Plan-Execute-Replan` 的执行框架和 SSE 交互，但部分外部工具仍依赖环境配置或后续完善。整体目标很明确，就是缩短告警发现后的定位和处理时间，减少纯人工排障。
    
4.  个人负责模块<br>答：【代码可证】如果按仓库实现来讲，我可以把自己的工作描述为两块。自然资源平台侧，我负责了 RAG 接口代理、LightRAG 集成、SHP 解析、空间关系计算、知识图谱生成和 Neo4j 同步；OnCall 侧，我负责了对话接口、流式输出、会话记忆、知识入库、Milvus 检索、工具接入以及告警分析工作流。<br>答：【需按实际补充】哪些模块是你主导、哪些是协作完成，需要你按真实情况再细化。
    
5.  团队人数与角色<br>答：【需按实际补充】代码仓里看不出确切团队人数和汇报关系，这类问题不要编。建议你按真实经历回答成“我负责后端主链路/算法集成/前后端联调中的哪几块”，重点讲 ownership，而不是泛泛说“都参与了”。
    
6.  最能体现深度的部分<br>答：【代码可证】自然资源项目里最能体现技术深度的是空间关系计算和图谱生成链路，因为这里同时涉及数据建模、算法复杂度、并行处理和图数据库落库。OnCall 项目里最能体现深度的是工具调用和规划执行链路，因为它不只是问答，而是把 RAG、MCP/工具、记忆和多轮执行设计成闭环，且 Java 版已经跑通一版，Go 版又把规划执行进一步结构化。
    
7.  更适合证明后端能力的项目<br>答：【代码可证】如果投后端开发岗，我会优先讲自然资源知识服务平台。这个项目的后端信号更强，既有 Java 服务编排，也有异步任务、缓存、Neo4j 批量写入、空间索引、并行计算、SSE/WebSocket、多存储协同，面试官更容易追问到系统设计和性能优化。
    
8.  更适合证明 Agent 能力的项目<br>答：【代码可证】如果投 Agent 应用开发岗，我会优先讲 OnCall Agent。因为它代码里就直接体现了 `ReactAgent`、工具调用、会话记忆、流式输出、Spring AI MCP 接入，以及 Go 版本里的 `Plan-Execute-Replan`，和岗位画像更贴。
    
9.  两个项目最难的问题<br>答：【代码可证】自然资源平台最难的是把大规模空间数据处理和知识图谱构建结合起来，核心难点在空间关系计算复杂度和图谱落库效率；OnCall 项目最难的是让 Agent 从“会回答”变成“能执行”，也就是如何把告警、文档、日志和工具调用放进可控的工作流里，而不是让模型自由发挥。
    
10.  是否上线、服务范围<br>答：【需按实际补充】仓库里没有直接记录上线范围、用户数量和内部覆盖团队。这类问题建议你按真实情况回答成“已在内部试用/已进入某团队工作流/还处于原型验证”，同时配一个真实场景，不要只说“已经上线”。
    
11.  如何证明不是 demo<br>答：【代码可证】自然资源平台不是只做页面展示，它有文件上传、异步构建、LightRAG 服务代理、Neo4j 持久化、SHP 空间关系计算、WebSocket 进度推送和完整的图谱预览/提交流程。OnCall 也不是单轮对话 demo，它有文件入库、Milvus 检索、工具集、会话记忆、SSE 流式接口和多版本演进路径。
    
12.  如何定义成功<br>答：我会把成功拆成两层。第一层是技术闭环是否完整，比如文档能否稳定入库、图谱能否稳定生成、检索和工具链路是否可用；第二层是业务闭环是否有效，比如自然资源项目能否让专业资料更快被查询和关联，OnCall 项目能否减少人工排障时间。<br>答：【需按实际补充】如果你有真实指标，比如构图耗时、问答可用率、定位时长下降比例，面试时一定要补上。
    
13.  做过哪些取舍<br>答：【代码可证】自然资源平台的取舍是 Java 负责业务编排，Python LightRAG 负责抽取和查询，而不是把所有东西都塞进一个服务里；图谱构建也分成“预览”和“提交到 Neo4j”两阶段，避免重任务直接阻塞用户。OnCall 的取舍是先用 Spring AI 快速把 RAG + 工具链跑通，再在 Go 版本里升级成更可控的 Plan-Execute-Replan 架构。
    
14.  如果重做会优先重构什么<br>答：【代码可证】自然资源平台我会优先把 `DataSourceCacheService` 这类进程内缓存升级成可持久化或可恢复的任务状态存储，并补上增量构图能力；OnCall 我会优先把工具调用、评测和失败兜底做成更标准的框架能力，而不是散落在具体实现里。
    
15.  最大的坑<br>答：【代码可证】自然资源平台最大的坑很可能就是空间计算的性能和大文件处理，因为代码里已经专门用 `STRtree + PreparedGeometry + parallelStream` 去优化，说明这里确实踩过；OnCall 项目最大的坑则是工具链路和外部依赖的稳定性，比如日志工具、Prometheus、MCP、模型超时，这也是为什么配置里有超时、重试、mock 模式和多版本演进。
