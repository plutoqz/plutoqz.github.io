# 智能 OnCall Agent：ReAct、规划执行与工具调用

> 说明：
> - `【代码可证】` 表示可以直接从仓库代码、配置或文档中落到实现细节。
> - `【需按实际补充】` 表示仓库里没有团队人数、上线规模、业务指标等信息，面试前需要替换成你的真实经历。

86. 为什么选 ReAct  
答：【代码可证】Java 版显式用了 `ReactAgent`，因为这个场景天然需要“思考一下接下来该调哪个工具，再根据工具结果继续回答”。和纯 Prompt 问答相比，ReAct 更适合把工具调用纳入同一轮推理。

87. ReAct 的执行过程  
答：【代码可证】Java 版里用户请求先进入 `ChatController`，再由 `ChatService` 创建 `ReactAgent`，系统提示里明确规定哪些问题该调用哪个工具；执行时 Agent 自动选择工具，流式接口会把模型输出、工具完成和最终结果按事件推给前端。

88. 为什么用 Plan-Execute-Replan  
答：【代码可证】Go 版用了 Eino 的 `planexecute.New`，说明项目已经从“单智能体即时反应”升级到“先规划、再执行、再重规划”。这更适合告警分析这种多步骤任务，因为执行中拿到的新证据可能会改变后续计划。

89. 它相对固定工作流的收益和代价  
答：收益是更灵活，能针对不同告警和不同工具结果动态调整。代价是控制难度更高，需要处理循环、工具失败、空结果和终止条件，所以对日志、评测和兜底要求更高。

90. 什么任务直接执行，什么任务先规划  
答：单步、确定性强的问题可以直接执行，比如“当前有哪些活跃告警”；需要多工具联动和证据汇总的问题则更适合先规划，比如“告警为什么触发、下一步该查什么、最终报告怎么组织”。这也是 Go 版把 AIOps 链路单独放进 `BuildPlanAgent` 的原因。

91. 规划结果的形式  
答：【代码可证】Java 版 Planner 的提示词要求在中间阶段输出 JSON，包括 `decision/step/tool/context`；Go 版的 `planexecute` 则把 Planner、Executor、Replanner 的消息作为事件流推进。也就是说，中间态是结构化的，最终态再转成面向用户的 Markdown 报告。

92. 如何把模型输出映射成工具调用  
答：【代码可证】Java 版依赖 Spring AI 的 `methodTools` 和 `ToolCallbackProvider`，模型只要选中工具名，就能映射到具体方法或 MCP 工具；Go 版则把工具注册到 `ToolsNodeConfig`，由框架负责把模型输出映射成工具执行。

93. 参数生成和校验  
答：这题我会答成“约束 + 兜底”。约束来自工具 schema 和系统提示，比如地域参数必须是 `ap-guangzhou` 这种格式；兜底来自默认值、错误分支和失败信息回传。  
答：【代码可证】Java 和 Go 的提示词里都专门强调了 region 参数格式。

94. 如何避免死循环和重复规划  
答：【代码可证】Go 版 `PlanExecuteAgent` 设置了 `MaxIterations: 20`，Executor 也会在提示里要求遇到连续失败时尽快返回失败；Java 版的 Planner/Executor/Supervisor 提示词同样要求三次失败后停止。也就是说，当前是“靠框架迭代上限 + 提示词终止规则”双重约束。

95. 什么情况下触发 replan  
答：当执行结果和原计划不一致，或者工具返回了新的关键证据时，就要 replan。比如查到日志里不是 CPU 高，而是数据库连接池耗尽，那后续步骤就不能继续沿着原来的方向走。

96. 工具调用是串行还是并行  
答：【代码可证】从当前实现看，主路径更偏串行决策，即先拿一步结果，再决定下一步。这样虽然没法极限压缩时延，但更容易保证证据链清楚，也更适合告警分析这种需要逐步收敛结论的任务。

97. 多工具结果冲突怎么办  
答：我会优先相信实时工具数据，再用历史知识做解释。比如文档是静态经验，告警和日志是实时事实；如果二者冲突，就应该把冲突本身写进结论，而不是为了“回答完整”强行统一。

98. 如何处理超时、失败、空结果  
答：【代码可证】当前实现主要靠两层。第一层是配置级超时、重试和 mock 模式；第二层是 Planner/Executor 提示里明确要求遇到失败和空数据要如实记录，连续多次失败要停止当前方向。这样至少不会把失败结果伪装成成功。

99. 是否支持人工接管  
答：【代码可证】当前仓库没有做完整的人机协作审批流，但接口层已经能把执行明细、证据和最终报告返回给前端。我的回答会是：当前版本默认是“AI 先给建议，人来决定是否执行”，真正的自动执行需要更严格的权限与审计机制。

100. 如何设计工具描述  
答：【代码可证】当前工具描述都尽量写明“什么时候该调用它”和“它返回什么”，比如文档检索工具会说明它用于检索内部文档，Prometheus 工具说明它查活动告警，日志工具会先暴露 topic 列表。这种描述比只给工具名更容易让模型选对。

101. 工具很多时怎么控制噪声  
答：我一般会从三层做控制。第一层是按场景拆工具集，不把所有工具都塞进一个 Agent；第二层是系统提示明确路由规则；第三层是对高风险工具做单独的调用门槛。  
答：【代码可证】当前 Java 和 Go 的工具集其实都比较收敛，主要围绕文档、告警、日志和时间。

102. 是否做过工具路由和分层编排  
答：【代码可证】Java 版里系统提示已经隐式做了工具路由，指定不同问题该走哪个工具；Go 版则更进一步，把 Planner、Executor、Replanner 分层，让“决定做什么”和“真正去执行”分开，这本身就是一种编排分层。

103. 多智能体怎么分工  
答：【代码可证】Java 版里是 `Planner Agent + Executor Agent + Supervisor Agent`，Supervisor 负责调度 Planner 和 Executor；Go 版里则是 `Planner + Executor + Replanner`。换句话说，职责划分很清楚：一个负责任务拆解，一个负责拿证据执行，一个负责根据新结果改计划。

104. 如果改回单 Agent 会怎样  
答：单 Agent 能做，但复杂任务的可控性会差很多。因为规划、执行和总结都混在一起后，很难追踪它为什么调错工具、为什么卡在某一步，也不容易对中间态做日志和评测。

105. 如何评估任务完成率  
答：我不会只看回答是否流畅，而会看“是否完成了预期动作”。比如有没有查到活跃告警、有没有拿到对应文档、有没有给出基于证据的根因分析、有没有产出结构化报告；这比看语言自然度更接近真实业务价值。
