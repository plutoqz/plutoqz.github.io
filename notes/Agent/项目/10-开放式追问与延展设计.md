# 开放式追问与延展设计

> 说明：
> - 这一部分题目本身就偏设计题，我用“面试现场可直接说”的方式来写。
> - 如果你有真实业务数据或真实限制条件，优先替换成你的实际情况。

151. 把自然资源平台迁到通用企业知识中台  
答：我会把当前实现拆成通用能力层和行业插件层。通用层负责文档接入、解析、索引、图谱持久化和问答接口，行业层再去扩展空间数据解析、专业 schema 和专业规则，这样平台本身才能复用到别的行业。

152. OnCall Agent 支持跨团队协作  
答：我会把“事件”和“会话”分离。事件是一次告警工单，跨团队共享；会话是每个团队围绕该事件的分析上下文；再在权限上控制谁能看哪些日志、谁能调用哪些工具、谁能提交最终处置意见。

153. 告警量暴增 10 倍怎么办  
答：第一步不是上更大模型，而是先做分层处理。高频简单告警走规则或轻量策略，复杂告警再进 Agent；同时把工具调用异步化、知识检索缓存化，并把模型调用限流，避免所有告警都挤进同一条高成本链路。

154. 模型成本上涨怎么降本  
答：我会先从检索和工作流入手，而不是直接降模型能力。具体包括：减少无效召回、缩短上下文、优先调用工具拿确定事实、对简单问题走轻量模型、缓存热点问题和热点文档回答。

155. 准确率上不去先改哪里  
答：先看数据和检索，再看 prompt 和模型。因为大多数 Agent/RAG 项目准确率问题，最后都能归结到“没拿到对的上下文”或者“拿到了太多噪声上下文”，这通常比单纯换模型更关键。

156. “建议很多但不敢执行”怎么办  
答：这说明系统缺的不只是智能，还缺可解释性和风险分级。我会把建议拆成“证据、判断、动作、风险”四段，再给动作打等级，比如“可直接执行/需人工确认/仅供参考”，让用户知道自己该怎么用。

157. 经常给出看似合理但错误的结论怎么排查  
答：我会先排检索和工具链，再排模型。因为如果上下文本身就错了，模型越聪明越会把错误说圆；只有确认上下文和工具证据没问题后，才去优化提示词或更换模型。

158. 现场设计一个告警自动排障 Agent  
答：我会画五层。最上层是接入层，接 Prometheus/日志/工单；第二层是工作流层，负责 Plan-Execute-Replan；第三层是知识层，负责文档库和历史 case；第四层是工具层，负责日志、指标、数据库等只读工具；最下层是治理层，负责权限、审计、评测和降级。

159. 如何比较 ReAct、工作流编排、Plan-Execute-Replan、多智能体  
答：我的取舍标准是任务复杂度和可控性。简单工具问答用 ReAct 就够，确定性流程适合工作流编排，复杂多步任务适合 Plan-Execute-Replan，需要职责分离和中间态治理时再上多智能体，不会一上来就堆最复杂方案。

160. 如何比较 RAG、知识图谱、GraphRAG、纯微调  
答：我会按“成本、更新频率、关系复杂度”三个维度来讲。文档型问题优先 RAG，关系型问题加图谱，既要文档又要关系时考虑 GraphRAG；纯微调适合稳定知识和风格适配，但不适合替代高频更新知识库。

161. 两个项目合成一个统一平台  
答：我会抽公共底座，而不是简单拼功能。公共底座包括数据接入、索引管理、任务调度、流式输出、工具注册和评测体系；自然资源和 OnCall 则作为两个业务域插件，共享底座但各自保留 schema、工具和界面。

162. 有无沉淀可复用能力  
答：【代码可证】其实已经有一些雏形了，比如自然资源平台里的“任务编排 + 进度推送 + 图谱落库”能力，OnCall 里的“工具注册 + 会话记忆 + 流式交互 + 知识入库”能力。面试时我会强调，这些都可以继续抽成中台组件，而不是局限在单项目里。

163. 真正的技术壁垒是什么  
答：我会把壁垒讲成“把复杂场景做成稳定闭环”的能力，而不是会几个术语。自然资源项目的壁垒在于专业空间数据和知识图谱的结合；OnCall 项目的壁垒在于把 RAG、工具和工作流变成可用的运维助手，而不是只会做聊天界面。

164. 给新人讲这两个项目先讲哪三件事  
答：第一讲业务目标，不先讲模型。第二讲主链路，也就是请求怎么流过系统、数据怎么进索引和图谱。第三讲风险点，包括性能瓶颈、工具稳定性、权限和评测，因为这些决定项目能不能从 demo 走向生产。

165. 最怕被继续深挖哪一块  
答：最怕的通常不是自己做过的难点，而是那些简历写得很大、但实际落得还不够深的点。比如如果你写了 `Multi-Agent/MCP/GraphRAG`，就一定要确保自己能讲清调用链、失败处理和为什么这么选；讲不深的点宁可收敛，也不要写得太满。
