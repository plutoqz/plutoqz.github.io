# 自然资源知识服务平台：空间数据处理与性能优化

> 说明：
> - `【代码可证】` 表示可以直接从仓库代码、配置或文档中落到实现细节。
> - `【需按实际补充】` 表示仓库里没有团队人数、上线规模、业务指标等信息，面试前需要替换成你的真实经历。

51. 空间关系计算在算什么  
答：【代码可证】系统主要在算地理要素之间的 `TOUCHES / OVERLAPS / CONTAINS / WITHIN / CROSSES` 等空间关系，然后把这些关系转成图谱边。典型场景就是把地块、规划单元、设施等对象的空间邻接和包含关系结构化表达出来。

52. 为什么空间数据会成为瓶颈  
答：【代码可证】因为朴素做法是要素两两比较，复杂度接近 `O(n²)`，而每一次几何判断又不是简单字符串比较，而是真正的几何计算。数据一大，CPU 和内存都会明显上升，文件读取和临时目录解压也会拖慢整体链路。

53. 为什么用 STRtree  
答：【代码可证】因为它能先按包络框做候选过滤，把“全量两两比较”改成“先通过空间索引找到邻近候选，再做精算”。`FileParsingService` 里已经是先建 `STRtree`，后续对每个要素只查索引命中的候选集合。

54. STRtree 的原理和价值  
答：它本质上是一个 R-Tree 家族的空间索引结构，先按空间包围盒组织数据，再通过树形结构快速缩小候选范围。对这个项目来说，它最大的价值不是“算得更准”，而是“少算了大量根本不可能相交的对象”。

55. 不用 STRtree 的朴素方案  
答：【代码可证】朴素方案就是双层循环把所有要素两两比较。报告里已经举了例子：一万要素下比较次数可能到一亿量级，这对几何运算来说非常贵。

56. 为什么用 PreparedGeometry  
答：【代码可证】因为很多空间谓词都需要重复利用某个几何对象的中间结果，`PreparedGeometry` 可以把这些中间结构预先准备好。代码里就是为 `feature1` 创建 `PreparedGeometry`，然后反复对候选 `geom2` 做 `intersects/contains/within` 判断。

57. PreparedGeometry 提升来自哪里  
答：它的提升来自“把复杂几何对象的重复计算前置并缓存”。尤其在多边形边界复杂、同一个 geometry 需要和很多候选几何重复比较时，收益很明显；但像 `touches/overlaps` 这类判断不一定都能完全用 prepared 版本替代，所以代码里仍然保留了部分原始几何调用。

58. 空间索引的粒度  
答：【代码可证】当前索引粒度是“要素级”。`FileParsingService` 读取 `SimpleFeature` 后，会对每个要素取默认 geometry 的包络框并插入 `STRtree`，不是按整个文件或整个图层只建一个粗索引。

59. 索引本身会不会成为瓶颈  
答：会，但比全量关系计算轻得多。当前实现是先一次性把要素读入内存，再建索引、再做关系计算，所以当数据继续放大时，索引构建和内存占用也会成为问题，这时候就要考虑分批、分块或者离线预处理。

60. 并行计算怎么做  
答：【代码可证】当前实现用的是 `featuresList.parallelStream()`，按要素维度并行处理，每个线程基于索引查候选、再计算关系，最后汇总为一个关系列表。这种方式利用了多核 CPU，又避免自己手写线程池拆分逻辑。

61. 如何避免线程争用和内存暴涨  
答：【代码可证】一方面，候选查询是基于只读索引和局部列表，不需要大量共享写；另一方面，每个并行分支只累积当前要素的 `foundRelations`，最终再统一收集。  
答：如果要再往前走，我会控制批大小、限制并行度、并把大任务拆成分片，避免一次把所有 feature 都压进内存。

62. 异步构建任务模型  
答：【代码可证】SHP 解析是 `@Async` 异步执行的，任务状态由 `StorageService` 管理，前端通过 `taskId` 轮询或订阅进度。当前实现更像“异步方法 + 状态存储 + SSE/WebSocket 通知”，还不是完整 MQ 型任务队列。

63. 如何避免请求线程阻塞  
答：【代码可证】文件先暂存，真正的解析和空间计算由异步方法执行，接口只返回 `taskId` 和状态信息。这样上传和解析被拆开了，请求线程不需要一直等到关系矩阵算完。

64. 如何定义和处理超时  
答：当前代码里没有非常重的“超时恢复状态机”，更多是通过异步执行、状态更新和前端进度监听来规避超长阻塞。面试时我会坦诚说明：这个版本主要先解决性能和异步化，下一步再补任务超时控制、重试和取消机制。

65. 任务执行中途失败怎么办  
答：【代码可证】当前会把任务状态置为 `FAILED`，并记录错误信息、清理临时文件和解压目录。  
答：严格意义上的断点续跑还没在这条 SHP 解析链路里看到，所以这题我会回答成“当前以失败回滚和重新提交为主，断点续跑是下一步优化方向”。

66. 如何监控构图和空间计算耗时  
答：【代码可证】代码里已经把流程拆成多个阶段打印耗时和内存，包括解压、找文件、读要素、建索引、算关系、Neo4j 持久化。也就是说，现阶段的可观测性主要依赖结构化日志和阶段化埋点。

67. 优化前后的指标  
答：【代码可证】仓库里的性能分析报告给了量级估算：千级数据能从分钟级降到秒级，万级数据能从小时级降到分钟级，核心收益来自 `STRtree + PreparedGeometry + 并行流`。  
答：【需按实际补充】如果你自己做过真实压测，一定要用真实数据替换报告中的估算表。

68. 如何定位性能瓶颈  
答：【代码可证】当前主要靠阶段耗时日志和内存日志来定位瓶颈，代码里会在每个阶段打印耗时、当前内存和总体耗时。面试时我会强调：先用日志把问题定位到“读文件、建索引还是算关系”，再决定是否上 profiler。

69. 如果数据再扩大 10 倍  
答：我会优先改两层。第一层是计算层，把一次性全内存处理改成分批或分区；第二层是任务层，把构图任务拆成可恢复的离线任务，必要时引入更强的队列和分布式执行，而不是继续依赖单机并行流。

70. 哪些优化可能不值得做  
答：如果数据规模还在中小量级，一上来就做特别重的分布式改造不一定划算。这个项目里最值钱的其实是先把 `STRtree` 和 `PreparedGeometry` 用好，因为它们改动小、收益大；相比之下，过早引入复杂调度系统未必是第一优先级。
