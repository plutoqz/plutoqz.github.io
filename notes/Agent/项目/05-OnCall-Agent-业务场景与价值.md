# 智能 OnCall Agent：业务场景与价值

> - `【需按实际补充】` 表示仓库里没有团队人数、上线规模、业务指标等信息，面试前需要替换成你的真实经历。

71. 业务背景和目标用户  
答：【代码可证】这个项目面向的是内部运维、SRE 或 OnCall 同学，不是普通客服问答。项目目标是把内部文档、告警、日志和时间工具接到统一入口，让值班同学在告警出现后更快完成“看告警、查文档、查日志、给结论”的闭环；从代码成熟度看，Java 版这条链路更完整，Go 版重点强化了规划执行框架。

72. 优化了传统 OnCall 的哪些环节  
答：传统人工 OnCall 往往是先看告警，再去翻文档、手动查日志、再整理结论，路径很长。当前系统把这些步骤收敛到一个 Agent 入口里，至少把“文档检索、告警读取、日志查询、报告输出”这几件事统一了。

73. 支持哪些场景  
答：【代码可证】从接口设计看，至少有三类场景。第一类是普通业务问答和内部知识咨询，对应 `/api/chat` 和 `/api/chat_stream`；第二类是告警自动分析，对应 `/api/ai_ops`；第三类是知识上传入库，对应 `/api/upload` 或 Go 版的 `FileUpload`。

74. 为什么适合用 Agent  
答：【代码可证】因为这个场景不是简单 FAQ，而是需要根据问题动态决定“查文档、查告警、查日志还是给出总结”。代码里已经把工具选择、对话记忆和工作流规划放进 Agent 里，说明场景本身就是多步决策型任务。

75. 自动应答和智能排查的边界  
答：我会把边界讲得很清楚。自动应答偏“知识解释和已有流程返回”，比如根据内部文档回答该怎么做；智能排查偏“基于实时告警和日志去判断问题原因”，它需要工具调用和多步分析，风险也更高。

76. 告警进入系统后的处理链路  
答：【代码可证】Java 版里 `/api/ai_ops` 会先创建模型和工具，再由 `AiOpsService` 启动 Planner/Executor/Supervisor 的多 Agent 编排，最终输出《告警分析报告》；Go 版里 `/api/ai_ops` 直接把任务交给 `BuildPlanAgent`，由 `Planner + Executor + Replanner` 迭代执行，最后返回结果和执行明细。

77. 接入了哪些数据源和工具  
答：【代码可证】当前最明确跑通的是内部文档检索、当前时间工具、Milvus 知识检索，以及 Java 版里的 Prometheus 告警查询和日志查询链路。Java 版通过 `methodTools + ToolCallbackProvider` 同时接本地工具和 Spring AI MCP，配置里能看到腾讯云 MCP SSE 连接；Go 版也定义了内部文档、Prometheus、时间工具和日志 MCP 的接入位，并预留了 `mysql_crud`，但像日志 MCP URL、Prometheus 实查这类外部依赖还需要按实际环境补齐后才能完整跑通。

答：【代码可证】从代码结构看，这个项目至少不是纯脚本实验，因为它有前端接口、会话管理、知识入库、工具调用和流式输出。  
答：【需按实际补充】但是否已经被某团队长期使用、是否接过真实告警，需要你用自己的真实试用或落地情况去补充。

79. 核心效果指标  
答：我会把指标分成四类。第一类是效率指标，比如首次定位时间、生成初步结论时间；第二类是质量指标，比如建议采纳率、误报率；第三类是稳定性指标，比如工具调用成功率和模型超时率；第四类是成本指标，比如节省的人力排查时间。  
答：【需按实际补充】具体数字代码仓没有，需要你补真实值。

80. 最能体现 Agent 工程化价值的一点  
答：【代码可证】我认为最能体现工程化价值的是它把“检索、工具、记忆、流式交互、告警分析”放在一个可运行链路里，而不是只做一个 prompt demo。尤其 Go 版的 `Plan-Execute-Replan`，说明项目已经开始从单轮智能问答往“任务执行器”演进。

81. 如何反驳“套壳问答系统”  
答：我会直接用链路反驳。这个项目不是只把用户问题喂给模型，而是先决定要不要调工具、查知识、读告警、查日志，再把证据整合成结论；以 Java 版主链路为例，它已经有工具注册、记忆、文件入库、Milvus 检索、MCP 接入和多 Agent 分工，Go 版又补上了规划执行框架，这些都不是单纯套壳聊天能替代的。

82. 常见失败案例  
答：最常见的失败点通常在三类地方。第一类是检索不到足够证据；第二类是外部工具超时或数据为空；第三类是模型把“看起来合理”的推断当成事实。  
答：【代码可证】这也是为什么配置里有超时、重试、mock 模式和异常分支。

83. 如何提高用户信任  
答：我会强调三件事。第一，尽量返回证据来源，比如具体日志和具体文档；第二，让系统在不确定时明确说“不确定”；第三，把工具查询和最终结论区分开，让用户知道哪些是事实、哪些是推断。

84. 为什么比普通客服更难  
答：因为 OnCall 场景的错误代价更高，而且它面对的是实时、动态、强上下文的问题。客服更多回答已有知识，OnCall 还要处理告警、日志、环境差异、工具失败和结果可执行性，所以对 Agent 的稳定性要求更高。

85. 如果继续增强会做什么  
答：我会优先做三件事。第一，补评测体系和 case 回放，让建议质量可以持续量化；第二，补工具权限和审计，让执行链更安全；第三，把知识库、历史工单和实时数据做更好的融合，而不是只依赖单次检索。
