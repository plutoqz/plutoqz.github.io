16.  平台的完整业务链路<br>答：【代码可证】平台的主链路是“文档/空间数据接入 -> 解析预处理 -> RAG 或图谱构建 -> 图谱预览/提交 -> 前端查询与问答”。文档侧通过 `/system/rag/build/start` 把文件转发给 Python LightRAG，查询侧通过 `/system/rag/stream-query` 做 SSE 流式问答；空间数据侧先解析 SHP，再算空间关系、缓存数据源，最后根据节点和关系定义生成图谱并写入 Neo4j。
    
17.  为什么要同时用 RAG 和知识图谱<br>答：【代码可证】因为这不是单纯的文本问答场景。RAG 更适合解决“从大段文档里找相关内容”的问题，知识图谱更适合表达“实体-关系-结构化约束”，尤其在自然资源这种强关系、强专业语义的领域里，两者结合比只做向量检索更稳。
    
18.  地理空间场景的特殊性<br>答：【代码可证】这个项目不只是处理 PDF/Word/TXT，还要处理 SHP 这类空间数据，并把“相邻、包含、相交、穿越”等空间关系转成图谱边。也就是说，它除了文本语义，还要理解几何关系和业务属性，这就是和普通知识库问答的最大差别。
    
19.  支持哪些文档类型<br>答：【代码可证】从系统文档和链路看，平台支持 PDF、Word、TXT 等文档用于 LightRAG 构建，也支持 Shapefile、GeoJSON 等空间数据用于半自动知识图谱构建。两条链路的数据类型不同，一条偏文本语义，一条偏结构化空间要素。
    
20.  文档上传后的处理步骤<br>答：【代码可证】文档先由 Java 后端接收并转发到 Python 服务，LightRAG 负责文档解析、分块、向量化、实体关系抽取和多存储写入；Java 端再通过状态查询、图谱预览和 commit 把结果同步回 Neo4j。也就是说，Java 负责编排，Python 负责 AI 核心处理。
    
21.  为什么选 Neo4j<br>答：【代码可证】因为这个项目要长期存实体和关系，并支持图式查询和可视化。`Neo4jService` 里专门做了节点/边批量导入、唯一约束、按 Label 分组、APOC 关系 merge，这说明 Neo4j 不是临时演示，而是图谱持久化的核心存储；如果换成 MySQL 或 ES，关系表达和图查询体验都会差很多。
    
22.  前后端和多语言分工<br>答：【代码可证】前端是 Vue 3 体系，`ruoyi-f` 里能看到 `vue3`、`vben`、`ant-design-vue`、`pinia`、`fetch-event-source`、图可视化相关依赖；Java 后端负责业务 API、鉴权、任务编排、Neo4j 持久化和 SHP 处理；Python LightRAG 负责 chunk、embedding、entity/relation extraction 和 query。这个分工比较清晰，适合把工程能力和 AI 能力拆开。
    
23.  为什么同时用 Java 和 Python<br>答：【代码可证】因为 Java 更适合承接 RuoYi 业务框架、权限体系和工程化接口，而 Python 更适合直接复用 LightRAG 的抽取和检索能力。实际代码也是这么落的，Java 端通过 `RestTemplate/WebClient` 代理 Python API，既没有强行把 AI 逻辑写进 Java，也没有把业务系统都搬到 Python。
    
24.  系统架构形态<br>答：【代码可证】这套系统更像“前后端分离 + AI 微服务”的混合架构，而不是纯单体。`ruoyi-b` 是 Spring Boot 后端，`ruoyi-f` 是独立前端，`LightRAG` 是独立 Python 服务，Neo4j/MySQL/Redis 分别承担不同存储角色，所以我会描述为“以 Spring Boot 为业务主服务、以 LightRAG 为算法微服务的分层架构”。
    
25.  文档处理、图谱构建、检索是同步还是异步<br>答：【代码可证】文档构建明显是异步的。`startBuild` 返回 `taskId` 后，Java 后台异步轮询 Python 状态，通过 WebSocket 推进度，最后用户再触发 commit；查询则是同步发起、流式返回结果。这样的设计避免了大任务阻塞请求线程。
    
26.  多用户大文件上传时的瓶颈<br>答：【代码可证】最容易打满的是 Python 文档处理和空间关系计算链路，因为它们涉及文件 IO、文本分块、LLM 抽取、向量化和几何运算。Java 端更多是代理和编排，真正重的是 Python 服务、Neo4j 写入和 SHP 几何计算。
    
27.  权限隔离怎么回答<br>答：【代码可证】系统底座用了 Sa-Token，说明基础鉴权是有的。<br>答：【需按实际补充】但从当前知识图谱和 LightRAG 链路代码看，没有非常强的“知识库级别租户隔离”实现，所以我会实话实说：当前版本先解决单租户或弱隔离场景，若做多租户会在任务、知识库、图谱命名空间和查询过滤层再补一层。
    
28.  最容易被忽视但关键的问题<br>答：【代码可证】我觉得最容易被忽视的是“长任务编排和状态一致性”。因为这个项目不是一个接口调完就结束，它有 Python 侧处理状态、Java 侧轮询状态、前端侧进度展示、Neo4j 最终提交四个环节，所以任务状态、失败恢复、提交时机比单纯的问答效果更关键。
    
29.  如果做成通用知识服务平台<br>答：我会把它抽成四层。第一层是数据接入层，统一文档、表格、空间数据和业务对象；第二层是索引与图谱层，分别承接向量索引、图数据库和元数据；第三层是查询编排层，负责 RAG、GraphRAG、图查询和混合检索；第四层是能力开放层，提供问答、图谱预览、引用溯源和任务管理 API。
    
30.  怎么向非技术面试官解释价值<br>答：这不是“让模型更聪明”的项目，而是“让专业资料更可用”的项目。我会用业务语言解释成：以前资料散在文档和空间数据里，查一件事要翻很多材料；现在系统把这些信息统一整理成可检索、可关联、可问答的知识底座，能明显提升专业人员找资料、看关系和做辅助判断的效率。
