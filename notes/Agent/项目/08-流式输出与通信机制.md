# 流式输出、交互体验与通信机制

> 说明：
> - `【代码可证】` 表示可以直接从仓库代码、配置或文档中落到实现细节。
> - `【需按实际补充】` 表示仓库里没有团队人数、上线规模、业务指标等信息，面试前需要替换成你的真实经历。

121. 为什么对话用 SSE 而不是 WebSocket  
答：【代码可证】因为这里的核心诉求是“服务端单向持续推文本”，SSE 足够简单，浏览器原生支持也更好。自然资源平台的 `/system/rag/stream-query`、OnCall Java 版 `/api/chat_stream` 和 Go 版 `ChatStream` 都是这种单向流式输出场景，所以用 SSE 更合适。

122. SSE 解决了什么体验问题  
答：它把“等模型整段答完再展示”改成“边生成边展示”。对长回答、RAG 查询和告警分析这类场景，用户能更早感知系统在工作，等待焦虑会明显下降。

123. 网络中断怎么处理  
答：【代码可证】当前实现更多是检测到错误后结束连接并返回 error/done 事件，没有做特别复杂的断点续传协议。面试时我会说明：当前版本是“重建连接重新发起”，如果要做得更稳，可以基于事件 ID、会话 ID 或中间状态做恢复。

124. 如何协调模型流、工具流和展示流  
答：【代码可证】Java 版在 `chat_stream` 里会区分 `AGENT_MODEL_STREAMING / AGENT_TOOL_FINISHED / AGENT_MODEL_FINISHED` 等事件类型，再统一通过 `SseEmitter` 推给前端；自然资源平台也类似，是先从 Python 收到字符串流，再由 Java 统一转发给前端。也就是说，后端已经承担了“把底层多类事件整理成前端可消费流”的角色。

125. 如果先输出一半又去调工具怎么办  
答：这类情况我会建议前端区分“推理中”“工具执行中”和“最终回答”状态，而不是把所有文本都当最终答案。  
答：【代码可证】Java 版已经能感知工具完成事件，只是展示层还能继续细化。

126. SSE 并发上来后的压力点  
答：压力主要在长连接数、线程/协程占用、内存缓冲和下游模型调用时长。尤其是 Spring MVC 的 `SseEmitter` 模式，如果连接很多而每个请求都很长，就要特别注意线程池、超时和连接清理。

127. Spring Boot 里怎么实现流式  
答：【代码可证】自然资源平台和 OnCall Java 版都用了 `SseEmitter`。前者是 `WebClient.bodyToFlux(String.class)` 订阅 Python 流后再 `emitter.send(...)`，后者是订阅 `agent.stream()` 的输出，并把 chunk 包装成统一消息格式推给前端。

128. 为什么一个项目里有 WebSocket，另一个强调 SSE  
答：【代码可证】自然资源平台里 WebSocket 主要用于构建进度推送，因为它绑定了 `taskId -> session` 的长期任务状态；SSE 主要用于问答文本流。换句话说，二者不是互斥，而是按场景分工：文本流优先 SSE，任务进度或已有框架沉淀则可以用 WebSocket。

129. 如果后续要双向实时交互怎么办  
答：如果只是单向输出，我还会优先用 SSE；如果后续需要前端在流中途动态打断、插入控制信号、频繁双向协作，那我会考虑切回 WebSocket。选型核心不是“哪个高级”，而是交互方向和复杂度。

130. 有无超时、取消和背压  
答：【代码可证】当前已经有连接超时控制，比如自然资源平台 `SseEmitter(300_000L)`、OnCall Java 版 `5~10` 分钟超时，Go 版也有基于 HTTP 连接的流式返回。  
答：真正完整的取消生成和背压机制还可以继续加强，我会把这题回答成“当前先做超时和错误结束，后续补用户主动取消和更细粒度的流控”。
