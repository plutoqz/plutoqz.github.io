# 背景、范围与知识地图

## 本章目标

-   回答“为什么现在大家都在讲 Agent”。
    
-   明确 Agent 与聊天机器人、RAG、工作流、自动化脚本的边界。
    
-   形成一个判断框架：什么问题适合做 Agent，什么问题不适合。
    

## 核心问题

-   Agent 到底比普通 LLM 应用多了什么？
    
-   为什么 tool calling、reasoning、workflow 结合后，系统形态会发生变化？
    
-   什么时候应该保守地退回到普通工作流？
    

## 正文

### 1\. 为什么 Agent 会成为一个独立话题

在早期 LLM 应用里，最常见的形态是“一次输入，一次输出”。这类系统强依赖 prompt 质量，本质上是一个更强的自然语言接口。随着模型在推理、多模态和工具调用上的能力增强，系统开始具备下面几种新特征：

-   能把大任务拆成多步任务。
    
-   能调用外部工具获取信息或执行动作。
    
-   能根据中间结果改变下一步计划。
    
-   能把执行状态跨轮次保留下来。
    

OpenAI 在 2026-03-25 可访问的《A practical guide to building agents》中把这类系统归纳为：能端到端执行 workflow、处理多步任务、跨工具采取行动的系统。这个表述很重要，因为它强调的是“执行流程”，而不是“回答得像不像人”。

### 2\. 一个实用定义

这里给出一个偏工程视角的定义：

> Agent 是一个围绕目标运行的 LLM 系统，它在执行循环中观察环境、做出决策、调用工具、更新状态，并在达到退出条件前持续推进任务。

这个定义是作者归纳，不是某篇论文里的标准术语，但它足够适合工程讨论。它包含五个关键点：

1.  有目标，不只是对话。
    
2.  有循环，不只是单次生成。
    
3.  有外部能力，不只是模型内部知识。
    
4.  有状态，不只是当下这一个 prompt。
    
5.  有退出条件，不会无限“自主”下去。
    

### 3\. Agent、RAG、工作流、脚本的关系

它们不是互斥关系，更像层层增强。

| 形态  | 核心机制 | 适合场景 | 局限  |
| --- | --- | --- | --- |
| 普通 LLM 应用 | 一次 prompt -> 一次输出 | 文案生成、分类、抽取、翻译 | 不能处理复杂多步任务 |
| RAG 应用 | 检索后生成 | 问答、知识库、文档助手 | 更像“增强上下文”，不等于会执行 |
| 工作流 | 预定义步骤与分支 | 流程明确、强可控的任务 | 对未知情况适应差 |
| Agent | 模型在循环里决定下一步 | 模糊、多步、动态决策任务 | 成本高、风险高、调试难 |

要注意两件事：

-   RAG 可以是 Agent 的一个工具，但 RAG 本身不等于 Agent。
    
-   工作流也可以承载 Agent。很多所谓的“Agent 系统”其实是“工作流 + 某些节点允许模型自主决策”。
    

### 4\. Agent 最适合解决什么问题

经验上，下面四类问题比较适合：

#### 4.1 任务是多步的

例如“分析一份财报并生成风险摘要，再把高风险项推送给对应负责人”。这不是一步生成，而是要做检索、理解、判断、格式化、分发。

#### 4.2 路径是不确定的

例如客服、代码修复、研究助理。系统不能事先写死每一步，因为下一步依赖中间结果。

#### 4.3 需要和外部世界交互

比如查数据库、调用搜索、读文件、发消息、执行审批、跑测试。没有工具，模型只能“想象”。

#### 4.4 规则很复杂，但又难以纯规则化

如果规则全写死，维护成本会爆炸；如果完全交给模型，又有失控风险。Agent 往往适合作为“规则 + 模型”之间的折中。

### 5\. 什么问题不该上 Agent

这是面试和实战里都很容易被忽略的一点。

#### 5.1 任务是确定性流程

如果你的任务只是“填表 -> 校验 -> 写库 -> 发通知”，且每一步都非常明确，普通代码和工作流比 Agent 更可靠、更便宜。

#### 5.2 低延迟刚需

Agent 常常意味着多轮模型调用和多次工具调用。只要用户对延迟非常敏感，比如毫秒级推荐、实时风控、同步交易判定，就要极度克制。

#### 5.3 错一次代价极高

比如大额转账、删除生产数据、法律承诺、医疗建议、合同签发。即便要上 Agent，也必须把 Agent 放在建议层或审批前层，而不是直接执行层。

#### 5.4 数据和工具边界不清晰

如果你还没理清工具权限、输入结构、输出约束、审计链路，先上 Agent 往往会放大混乱。

### 6\. Agent 的知识地图

理解 Agent 最好从三层去看：

#### 6.1 能力层

-   推理
    
-   规划
    
-   工具使用
    
-   记忆
    
-   反思和纠错
    

#### 6.2 运行层

-   指令系统
    
-   执行循环
    
-   状态持久化
    
-   工具注册与调用
    
-   观察与回写
    
-   终止条件
    

#### 6.3 治理层

-   权限与审批
    
-   Guardrails
    
-   评测与回放
    
-   监控与追踪
    
-   成本与 SLA
    

很多入门文章只讲能力层，真正的生产问题往往出在运行层和治理层。

### 7\. 一个最小认知框架

如果只记一句话，可以记这个：

> Agent 不是“会自己想很多”的模型，而是“被放进一个可观察、可约束、可恢复执行循环里的模型”。

这句话能帮你避开很多空洞讨论。只谈“智能”，不谈循环、状态、工具、副作用和审计，通常都落不到工程上。

## 工程细节与取舍

### 1\. 不要把“自主性”当唯一指标

更自主不一定更好。自主性提升通常伴随：

-   更多 token 消耗
    
-   更多错误分支
    
-   更复杂的测试矩阵
    
-   更高的安全风险
    

工程上真正追求的是“在必要的范围内让模型自主”。

### 2\. Agent 是业务设计问题，不只是模型问题

你需要先回答：

-   目标是什么？
    
-   输入输出是什么？
    
-   哪些步骤可以让模型决定？
    
-   哪些步骤必须规则化？
    
-   哪些动作需要审批？
    

如果这些问题没想清楚，换更强模型也解决不了根因。

### 3\. 先做窄任务，再扩边界

OpenAI 的公开指南明确建议先从单 Agent 开始，复杂度提升后再演进到多 Agent。这个建议本质上是在提醒你：评测和维护能力，通常跟不上你对“智能体”的想象速度。

## 常见误区

### 误区 1：会调用工具就是 Agent

不完全对。一次函数调用的问答机器人，可能只是“tool-augmented LLM”。只有当系统在循环中根据中间结果持续决策时，才更接近 Agent。

### 误区 2：多 Agent 一定比单 Agent 强

不对。多 Agent 更多是在做模块化和职责隔离，未必直接提高任务成功率。很多时候只是把一个难调的 prompt，变成了多个难同步的 prompt。

### 误区 3：有记忆就更智能

不一定。记忆也会引入噪声、陈旧信息、隐私风险和 token 成本。记忆首先是状态管理问题，其次才是“更像人”。

### 误区 4：Agent 就是未来，所有 AI 应用都应该升级

错误。很多业务最优解依然是“检索 + 结构化生成 + 规则工作流”，而不是完全自治。

## 面试与实战问答

### 问：什么是 Agent？

短答： 能围绕目标进行多步执行、调用工具、根据中间结果调整路径并在退出条件满足前持续推进的 LLM 系统。

深答： 重点不在“像不像人”，而在“是否存在可持续的执行循环”。普通问答应用是一轮生成，Agent 是运行时系统。

### 问：什么时候应该做 Agent？

短答： 任务多步、路径不确定、需要外部工具、规则难以完全写死时。

深答： 如果任务本身是确定流程，优先考虑工作流；如果只是知识增强，优先考虑 RAG；只有在动态决策成为核心瓶颈时，Agent 才值得引入。

### 问：Agent 和工作流有什么区别？

短答： 工作流强调预定义路径，Agent 强调模型在运行中决定下一步。

深答： 很多生产系统其实是两者结合：整体流程由工作流控制，局部复杂节点由 Agent 决策。
