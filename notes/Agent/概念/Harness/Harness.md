# Harness

---

# 一、Harness 在 Agent 领域到底是什么

## 1\. 一个实用定义

在 Agent 系统里，**harness 是一个用于封装、调度、观测、约束、测试和评估 Agent 行为的外层控制与集成层**。

它通常不等于 Agent 本身，而是围绕 Agent 的一组机制，用来回答这些问题：

-   Agent 接收什么输入？
    
-   它能调用哪些工具？
    
-   运行流程如何编排？
    
-   如何记录每一步推理、调用和结果？
    
-   如何回放、调试、评测？
    
-   出错时怎么重试、回滚、降级？
    
-   如何限制成本、权限和风险？
    
-   怎样把 Agent 接入真实业务系统？
    

所以，**harness 本质上是“Agent 的工程化运行与验证载体”**。

---

## 2\. 为什么会出现 harness

早期很多人把 Agent 理解为：

> 一个大模型 + prompt + tools

但一旦真的进入生产场景，很快就会遇到问题：

-   输出不稳定
    
-   工具调用失败
    
-   多步推理难以追踪
    
-   成本不可控
    
-   很难评估“这次成功是不是偶然”
    
-   不知道是模型问题、提示词问题还是工具问题
    
-   线上出了问题没法复现
    
-   Agent 可能越权操作或产生危险动作
    

于是就需要一层“工程壳”来管理这些复杂性，这一层就是 harness。

---

## 3\. 直观类比

### 类比 1：自动驾驶

-   **Agent**：驾驶决策系统
    
-   **Harness**：仿真平台、数据记录器、安全限制器、A/B 测试系统、事故回放系统
    

### 类比 2：机器人

-   **Agent**：控制机器人的“智能体”
    
-   **Harness**：传感器接线、动作限制、安全停机、测试场景、日志记录、仿真环境
    

### 类比 3：软件测试

-   **业务代码**：Agent 的核心决策逻辑
    
-   **Test harness**：测试输入、mock 服务、断言、报告、回放环境
    

实际上，“harness”这个词在软件工程里原本就有“测试支架/运行支架”的意味。到了 Agent 领域，这个概念被扩展到了**运行控制 + 评估验证 + 集成接入**。

---

# 二、Harness 的核心组成

一个成熟的 Agent harness 往往包含以下模块。

## 1\. 输入封装层

负责把用户请求、上下文、历史状态、环境信息整理成 Agent 可消费的格式。

常见职责：

-   用户输入标准化
    
-   会话状态注入
    
-   任务上下文拼装
    
-   记忆检索结果注入
    
-   权限和租户信息注入
    

例子：

-   用户说“帮我总结这个季度销售异常原因”
    
-   harness 会自动补上：
    
    -   用户部门与权限
        
    -   允许访问的数据源
        
    -   当前日期与组织上下文
        
    -   相关 dashboard 链接
        
    -   可调用工具列表
        

---

## 2\. 工具与环境适配层

Agent 理论上可以“调用工具”，但真实世界里工具五花八门，接口也不统一。<br>harness 会把这些能力抽象出来。

常见职责：

-   统一工具调用协议
    
-   包装 API、数据库、搜索、代码执行、浏览器操作
    
-   做鉴权、限流、超时控制
    
-   参数校验与返回结构标准化
    

这很关键，因为很多 Agent 的不稳定，其实不是模型本身，而是工具链不稳定。

---

## 3\. 调度与控制层

负责决定 Agent 怎么跑。

包括：

-   单步 / 多步任务执行
    
-   ReAct-style 推理 + 工具调用循环
    
-   plan-and-execute
    
-   多 Agent 协作调度
    
-   超时、最大步数、预算控制
    
-   重试和fallback策略
    

这部分很像一个轻量 orchestration engine。

---

## 4\. 观测与日志层

这是 harness 最有价值的部分之一。

记录：

-   输入 prompt
    
-   中间推理痕迹（有时是安全脱敏版本）
    
-   工具调用参数与结果
    
-   token 消耗
    
-   延迟
    
-   失败原因
    
-   最终输出
    
-   用户反馈
    

没有这个层，你几乎无法系统性改进 Agent。

---

## 5\. 评测与回放层

Agent 不能只看“感觉好不好用”，要能系统评估。

harness 往往支持：

-   benchmark 数据集运行
    
-   case replay（线上案例回放）
    
-   基线对比（prompt A vs prompt B，model A vs model B）
    
-   自动评分 / LLM-as-a-judge / rule-based judge
    
-   任务成功率统计
    
-   失败模式分类
    

这能把 Agent 开发从“炼丹”推进到“可迭代工程”。

---

## 6\. 安全与治理层

Agent 比普通生成式应用更危险，因为它可能“采取行动”。

因此 harness 经常承担：

-   权限隔离
    
-   工具白名单
    
-   高风险动作二次确认
    
-   沙箱执行
    
-   PII 脱敏
    
-   内容安全过滤
    
-   审计日志
    
-   人类审批节点（human-in-the-loop）
    

---

## 7\. 状态与记忆管理

很多 Agent 需要跨轮次、跨任务保持状态。

harness 可能管理：

-   短期会话状态
    
-   长期记忆
    
-   任务中间状态
    
-   工作流 checkpoint
    
-   失败恢复点
    

如果没有这层，Agent 会像“失忆症患者”，每次都从头开始。

---

# 三、Harness 和几个相近概念的区别

这是理解 Agent 工程栈的关键。

## 1\. Harness vs Agent

-   **Agent**：负责感知、推理、决策、行动
    
-   **Harness**：负责把 Agent 放进一个可控、可观测、可评估、可上线的系统里
    

一句话： **Agent 是智能行为体，harness 是智能行为体的工程化外壳。**

---

## 2\. Harness vs Framework

-   **Framework**：开发 Agent 的编程框架，比如提供抽象、组件和编排能力
    
-   **Harness**：更强调运行、测试、验证、控制和集成
    

Framework 偏“开发构件”，Harness 偏“操作与验证支架”。

当然现实里两者会重叠。很多 Agent 框架会内置 harness 能力。

---

## 3\. Harness vs Runtime

-   **Runtime**：Agent 实际执行时所在的运行时环境
    
-   **Harness**：可能包住 runtime，并在其外侧加上日志、评测、安全、调试等能力
    

Runtime 更偏执行；Harness 更偏控制和验证。

---

## 4\. Harness vs Workflow / Orchestrator

-   **Workflow / Orchestrator**：关注步骤编排
    
-   **Harness**：不仅编排，还包括测试、回放、观测、安全、评估
    

你可以把 orchestrator 看成 harness 的一个子模块。

---

## 5\. Harness vs Eval

-   **Eval**：评价 Agent 效果的方法和体系
    
-   **Harness**：承载 eval 执行的环境
    

Eval 是“怎么评”，Harness 是“在哪评、如何接入真实系统评”。

---

# 四、Harness 在 Agent 里的典型应用

---

## 1\. 开发期：快速实验和迭代

这是最常见场景。

开发 Agent 时，团队通常需要：

-   快速切换模型
    
-   快速改 prompt
    
-   增减工具
    
-   查看每一步调用
    
-   用同一批测试集反复跑
    

这时 harness 就像实验平台。

### 例子

一个企业知识问答 Agent：

-   版本 A：只用 RAG
    
-   版本 B：RAG + SQL 查询工具
    
-   版本 C：增加澄清提问步骤
    

通过 harness，你能在统一测试集上比较：

-   回答准确率
    
-   幻觉率
    
-   平均延迟
    
-   平均成本
    
-   工具调用次数
    

没有 harness，这种对比几乎全靠人工感觉。

---

## 2\. 测试期：场景验证和回归测试

Agent 非常容易“改一处、坏很多处”。

比如：

-   换了新模型后，数据分析能力提升，但工具使用格式变差
    
-   prompt 更保守后，安全性好了，但任务完成率下降
    
-   增加记忆模块后，回答更个性化，但也更容易引入错误旧信息
    

harness 可以做：

-   regression suite
    
-   adversarial test
    
-   edge case test
    
-   offline replay
    

这非常像传统软件测试，但对象从“函数输出”变成“复杂智能行为”。

---

## 3\. 生产运行：可观测与故障诊断

线上最怕的是：

-   用户说“它刚才答错了”
    
-   但你完全不知道它看到了什么、调了什么工具、哪里失败了
    

harness 可以记录完整轨迹：

-   原始输入
    
-   检索结果
    
-   工具响应
    
-   决策链
    
-   最终答案
    
-   审核结果
    

这样能快速区分：

-   是检索没召回
    
-   还是 SQL 生成错了
    
-   还是工具超时
    
-   还是模型推理偏了
    
-   还是 prompt 指令冲突
    

---

## 4\. 多 Agent 系统中的协调支架

当系统里不是一个 Agent，而是多个角色协作时，harness 更重要。

例如：

-   Planner Agent：分解任务
    
-   Research Agent：检索资料
    
-   Analyst Agent：生成分析
    
-   Executor Agent：调用业务系统执行
    

harness 负责：

-   角色之间消息传递
    
-   状态管理
    
-   执行顺序
    
-   失败回退
    
-   人工接管
    
-   统一日志和评估
    

没有 harness，多 Agent 往往会变成不可调试的黑盒群聊。

---

## 5\. 高风险场景中的安全围栏

比如下面这些任务：

-   自动发邮件给客户
    
-   修改数据库记录
    
-   下采购单
    
-   触发财务审批
    
-   在云平台执行脚本
    

这类任务不能让 Agent“想到什么做什么”。

harness 会加：

-   权限控制
    
-   sandbox
    
-   action confirmation
    
-   policy engine
    
-   审批流
    
-   kill switch
    

所以在企业场景，**harness 不只是增强体验，而是生产落地前提**。

---

## 6\. 评测平台与 benchmark 系统

很多团队会专门做一个 harness 来跑 benchmark。

比如评估客服 Agent：

-   是否正确识别用户意图
    
-   是否遵守退款政策
    
-   是否错误承诺
    
-   是否在规定轮数内解决问题
    
-   是否调用正确后台工具
    

这种 harness 通常支持：

-   标准化输入样本
    
-   模拟用户环境
    
-   自动判分
    
-   报告汇总
    
-   版本对比
    

---

# 五、Harness 的价值：它解决了什么根本问题

## 1\. 把“随机智能”变成“可管理系统”

LLM/Agent 的一个大问题是非确定性。<br>harness 不会消除非确定性，但会让它变得**可测、可控、可分析、可约束**。

---

## 2\. 把 Agent 从 demo 推向 production

很多 Agent demo 看起来很惊艳，但一上线就崩。

差别往往不在模型，而在有没有：

-   输入治理
    
-   工具治理
    
-   权限治理
    
-   评估治理
    
-   运行治理
    

这些基本都属于 harness 范畴。

---

## 3\. 让优化有抓手

如果没有 harness，优化 Agent 往往只能凭主观经验：

-   换个 prompt 试试
    
-   换个模型试试
    
-   再加个工具试试
    

有了 harness，你可以定量分析：

-   错误集中在哪一步
    
-   成本花在哪
    
-   哪些用户意图成功率最低
    
-   哪些工具最常失败
    
-   哪种路由策略效果最好
    

---

## 4\. 支撑组织协作

Agent 项目往往不是一个人能完成的，会涉及：

-   Prompt / Agent 设计
    
-   后端工程
    
-   工具接口
    
-   安全合规
    
-   QA
    
-   产品运营
    

harness 提供一个共同工作界面，让团队围绕统一的数据和运行轨迹协作。

---

# 六、Harness 的常见实现形态

不同团队会把 harness 做成不同形式。

## 1\. 轻量代码层 harness

最简单形态，就是一层 Python/TypeScript 封装：

-   定义输入输出 schema
    
-   包装模型调用
    
-   包装工具
    
-   统一日志
    
-   批量测试脚本
    

适合：

-   POC
    
-   早期研发
    
-   单 Agent 项目
    

---

## 2\. Agent 平台型 harness

更成熟的团队会做平台：

-   Prompt 管理
    
-   版本管理
    
-   Trace 可视化
    
-   数据集管理
    
-   自动 eval
    
-   在线监控
    
-   人工标注
    
-   A/B 实验
    

适合：

-   多团队协作
    
-   多业务线 Agent
    
-   企业级治理
    

---

## 3\. 仿真环境 harness

在更复杂的 Agent 场景，harness 会变成一个模拟世界。

例如：

-   模拟用户
    
-   模拟网页环境
    
-   模拟软件界面
    
-   模拟 API 响应
    
-   模拟经济环境或游戏环境
    

Agent 在这个“沙盒世界”里跑，harness 负责观察和评分。

这种方式在以下方向非常重要：

-   computer-use agents
    
-   embodied agents
    
-   web agents
    
-   game agents
    
-   autonomous coding agents

# 七、Agent Harness 的设计要点

如果你想真正理解 harness，最好看它设计时的关键原则。

## 1\. 标准化接口

Agent、工具、环境、记忆、评估器之间最好有统一接口。<br>否则系统会迅速耦合失控。

例如：

-   `run(task, context) -> result`
    
-   `tool.call(args) -> observation`
    
-   `evaluate(trace) -> score`
    

---

## 2\. 可观测优先

很多团队早期只关注“让它跑起来”，后来最大的问题是“根本不知道它怎么跑的”。

所以 harness 应该默认记录：

-   请求上下文
    
-   prompt 版本
    
-   模型参数
    
-   工具调用
    
-   中间状态
    
-   输出结果
    
-   成本与时延
    

---

## 3\. 可复现

可复现很难，但要尽量做：

-   固定测试集
    
-   固定工具返回 mock
    
-   模型版本固定
    
-   prompt 版本固定
    
-   环境快照
    
-   trace replay
    

这对排查 Agent 问题非常关键。

---

## 4\. 安全内建，而不是事后补丁

权限、审计、沙箱、审批，不应该等快上线了再补。<br>对 Agent 来说，这些最好就是 harness 的默认能力。

---

## 5\. 人机协同接口

许多任务不是“全自动”才最好，而是“关键节点自动 + 人工确认”。

好的 harness 会支持：

-   人工 review 节点
    
-   手动修正工具参数
    
-   中断后续动作
    
-   指定 fallback 路径
    

---

## 6\. 评测闭环

生产 trace 应该回流到评测集。<br>失败案例应该变成回归测试样本。<br>这会形成真正的 Agent 迭代飞轮。

---

# 八、Harness 的难点与局限

讲潜力之前，先看现实难点。

## 1\. Agent 的成功标准很难定义

传统软件常有明确输入输出。<br>Agent 任务往往开放性很强，例如：

-   “做一个市场分析”
    
-   “帮我安排一次合理的出差计划”
    
-   “与用户沟通直到解决问题”
    

这类任务很难像单元测试那样简单断言。<br>因此 harness 的 eval 往往需要：

-   多维指标
    
-   规则评分
    
-   人工标注
    
-   LLM judge
    
-   业务结果反馈
    

---

## 2\. 非确定性导致复现困难

即使固定 prompt，模型输出也可能波动。<br>再叠加外部工具变化，问题更复杂。<br>因此 harness 只能“提高复现度”，很难做到完全复现。

---

## 3\. 工具环境太复杂

真实世界工具很多：

-   API 变更
    
-   权限漂移
    
-   返回格式不稳定
    
-   网络超时
    
-   数据实时变化
    

所以 harness 不是一劳永逸，它需要持续维护。

---

## 4\. 观测与隐私/安全的矛盾

为了调试，你希望记录所有轨迹。<br>为了合规，你又不能记录敏感数据。<br>所以 harness 需要在可观测性和合规之间平衡：

-   脱敏
    
-   最小化存储
    
-   分级访问
    
-   审计留痕
    

---

## 5\. 评估本身可能不稳定

如果使用 LLM-as-a-judge，评委模型本身也可能波动或偏见。<br>因此很多成熟团队会采用混合评估：

-   规则检查
    
-   程序化断言
    
-   LLM judge
    
-   人工抽检
    

---

# 九、Harness 的潜力：为什么它可能成为 Agent 时代的重要基础设施

这是最值得展开的部分。

## 1\. 它可能成为 Agent 的“软件工程层”

今天的软件工程有：

-   测试框架
    
-   CI/CD
    
-   监控
    
-   APM
    
-   权限系统
    
-   灰度发布
    
-   可观测平台
    

Agent 时代也需要对应层。<br>而 harness 很可能就是这个“统一承载层”的雏形。

换句话说：

> 如果模型是 Agent 时代的“CPU”，<br>那 harness 可能会成为 Agent 时代的“操作系统外设管理 + 调试工具链 + QA/治理平台”的组合。

---

## 2\. 它会决定 Agent 能否真正进入企业核心流程

企业不关心 demo 有多聪明，企业关心的是：

-   稳不稳定
    
-   能不能审计
    
-   能不能控制权限
    
-   能不能回溯责任
    
-   能不能持续优化
    
-   出错后怎么止损
    

这些都不是“模型单点更强”能自动解决的。<br>必须靠 harness 类基础设施补齐。

---

## 3\. 它是多 Agent 和自治系统扩展的前提

Agent 越自主，harness 越重要。<br>因为自主性提升会带来：

-   更长决策链
    
-   更多环境交互
    
-   更高风险
    
-   更复杂调试
    
-   更难验证
    

所以： **越接近 Autonomous Agent，越离不开强 harness。**

---

## 4\. 它会成为数据飞轮的核心节点

好的 harness 能把：

-   线上真实任务
    
-   用户反馈
    
-   失败案例
    
-   工具调用轨迹
    
-   人工修正结果
    

沉淀成可训练、可评估、可优化的数据资产。

这意味着 harness 不只是控制层，也是**学习闭环的入口**。

# 十、未来发展方向：Harness 在 Agent 领域会往哪里演进

## 1\. 从“调试支架”演变为“自治治理层”

最开始 harness 可能只是日志和测试工具。<br>未来它会更像：

-   policy engine
    
-   execution governor
    
-   audit system
    
-   adaptive router
    
-   safety supervisor
    

即不仅观察 Agent，还主动约束和引导 Agent。

---

## 2\. 从静态规则控制走向动态控制

未来 harness 可能动态决定：

-   当前任务该用哪个模型
    
-   是否允许调用某个工具
    
-   是否让高风险动作进入审批
    
-   遇到不确定时是否要求 Agent 自检
    
-   是否切换到更保守模式
    

这相当于“Agent 的上位控制器”。

---

## 3\. 与仿真、训练、评测更深融合

未来 harness 很可能同时承担：

-   训练数据采集
    
-   仿真评测
    
-   线上监控
    
-   失败回放
    
-   强化学习/偏好优化数据生成
    

即同一套运行轨迹，服务于开发、评测、训练、治理多个环节。

---

## 4\. 与企业系统原生融合

Agent 不是孤立玩具，它要接 ERP、CRM、工单系统、知识库、代码仓库、云平台。<br>所以 harness 会越来越像企业 AI 中台的一部分。

---

## 5\. 专用化、垂直化

不同 Agent 类型需要不同 harness：

-   **Coding Agent Harness**：代码执行沙箱、测试运行、diff 审核、回滚
    
-   **Customer Support Agent Harness**：政策规则检查、情绪监测、工单系统联动
    
-   **Sales Agent Harness**：CRM 权限控制、外呼审计、合规检查
    
-   **Data Analyst Agent Harness**：SQL 安全、结果校验、图表生成追踪
    
-   **Browser/Web Agent Harness**：页面状态捕获、动作回放、DOM diff、失败重试
    

未来不会只有一种通用 harness，而会出现大量行业化和任务化 harness。

# 十一、一个具体例子：数据分析 Agent 的 Harness 长什么样

假设你要做一个企业数据分析 Agent，用户输入：

> “分析一下华东区最近两个月销售额下滑的主要原因，并给我可执行建议。”

一个成熟 harness 会做什么？

## 输入阶段

-   识别用户身份、部门、权限
    
-   加载华东区相关业务上下文
    
-   检查用户是否有访问销售数据库权限
    
-   附带当前时间范围、组织层级信息
    

## 规划阶段

-   让 Agent 先输出分析计划
    
-   限制最大步骤数
    
-   设置 token 和成本预算
    

## 工具阶段

允许 Agent 调用：

-   SQL 查询工具
    
-   指标字典工具
    
-   BI dashboard 检索工具
    
-   文档检索工具
    

同时 harness 会：

-   校验 SQL 安全性
    
-   限制只读权限
    
-   防止全表扫描
    
-   记录每次查询
    

## 执行阶段

-   捕获每一步工具调用
    
-   如果 SQL 执行失败，自动反馈错误给 Agent 重试
    
-   如果 Agent 准备给出强结论但证据不足，插入“请补充验证”检查
    

## 输出阶段

-   要求输出带证据引用
    
-   标注哪些结论是“基于数据验证”，哪些是“推测”
    
-   生成结构化结果：摘要、原因、证据、建议、风险
    

## 安全阶段

-   检查是否泄露敏感数据
    
-   检查建议是否超出权限边界
    

## 评估阶段

-   将这次 trace 记录下来
    
-   如果用户点了“无帮助”，自动进入失败案例池
    
-   作为未来回归测试样本
    

这个例子里，你会发现真正支撑系统落地的，不只是 Agent 本身，而是它背后整套 harness。
