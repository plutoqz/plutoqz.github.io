# 推理、规划与工具使用模式

## 本章目标

-   理解 Agent 常见的决策和工具使用模式。
    
-   建立“不同任务该用哪种模式”的判断能力。
    
-   识别 ReAct、planner-worker、evaluator 等模式的优缺点。
    

## 核心问题

-   Agent 如何在推理、规划和行动之间切换？
    
-   为什么工具调用设计会直接影响 Agent 上限？
    
-   怎样避免“看起来很聪明，实际上一直乱试”的假自治？
    

## 正文

### 1\. 推理、规划、行动不是一回事

在 Agent 里，这三个词经常被混着用，但最好分开：

-   `推理`：基于当前上下文做判断。
    
-   `规划`：决定后续步骤的结构或顺序。
    
-   `行动`：真正发起工具调用、修改状态或触发外部操作。
    

很多系统失败的原因是：模型很会“讲计划”，但不擅长“稳定执行计划”。

### 2\. ReAct：最经典的交替模式

ReAct 论文提出把 reasoning trace 和 action interleave，也就是让模型在“想一想”和“做一步”之间交替。它的重要意义不只是论文效果，而是给 Agent 设计提供了一个非常自然的运行范式：

1.  观察当前信息
    
2.  形成下一步想法
    
3.  执行动作
    
4.  获得新观察
    
5.  继续循环
    

ReAct 特别适合：

-   信息不全，需要边查边推理
    
-   任务路径依赖中间结果
    
-   希望中间过程更可解释
    

但它也有明显问题：

-   轮数可能很多
    
-   token 消耗高
    
-   如果工具设计不好，会在错误路径上反复试探
    

### 3\. Toolformer：把工具使用能力当成模型能力的一部分

Toolformer 论文强调：模型不仅要会“调用工具”，还要学会判断何时调用、调用什么、如何把结果融合回后续预测。对工程的启发是：

-   工具调用不是外挂，而是能力系统的一部分。
    
-   工具越类型化、返回越稳定，模型越容易学会高质量使用。
    
-   真正高水平的 Agent，不是“能用工具”，而是“知道什么时候不用工具”。
    

### 4\. 五种常见模式

#### 4.1 ReAct 模式

特点：

-   一轮一轮思考和行动
    
-   非常通用
    
-   易于从单 Agent 起步
    

适合：

-   通用助理
    
-   搜索型研究 Agent
    
-   代码修复与调试 Agent
    

#### 4.2 Plan-and-Execute 模式

先产出全局计划，再逐步执行。

优点：

-   大任务结构更清楚
    
-   更容易加入检查点和人工审核
    

缺点：

-   初始计划可能建立在不完整信息上
    
-   环境变化时，原计划容易失效
    

适合：

-   结构化研究
    
-   长文档处理
    
-   项目级任务拆分
    

#### 4.3 Sequential Workflow 模式

按固定顺序把任务拆成若干节点，每个节点可能由模型或工具完成。

优点：

-   强可控
    
-   易测
    
-   易审计
    

缺点：

-   灵活性差
    
-   不适合开放式探索
    

适合：

-   审批流
    
-   信息抽取后再归档
    
-   固定格式报告生成
    

#### 4.4 Parallel / Fan-out 模式

把子任务拆给多个并行 worker，再汇总结果。

优点：

-   节省总耗时
    
-   适合天然可分块任务
    

缺点：

-   需要良好的结果聚合
    
-   容易出现重复工作和相互矛盾
    

适合：

-   多来源检索
    
-   多文档摘要
    
-   多方案对比
    

#### 4.5 Evaluator-Optimizer 模式

一个执行器先产出结果，另一个评估器负责打分、指出问题，再由执行器修正。

优点：

-   适合质量要求较高的产出
    
-   更接近“生成 + 审稿”的现实流程
    

缺点：

-   额外成本高
    
-   如果 evaluator 标准不清，会形成自我循环
    

适合：

-   代码生成与修复
    
-   高质量写作
    
-   风险摘要与合规审阅
    

Anthropic 的《Building Effective AI Agents》明确提到 sequential、parallel、evaluator-optimizer 等模式，这些名字在很多框架里写法不同，但本质思想是通用的。

### 5\. 如何选择模式

可以用一个简单判断法：

| 问题特征 | 更适合的模式 |
| --- | --- |
| 边查边想，路径动态变化 | ReAct |
| 大任务先拆计划再推进 | Plan-and-Execute |
| 步骤相对固定 | Sequential Workflow |
| 子问题可拆且彼此独立 | Parallel |
| 结果质量重于速度 | Evaluator-Optimizer |

很多生产系统并不是只用一种模式，而是混合使用：

-   上层用 workflow 控制大节点
    
-   节点内部用 ReAct 做局部决策
    
-   输出阶段再加 evaluator 做质量把关
    

### 6\. 工具设计决定 Agent 的真实上限

模型再强，如果工具不好，Agent 也很难稳定。

好的工具设计至少包含：

-   清晰的名字：一眼看出用途
    
-   明确的输入参数：字段有含义、有类型
    
-   明确的副作用说明：是只读还是会写
    
-   结构化返回：便于后续模型稳定消费
    
-   错误语义：告诉模型是权限问题、参数问题还是资源不存在
    

### 7\. 工具选择的常见失败

#### 7.1 工具重叠

例如两个搜索工具、两个数据库查询工具、三个类似写入工具。模型会犹豫、误选，成功率明显下降。

#### 7.2 工具返回过长

返回大段原文或超大 JSON，会把上下文打爆，也增加注入风险。

#### 7.3 工具不幂等

一旦 Agent 因失败重试，可能重复创建任务、重复扣费、重复发消息。

#### 7.4 工具错误信息不可操作

如果工具只返回“failed”，模型无法判断该重试、换参数还是直接终止。

### 8\. 一个非常实用的经验

不要默认让模型输出完整“思维链”。在工程上，更稳的做法通常是让模型输出：

-   当前计划摘要
    
-   选择某工具的简短理由
    
-   结构化的行动参数
    
-   工具结果后的下一步决策
    

这样既保留可审计性，也不会把大量自由文本推理暴露成新的上下文污染源。

## 工程细节与取舍

### 1\. 规划越早做，不代表越好

如果任务前期信息严重不足，过早生成长计划往往是假精确。此时更适合先做轻量探索，再逐渐细化计划。

### 2\. 并行不总是更快

如果并行结果后续很难汇总，或者每个 worker 都需要大量共享上下文，最终可能更慢、更贵、更难调。

### 3\. evaluator 要有稳定标准

否则它只是在“再调用一次模型”。一个有用的 evaluator，应该有清晰评分维度，如：

-   是否调用了正确工具
    
-   是否遗漏关键字段
    
-   是否违反安全策略
    
-   输出结构是否合规
    

## 常见误区

### 误区 1：ReAct 就是万能模板

不对。ReAct 很通用，但不是所有任务都需要边想边做。固定流程任务直接 workflow 往往更稳。

### 误区 2：先规划一定比直接执行更聪明

不对。复杂计划建立在错误前提上时，只会让后续执行更偏。

### 误区 3：工具多就代表能力强

不对。能力来自“正确调用合适工具”，不是“工具列表很长”。

## 面试与实战问答

### 问：什么是 ReAct？

短答： 一种让模型交替进行推理和行动的 Agent 模式。

深答： 核心价值是把“思考”和“调用外部环境”串成闭环，使模型能边获取信息边修正路径。

### 问：什么时候适合 plan-and-execute？

短答： 任务规模较大、适合先拆阶段，再逐步推进时。

深答： 比如研究、项目规划、文档编制。这类任务更需要全局结构感，但要允许计划在执行中重写。

### 问：Agent 工具设计最重要的原则是什么？

短答： 边界清晰、参数强类型、返回结构化、副作用可控、尽量幂等。

深答： 如果工具设计得像自然语言黑箱，Agent 很容易退化成“乱试型自动化”。
