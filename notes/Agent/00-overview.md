# Agent

## 学习目标

-   建立一套对 Agent 的统一认知：知道它是什么、为什么需要它、什么时候不该用它。
    
-   能把 Agent 拆成可工程化的部件：模型、指令、工具、状态、执行循环、记忆、评测、守护机制。
    
-   能从单 Agent 设计到多 Agent 协作，并解释这样拆分的收益、代价和常见失败点。
    
-   能把 Agent 相关面试题回答到“可追问”的深度，而不只是背几个术语。
    

## 适合人群

-   想系统学习 LLM Agent 的后端、平台、应用工程师。
    
-   已经在做 RAG、工作流编排、Copilot 或 AI 应用，但对“Agent 到底多了什么”还不够踏实的人。
    
-   需要准备大模型、AI 应用、智能体平台相关面试的人。
    

## 范围边界

-   本笔记聚焦软件 Agent，尤其是基于 LLM 的任务型 Agent、工具调用 Agent、工作流 Agent。
    
-   会简要提到具身 Agent、长时自治、多 Agent，但不会展开机器人控制、强化学习细节。
    
-   不把某个框架当作“标准答案”。OpenAI Agents SDK、LangGraph、MCP 等只作为理解和工程落地的参考。
    
-   不把 Agent 等同于“万能自动驾驶程序”。本笔记强调可控、可评测、可恢复的工程系统。
    

## 前置知识

-   LLM 基础：prompt、上下文窗口、函数调用、结构化输出。
    
-   工程基础：HTTP/API、数据库、异步任务、日志与监控。
    
-   应用基础：RAG、工作流、状态机、权限控制。
    

## 一页结论

-   一个实用定义：`Agent = Model + Instructions + Tools + State + Loop + Policy`。
    
-   Agent 和普通问答应用的区别，不在于“会不会聊天”，而在于它能否围绕目标进行多步执行、调用外部能力、根据中间结果调整路径，并在满足退出条件前持续推进。
    
-   Agent 不等于“越自主越好”。越自主，越需要状态管理、可观测性、守护机制和人工兜底。
    
-   真正难的不是让模型“思考”，而是把工具、副作用、记忆、重试、失败恢复、安全边界做成稳定系统。
    
-   工程上通常应先把单 Agent 做好，再在明确的边界上拆为多 Agent。过早多 Agent 往往会增加路由成本、状态同步复杂度和调试难度。

## 术语表

| 术语  | 本笔记中的含义 |
| --- | --- |
| Agent | 围绕目标执行多步任务的 LLM 驱动系统，不只是一次性文本生成 |
| Run | 一次完整执行，从接收输入到结束条件成立 |
| Turn | Agent 在一次 run 内的一轮模型决策 |
| Tool | Agent 可调用的外部能力，可能是查询、写操作、另一个 Agent、MCP 服务等 |
| Observation | 模型看到的中间结果，如工具返回值、环境状态、用户反馈 |
| State | 当前 run 或会话中的显式状态，供模型和运行时读取 |
| Memory | 为未来决策保留的信息，可以是短期上下文，也可以是长期存储 |
| Guardrail | 对输入、输出、工具调用和行为边界的限制与校验 |
| Handoff | 一个 Agent 把控制权移交给另一个 Agent |

## 本笔记的学习完成标准

如果你读完后能做到下面这些事，就说明这套笔记达标了：

-   不看资料，口头解释 Agent、workflow、RAG、tool calling、multi-agent 之间的关系。
    
-   针对一个任务，判断该用普通 LLM、工作流，还是 Agent。
    
-   画出一个单 Agent 或多 Agent 的最小可行架构，并指出主要风险点。
    
-   解释为什么记忆、幂等、副作用、重试、人工审批是 Agent 系统的关键难点。
    
-   面试里被追问“为什么要拆多 Agent”“如何评测”“如何防 prompt injection”时，能给出工程化答案。
