# 面试题与实战设计题

## 本章目标

-   把前面七章压缩成可直接回答的面试语言。
    
-   提供常见问答模板和系统设计思路。
    
-   帮你把“知道概念”转成“会讲、会拆、会权衡”。
    

## 核心问题

-   面试官最爱追问什么？
    
-   如何避免把 Agent 讲成空泛 buzzword？
    
-   系统设计题应该按什么顺序展开？
    

## 正文

### 1\. 高频问答

#### Q1：什么是 Agent？

短答： Agent 是围绕目标执行多步任务的 LLM 系统。它能在执行循环里调用工具、根据中间结果调整路径，并在满足退出条件前持续推进任务。

展开答法：

-   重点不只是“会回答”，而是“会执行”。
    
-   一般由模型、指令、工具、状态、loop、guardrails 组成。
    
-   和普通聊天应用的差别，在于它是一个运行时系统。
    

#### Q2：Agent 和 RAG 的区别是什么？

短答： RAG 是给模型补知识，Agent 是让模型在循环中做决策并执行任务。

展开答法：

-   RAG 通常回答“该知道什么”。
    
-   Agent 还要回答“下一步该做什么”。
    
-   在工程上，RAG 常常只是 Agent 的一个数据工具。
    

#### Q3：Agent 和工作流的区别是什么？

短答： 工作流更强调预定义步骤，Agent 更强调运行时决策。

展开答法：

-   如果流程非常确定，用工作流更稳。
    
-   如果中间分支依赖实时观察，Agent 更合适。
    
-   生产上常见做法是“工作流控主干，Agent 控局部复杂节点”。
    

#### Q4：Agent 的最小组成有哪些？

短答： 模型、指令、工具、状态、执行循环、守护机制。

展开答法：

-   模型负责决策
    
-   工具负责连接外部能力
    
-   状态负责执行进度
    
-   loop 负责持续推进
    
-   guardrails 负责边界控制
    

#### Q5：为什么很多场景要先做单 Agent？

短答： 因为单 Agent 更容易评测、调试、维护和治理。

展开答法：

-   多 Agent 会引入路由、handoff、共享记忆、状态同步等额外复杂度。
    
-   如果单 Agent 还没跑顺，多 Agent 往往只是把问题拆散，并没有解决问题。
    

#### Q6：什么时候拆多 Agent？

短答： 当单 Agent 出现 prompt 过载、工具过载、天然分域、权限边界冲突时。

展开答法：

-   研究和执行分离
    
-   路由和专业处理分离
    
-   只读和可写能力分离
    
-   生成和审核分离
    

#### Q7：manager 和 handoff 的区别是什么？

短答： manager 是中心调度，handoff 是控制权转移。

展开答法：

-   manager 更适合统一用户体验和集中治理
    
-   handoff 更适合路由和专业域接管
    

#### Q8：Agent 的 memory 应该怎么做？

短答： 分层做：上下文、显式状态、会话记忆、长期结构化记忆、外部检索。

展开答法：

-   不是所有信息都放消息历史里
    
-   当前执行态和长期偏好要分开
    
-   对大规模信息更适合检索，而不是硬塞上下文
    

#### Q9：Agent 为什么更容易出现安全问题？

短答： 因为它会读取不可信输入，还可能基于这些输入调用真实工具。

展开答法：

-   风险不只是输出错误
    
-   还包括越权读写、数据泄露、错误执行和流程误导
    

#### Q10：如何评测一个 Agent？

短答： 要同时看结果、路径、安全、成本和延迟。

展开答法：

-   最终任务成功率
    
-   工具调用正确率
    
-   trace 是否存在坏路径
    
-   unsafe action rate
    
-   平均轮数和成本
    

### 2\. 一套回答系统设计题的顺序

面试里遇到“设计一个 Agent 系统”，建议固定按下面顺序答：

1.  先定义任务边界
    
2.  判断是否真的需要 Agent
    
3.  设计最小闭环
    
4.  设计工具和数据边界
    
5.  设计状态和记忆
    
6.  设计 guardrails 和人工兜底
    
7.  设计评测与观测
    
8.  最后再谈是否要多 Agent
    

这个顺序的好处是：你不会一上来就谈 fancy 架构，而是先证明自己理解业务和风险。

### 3\. 设计题示例：做一个企业客服退款 Agent

#### 3.1 任务理解

目标：

-   回答订单与退款问题
    
-   能查订单状态
    
-   能判断是否满足退款条件
    
-   对高风险退款触发人工审批
    

#### 3.2 先判断是否需要 Agent

这个场景适合做受控 Agent，而不是纯工作流，原因是：

-   用户表达自然语言且不稳定
    
-   需要多轮澄清
    
-   需要查多个系统
    
-   需要在策略允许范围内决定下一步
    

但也不能让它完全自治，因为退款是有副作用和风险的动作。

#### 3.3 最小闭环设计

-   输入：用户请求
    
-   数据工具：查订单、查退款政策、查历史工单
    
-   动作工具：创建退款申请、升级人工
    
-   显式状态：订单 ID、资格判断结果、审批状态、已发起动作
    
-   退出条件：给出答复、创建申请成功、升级人工
    

#### 3.4 工具设计

-   `get_order(order_id)`：只读
    
-   `get_refund_policy(order_type, order_age)`：只读
    
-   `create_refund_request(order_id, reason)`：可写，需审批
    
-   `escalate_to_human(ticket_payload)`：可写，但低风险
    

#### 3.5 Guardrails

-   输入越权检测
    
-   敏感信息脱敏
    
-   高风险工具调用审批
    
-   结构化输出约束
    
-   最大轮数限制
    

#### 3.6 评测

-   能否正确路由订单问题与退款问题
    
-   能否正确判断退款资格
    
-   是否误发退款
    
-   是否在不确定时升级人工
    

#### 3.7 是否拆多 Agent

初期不建议。先做单 Agent。

后续若出现：

-   路由复杂
    
-   政策判断复杂
    
-   退款执行需独立审计
    

可以拆成：

-   triage Agent
    
-   policy Agent
    
-   execution Agent
    

### 4\. 设计题示例：做一个代码修复 Agent

回答框架可以这样讲：

-   输入：issue、报错日志、代码仓库
    
-   数据工具：代码搜索、读文件、测试结果、文档检索
    
-   动作工具：修改文件、运行测试、提交 patch
    
-   状态：已读文件、候选根因、修改记录、测试结果、回滚点
    
-   守护：限制可写路径、限制危险命令、强制测试、人工 review
    
-   模式：单 Agent + ReAct 起步，复杂时加 reviewer Agent
    

重点亮点：

-   说明为什么 durable execution 很重要
    
-   说明为什么写操作需要幂等和可回滚
    
-   说明为什么 trace 对调试 Agent 很关键
    

## 工程细节与取舍

### 1\. 面试里不要把所有场景都说成多 Agent

这通常会暴露你对复杂度成本不敏感。成熟回答往往会先说“先单 Agent，再按边界拆”。

### 2\. 讲安全时不要只说 prompt injection

还应该提：

-   权限控制
    
-   审批
    
-   数据泄露
    
-   审计
    
-   幂等和回滚
    

### 3\. 讲评测时不要只说 benchmark

要提：

-   回放
    
-   trace grading
    
-   工具调用正确率
    
-   高风险用例人工评审
    

## 常见误区

### 误区 1：回答 Agent 面试题时全讲论文

论文重要，但面试官更关心你会不会把系统做稳。

### 误区 2：设计题一上来就画复杂架构

如果没先说明业务边界、风险、工具、副作用，复杂图反而显得不落地。

### 误区 3：一谈安全就说“加 guardrail”

这太空了。更好的说法是结构化输出、审批、RBAC、trace、工具边界、人工介入。

## 面试与实战问答

### 问：如果面试官问“为什么不用普通工作流”，怎么答？

短答： 如果路径高度确定，我就用工作流；只有在中间决策依赖实时观察时，才会让 Agent 接管局部节点。

### 问：如果面试官问“多 Agent 一定更好吗”，怎么答？

短答： 不一定。它主要解决复杂度和职责隔离，不保证成功率线性提升。

### 问：如果面试官问“你最担心 Agent 的什么问题”，怎么答？

短答： 我最担心的是不受控副作用，其次是状态恢复和安全边界不清。

展开答法： 因为这些问题不会只导致“答错”，而会导致真实业务后果。
