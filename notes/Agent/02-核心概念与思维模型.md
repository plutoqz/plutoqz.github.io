# 核心概念与思维模型

## 本章目标

-   把 Agent 拆成最关键的抽象部件。
    
-   建立几种工程上最好用的思维模型，而不是停留在概念堆砌。
    
-   理清工具、状态、记忆、规划、守护机制之间的关系。
    

## 核心问题

-   Agent 最小由哪些部件构成？
    
-   工具、状态、记忆分别解决什么问题？
    
-   为什么很多 Agent 问题本质上是运行时设计问题？
    

## 正文

### 1\. 一个工程上好用的公式

本笔记把 Agent 抽象成：

`Agent = Model + Instructions + Tools + State + Loop + Policy`

这是作者归纳，但非常适合系统设计：

-   `Model`：负责理解、选择、生成。
    
-   `Instructions`：定义角色、目标、边界、风格和流程偏好。
    
-   `Tools`：给模型接入外部能力。
    
-   `State`：让系统知道“当前进展到哪里了”。
    
-   `Loop`：让系统可以根据观察继续执行，而不是一次性结束。
    
-   `Policy`：对输入、输出、权限、审批、安全和合规进行约束。
    

如果六个部分里少掉一两个，系统通常会退化成普通 LLM 应用或脆弱脚本。

### 2\. Model：决策器，不是整个系统

LLM 在 Agent 中更像“策略引擎”或“决策器”，而不是整个应用本身。它负责：

-   理解用户意图和上下文。
    
-   判断是否需要工具。
    
-   决定调用哪个工具、传什么参数。
    
-   整合工具返回结果。
    
-   在中间结果不理想时决定下一步。
    

但它并不天然擅长：

-   长期精确状态管理
    
-   幂等控制
    
-   严格权限判断
    
-   可靠重放与恢复
    
-   复杂副作用编排
    

所以不要把“模型够强”误解为“系统足够稳”。

### 3\. Instructions：提示词不是文案，而是策略

很多人把 system prompt 当成“描述人格”的地方，但在 Agent 系统里，它更像运行策略。

好的 instructions 通常要同时回答：

-   目标是什么？
    
-   允许做什么？
    
-   不允许做什么？
    
-   遇到不确定情况怎么办？
    
-   何时调用工具？
    
-   何时请求人工确认？
    
-   输出必须满足什么结构？
    

OpenAI 的 Agent 指南建议复用现有操作手册、知识库政策、流程文档，把它们转成 LLM 更容易执行的结构化指令。这一点非常实用，因为它让 prompt 从“灵感写作”变成“策略编写”。

### 4\. Tools：Agent 的外部行动面

没有工具，Agent 的“行动”大多只是文本生成。工具把它接到真实世界。

OpenAI 在实用指南里把工具分成三类：

| 工具类型 | 作用  | 例子  |
| --- | --- | --- |
| 数据工具 | 取上下文和事实 | 搜索、数据库查询、文件读取、RAG 检索 |
| 动作工具 | 对外部系统执行操作 | 发邮件、写工单、创建日历、更新 CRM |
| 编排工具 | 把其他 Agent 或子流程当能力暴露 | 研究 Agent、退款 Agent、代码修复 Agent |

这张表有两个价值：

-   它提醒你“工具不只是查数据”，还包括真正有副作用的操作。
    
-   它提醒你“另一个 Agent 本身也可以被当成工具”。
    

### 5\. State：Agent 不是只活在 prompt 里

State 是很多入门资料最容易忽略的部分。它描述当前执行所处的位置和中间结果，常见内容包括：

-   当前任务 ID
    
-   已完成步骤
    
-   工具调用结果
    
-   错误计数
    
-   重试次数
    
-   审批状态
    
-   当前计划
    
-   待办子任务列表
    

为什么 state 很重要？

-   没有 state，Agent 只能“重读历史猜进度”。
    
-   state 显式化以后，系统才容易做恢复、回放、审计和人机接力。
    

一个重要区分：

-   `State` 更偏当前执行态。
    
-   `Memory` 更偏跨轮次、跨 run 保留的信息。
    

### 6\. Loop：Agent 的本质在循环

Agent 和一次性生成应用的分水岭，往往就在 loop。

一个典型循环是：

1.  读当前输入和状态。
    
2.  调用模型做决策。
    
3.  如果需要工具，则执行工具。
    
4.  把观察结果写回状态。
    
5.  判断是否结束，否则继续下一轮。
    

OpenAI 的 Agents SDK 文档明确提供了内建 agent loop，并把“直到任务完成为止持续处理工具调用”作为核心特性之一。这说明 loop 不是实现细节，而是 Agent 的第一性原理。

### 7\. Policy：没有约束的 Agent 只是高风险自动化

Policy 可以理解为系统的治理层，包括：

-   输入过滤
    
-   输出校验
    
-   工具权限
    
-   审批开关
    
-   敏感动作确认
    
-   结构化输出约束
    
-   访问控制
    
-   审计记录
    

如果把 Agent 直接接到真实系统，没有 policy，就相当于给一个高不确定性的决策器开了生产权限。

### 8\. 四个非常重要的思维模型

#### 8.1 把 Agent 当成状态机

不要把 Agent 想成“会思考的生命体”，而要把它想成“由模型驱动状态转移的状态机”。

这样你会自然去问：

-   当前状态是什么？
    
-   哪些输入会触发状态转移？
    
-   什么情况下终止？
    
-   什么情况下转人工？
    

#### 8.2 把工具当成能力边界

模型能做什么，最终由工具决定上限；模型会不会乱做，则由工具定义和权限设计决定下限。

一个好工具要做到：

-   名称清晰
    
-   参数清晰
    
-   副作用明确
    
-   结果可校验
    
-   尽量幂等
    

#### 8.3 把记忆当成上下文预算分配

记忆不是“人格化大脑”，而是一个上下文预算系统。核心问题是：

-   什么值得保留？
    
-   保留多久？
    
-   存哪里？
    
-   什么时候压缩、总结、淘汰？
    

#### 8.4 把提示词当成策略配置，不当成业务逻辑本体

真正关键的流程控制、权限判断、回滚、幂等，尽量放在代码和运行时里。Prompt 更适合描述目标、偏好、决策准则和异常处置原则。

## 工程细节与取舍

### 1\. Tool schema 比“写得很长的工具描述”更重要

工具说明当然重要，但很多时候真正决定稳定性的，是：

-   参数是否强类型
    
-   是否要求必填字段
    
-   是否有枚举约束
    
-   返回结果是否结构化
    

因为这决定模型和运行时之间是“自然语言接口”还是“类型化接口”。

### 2\. State 应尽量显式，不要全靠历史消息隐式推断

隐式状态的问题是：

-   难恢复
    
-   难审计
    
-   难让其他 Agent 接手
    
-   难人工介入
    

好的 Agent 系统通常同时保存消息历史和显式状态。

### 3\. Policy 应与业务风险匹配

只读搜索 Agent、代码解释 Agent、企业知识库问答 Agent，对政策强度的要求差异极大。不要为了“统一平台设计”把所有 Agent 都做成同一个安全等级。

## 常见误区

### 误区 1：Prompt 就是 Agent

不对。Prompt 只是 Agent 的一部分。缺少 loop、tools、state、policy，顶多算能力增强的聊天应用。

### 误区 2：记忆越多越好

不对。过多记忆会带来上下文污染、召回误差、成本上升和隐私风险。

### 误区 3：工具越多越强

不对。工具太多且边界重叠，会让模型选错工具。OpenAI 的指南提到，问题不只是工具数量，更是工具之间是否重叠、描述是否清晰。

### 误区 4：有 guardrails 就安全了

不对。Guardrails 是分层防线之一，不替代鉴权、审批、沙箱、日志、限流和人工复核。

## 面试与实战问答

### 问：Agent 的最小组成是什么？

短答： 模型、指令、工具、状态、执行循环、约束策略。

深答： 其中模型负责决策，工具负责行动，状态负责进度，循环负责持续执行，策略负责边界控制。缺少其中任一项，系统会退化。

### 问：状态和记忆有什么区别？

短答： 状态更偏当前执行态，记忆更偏跨轮次保留的信息。

深答： 状态强调当前 run 是否可恢复、可追踪；记忆强调后续决策是否能继承上下文。

### 问：为什么说 Agent 的本质是 loop？

短答： 因为 Agent 需要根据中间观察决定下一步，而不是一次性给出最终文本。

深答： 只要没有循环，系统就很难真正处理动态多步任务，最多只是“先想好一套步骤再输出”。
