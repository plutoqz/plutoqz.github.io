
# 力扣 322 题：零钱兑换

## 题目理解

给定一个整数数组 `coins`，表示不同面额的硬币；再给定一个整数 `amount`，表示总金额。

每种硬币都可以使用 **无限次**，要求凑出总金额 `amount` 所需的 **最少硬币个数**。如果无法凑出，返回 `-1`。

例如：

- `coins = [1,2,5], amount = 11`
- 最优方案是 `5 + 5 + 1`
- 返回 `3`

---

## 这题的本质

这题本质上是：

- **完全背包问题**
- 也可以理解成：**从 0 到 amount 的最短路径问题**
- 状态目标是：**最少使用多少枚硬币**

注意它和“零钱兑换 II（力扣 518）”不一样：

- **322**：求最少硬币数，核心是 `min`
- **518**：求组合总数，核心是 `count`

---

## 为什么不能用贪心

很多人第一反应会觉得：

> 每次优先选最大面额硬币，不就好了？

这是错的。

例如：

- `coins = [1, 3, 4]`
- `amount = 6`

贪心会选：

- `4 + 1 + 1 = 3 枚`

但最优是：

- `3 + 3 = 2 枚`

所以这题 **不能直接用贪心**，必须用搜索 / 动态规划。

---

## 解法总览

按“次优到最优”的顺序来看：

| 解法 | 思路 | 时间复杂度 | 空间复杂度 | 适用场景 |
|---|---|---:|---:|---|
| 解法一 | 回溯 + 排序剪枝 | 最坏指数级 | `O(n)` 递归栈 | 适合理解搜索过程，不适合作为主解 |
| 解法二 | 记忆化搜索 | `O(n * amount)` | `O(amount)` | 从递归自然过渡到 DP |
| 解法三 | 一维动态规划（完全背包） | `O(n * amount)` | `O(amount)` | 面试首选、最稳妥、工程中也最常用 |

---

## 解法一：回溯 + 排序剪枝

### 思路

先将硬币按面额从大到小排序，优先尝试大面额。

例如：

- 当前硬币是 `5`
- `remain = 11`
- 那么最多能用 `2` 个 `5`
- 就依次尝试：
  - 用 `2` 个 `5`，剩下 `1`
  - 用 `1` 个 `5`，剩下 `6`
  - 用 `0` 个 `5`，剩下 `11`

这样做的好处是：

- 更容易更快得到一个较优解
- 一旦当前使用的硬币数已经不可能比当前最优答案更小，就可以剪枝

### 剪枝核心

设当前已经用了 `used` 枚硬币，当前面额最多还能选 `count` 枚：

- 如果 `used + count >= ans`
- 那么即使后面能凑出来，也不会更优
- 可以直接跳过

### 复杂度

- 最坏情况下依然是 **指数级**
- 只是通过剪枝在很多数据上会快很多
- 不适合作为标准答案，但适合帮助理解“搜索 + 剪枝”

### Java 代码

```java
import java.util.Arrays;

class Solution {
    private int ans = Integer.MAX_VALUE;

    public int coinChange(int[] coins, int amount) {
        if (amount == 0) {
            return 0;
        }

        Arrays.sort(coins); // 先升序排序
        reverse(coins);     // 再翻转成降序，优先用大硬币

        dfs(coins, 0, amount, 0);

        return ans == Integer.MAX_VALUE ? -1 : ans;
    }

    private void dfs(int[] coins, int index, int remain, int used) {
        if (remain == 0) {
            ans = Math.min(ans, used); // 找到一种可行解，更新最优答案
            return;
        }

        if (index == coins.length) {
            return; // 没有硬币可选了
        }

        int coin = coins[index];
        int maxCount = remain / coin; // 当前面额最多可选多少枚

        // 从多选到少选，优先尝试大面额数量更多的情况
        for (int count = maxCount; count >= 0; count--) {
            int nextUsed = used + count;

            // 剪枝：当前已用硬币数已经不可能优于最优解
            if (nextUsed >= ans) {
                continue;
            }

            int nextRemain = remain - count * coin;
            dfs(coins, index + 1, nextRemain, nextUsed);
        }
    }

    private void reverse(int[] nums) {
        int left = 0, right = nums.length - 1;
        while (left < right) {
            int temp = nums[left];
            nums[left] = nums[right];
            nums[right] = temp;
            left++;
            right--;
        }
    }
}
```

### 适用场景

- 适合讲清楚“这题可以搜索”
- 适合引出“为什么要记忆化”
- 如果面试官允许你从暴力优化到 DP，这个过程很加分

## 解法二：记忆化搜索

### 思路

定义函数：

- `dfs(remain)`：凑出金额 `remain` 所需的最少硬币数

那么：

- 如果 `remain == 0`，答案就是 `0`
- 如果 `remain < 0`，说明无解，返回 `-1`
- 否则枚举每一种硬币 `coin`：
  - 如果先用了一个 `coin`
  - 那么问题就变成：`dfs(remain - coin)`
  - 若子问题有解，则当前答案候选是 `dfs(remain - coin) + 1`

递推式可以理解为：

```text
dfs(remain) = min(dfs(remain - coin) + 1)
```

其中 `coin` 遍历所有硬币。

### 为什么要记忆化

如果不用记忆化，会出现大量重复计算。

例如求 `dfs(11)` 时，会递归到：

- `dfs(10)`
- `dfs(9)`
- `dfs(6)`

而 `dfs(6)` 在别的路径里也可能反复被计算。

所以用数组 `memo` 缓存结果：

- `memo[x] = -2`：没算过
- `memo[x] = -1`：无解
- `memo[x] >= 0`：最少硬币数

### 复杂度

- 每个 `remain` 最多只会算一次
- 每次计算遍历所有硬币

所以：

- 时间复杂度：`O(n * amount)`
- 空间复杂度：`O(amount)`

### Java 代码

```java
import java.util.Arrays;

class Solution {
    private int[] coins;
    private int[] memo;

    public int coinChange(int[] coins, int amount) {
        this.coins = coins;
        this.memo = new int[amount + 1];

        Arrays.fill(this.memo, -2); // -2 表示未计算
        this.memo[0] = 0;           // 凑出 0 元需要 0 枚硬币

        return dfs(amount);
    }

    private int dfs(int remain) {
        if (remain < 0) {
            return -1; // 金额变成负数，无解
        }

        if (memo[remain] != -2) {
            return memo[remain]; // 之前已经计算过，直接返回
        }

        int minCount = Integer.MAX_VALUE / 2; // 防止后续 +1 溢出

        for (int coin : coins) {
            int sub = dfs(remain - coin); // 先用一枚当前硬币
            if (sub >= 0) {
                minCount = Math.min(minCount, sub + 1);
            }
        }

        memo[remain] = (minCount == Integer.MAX_VALUE / 2) ? -1 : minCount;
        return memo[remain];
    }
}
```

### 适用场景

- 适合递归思维强的人
- 适合从“暴力递归”自然优化到“DP”
- 但在 Java 中，递归层数过深时可能有栈风险，所以工程和面试中通常更推荐下一种 **自底向上的 DP**

## 解法三：一维动态规划（完全背包，最优主解）

这是这题最经典、最稳定、最推荐的写法。

---

### 状态定义

定义：

- `dp[i]`：凑出金额 `i` 所需的最少硬币数

目标就是求：

- `dp[amount]`

---

### 初始值

- `dp[0] = 0`
  - 凑出 0 元不需要任何硬币
- 其他位置初始设为一个“不可能的大值”
  - 常用 `amount + 1`

为什么 `amount + 1` 可以？

- 因为如果存在解，最多也不会超过 `amount` 枚硬币（极端情况下全用面额 1）
- 如果没有面额 1，也没关系，`amount + 1` 只是一个哨兵值，用来表示“还没法凑出来”

---

### 状态转移

对于每个硬币 `coin`：

如果要更新金额 `i`，并且 `i >= coin`：

```text
dp[i] = min(dp[i], dp[i - coin] + 1)
```

含义是：

- 如果最后一枚硬币选了 `coin`
- 那么前面必须先凑出 `i - coin`
- 总硬币数就是 `dp[i - coin] + 1`

---

### 为什么这是完全背包

因为每种硬币可以用无限次。

在完全背包中，内层循环应当：

- **从小到大遍历金额**

也就是：

```java
for (int i = coin; i <= amount; i++)
```

这样当前轮更新 `dp[i]` 时，`dp[i - coin]` 可能已经用过当前硬币，从而达到“可重复使用”的效果。

如果你写成从大到小，那就变成了 **0/1 背包** 的思路，这是错的。

---

### 复杂度

- 时间复杂度：`O(n * amount)`
- 空间复杂度：`O(amount)`

这是这题最优秀、最稳的通用解法。

---

### Java 代码

```java
import java.util.Arrays;

class Solution {
    public int coinChange(int[] coins, int amount) {
        int[] dp = new int[amount + 1];
        int INF = amount + 1; // 哨兵值，表示当前金额还无法凑出

        Arrays.fill(dp, INF);
        dp[0] = 0; // 凑出 0 元需要 0 枚硬币

        for (int coin : coins) {
            for (int i = coin; i <= amount; i++) {
                dp[i] = Math.min(dp[i], dp[i - coin] + 1); // 完全背包：同一种硬币可重复使用
            }
        }

        return dp[amount] == INF ? -1 : dp[amount];
    }
}
```

## 用例推演

以：

- `coins = [1, 2, 5]`
- `amount = 11`

为例。

初始：

```text
dp[0] = 0
其他都为 INF
```

### 处理硬币 1

所有金额都能由 1 凑出来：

```text
dp[1] = 1
dp[2] = 2
dp[3] = 3
...
dp[11] = 11
```

### 处理硬币 2

尝试更新：

- `dp[2] = min(2, dp[0] + 1) = 1`
- `dp[3] = min(3, dp[1] + 1) = 2`
- `dp[4] = min(4, dp[2] + 1) = 2`
- `dp[6] = min(6, dp[4] + 1) = 3`

此时很多值都会变小。

### 处理硬币 5

继续更新：

- `dp[5] = min(3, dp[0] + 1) = 1`
- `dp[10] = min(5, dp[5] + 1) = 2`
- `dp[11] = min(6, dp[6] + 1) = 3`

最终：

```text
dp[11] = 3
```

## 这题的关键技巧

### 1. 无解状态要用“大值”表示

不能一开始把 `dp[i]` 默认设为 `0`，否则你没法区分：

- “真的只需要 0 枚”
- “还没算出来”

所以要用 `INF` 表示“不可能/未到达”。

---

### 2. 防止整数溢出

如果你写：

```java
int INF = Integer.MAX_VALUE;
dp[i] = Math.min(dp[i], dp[i - coin] + 1);
```

那么当 `dp[i - coin]` 本身就是 `Integer.MAX_VALUE` 时，再 `+1` 就溢出了。

所以更推荐：

- `INF = amount + 1`
- 或者 `Integer.MAX_VALUE / 2`

这是非常常见的面试细节。

---

### 3. 完全背包的循环方向要对

这题是完全背包，因此：

```java
for (int i = coin; i <= amount; i++)
```

是正序。

如果写成倒序：

```java
for (int i = amount; i >= coin; i--)
```

那是 0/1 背包写法，会错误地限制每种硬币只能用一次。

---

### 4. `amount = 0` 要直接返回 0

因为不需要任何硬币。

这是边界条件，面试中常被问到。

---

### 5. 不要把 322 和 518 混了

很多人学到“零钱兑换”时容易混淆：

#### 力扣 322
- 求最少硬币数
- 用 `min`

#### 力扣 518
- 求组合数
- 用 `+=`

转移完全不同。

---

## 面试时推荐怎么说

如果这是面试题，建议表达顺序如下：

### 第一层：先排除贪心
先说：

> 这题不能直接贪心，因为有反例，比如 `[1,3,4]` 凑 `6`。

这一步很重要，说明你没有机械套模板。

---

### 第二层：说出本质
然后说：

> 这是一个完全背包问题。每种硬币可以无限使用，目标是求最少硬币数量。

这会让面试官快速知道你识别题型的能力。

---

### 第三层：给出状态定义
继续说：

> 设 `dp[i]` 表示凑出金额 `i` 所需的最少硬币数，初始 `dp[0]=0`，其余设为无穷大。

---

### 第四层：给出转移
再说：

> 如果最后一枚硬币选 `coin`，那么 `dp[i] = min(dp[i], dp[i - coin] + 1)`。

---

### 第五层：强调循环方向
最后补一句：

> 因为硬币可以重复使用，所以这是完全背包，金额要正序遍历。

这句经常是面试加分点。

## 实际工作中的建议

虽然这题是算法题，但放到工程里也有一些值得注意的点。

### 1. 如果有大量重复查询，优先考虑预处理

如果：

- 硬币面额集合固定
- 但会反复查询很多不同 `amount`

那么最好的做法不是每次重新算，而是：

- 先求出某个最大金额范围内的 `dp`
- 后续查询直接 `O(1)` 返回

这在计费系统、券组合计算、资源打包等场景里很常见。

---

### 2. 要清楚“最少数量”和“最优成本”不是一回事

现实业务里“硬币数最少”只是抽象模型。

实际问题里往往变成：

- 最少张数
- 最低手续费
- 最少包数
- 最少机器数

这时候状态转移思想不变，但“代价”定义会变。

---

### 3. 数据规模决定你选递归还是迭代

在 Java 项目里：

- 递归写法更直观
- 但大输入下要注意栈深度

实际工程中，迭代版 DP 通常更稳，也更容易监控和优化。

---

### 4. 先验证题目是不是“无限使用”

很多业务题看起来像零钱兑换，但实际约束可能是：

- 每种资源只能用一次
- 每种资源只能用固定次数
- 顺序不同算不同方案

这些条件一变，模型就不再是这道题了。

## 推荐记忆模板

这题最值得记住的是这一版：

```java
import java.util.Arrays;

class Solution {
    public int coinChange(int[] coins, int amount) {
        int[] dp = new int[amount + 1];
        int INF = amount + 1;

        Arrays.fill(dp, INF);
        dp[0] = 0;

        for (int coin : coins) {
            for (int i = coin; i <= amount; i++) {
                dp[i] = Math.min(dp[i], dp[i - coin] + 1);
            }
        }

        return dp[amount] == INF ? -1 : dp[amount];
    }
}
```

你可以把它作为“完全背包求最小值”的标准模板。

## 总结

这题核心就三句话：

1. **不能贪心，因为存在反例**
2. **这是完全背包问题**
3. **`dp[i] = min(dp[i], dp[i - coin] + 1)`**

如果是面试，最推荐主答：

- **一维 DP（完全背包）**

如果面试官喜欢看你推导过程，可以先从：

- 回溯
- 再到记忆化搜索
- 最后落到迭代 DP

这个完整链路会非常漂亮。
