# 力扣 139 题：单词拆分

## 题目理解

给定一个字符串 `s` 和一个字符串列表 `wordDict`，判断 `s` 能不能被拆分成若干个字典中的单词，字典中的单词可以重复使用。

例如：

-   `s = "leetcode"`
    
-   `wordDict = ["leet", "code"]`
    

结果是 `true`，因为可以拆成 `"leet" + "code"`。

---

## 这题的本质

这题本质上是在问：

> 从左到右扫描字符串，某个位置之前如果已经可以被成功拆分，那么后面再接上一个字典中的单词后，新的位置是否也可以被成功拆分？

所以它是一个非常经典的：

-   **状态可达问题**
    
-   **动态规划问题**
    
-   也可以用 **记忆化搜索**
    
-   进一步还能用 **Trie（前缀树）优化匹配过程**
    

---

## 解法一：记忆化搜索（DFS + Memo）

### 思路

定义函数：

-   `dfs(start)` 表示：从下标 `start` 开始的后缀字符串，能否被成功拆分
    

递归过程：

1.  如果 `start == s.length()`，说明整个字符串都拆完了，返回 `true`
    
2.  枚举字典中的每个单词 `word`
    
3.  如果 `s` 从 `start` 开始能匹配上这个 `word`
    
4.  那么递归判断 `dfs(start + word.length())`
    
5.  只要有一个分支返回 `true`，整体就返回 `true`
    

为了避免重复计算，用一个 `memo` 数组记录：

-   `-1`：没算过
    
-   `0`：不能拆分
    
-   `1`：可以拆分
    

---

### 为什么需要记忆化

比如字符串是：

```text
aaaaaaa...
```

字典中有很多 `"a"`, `"aa"`, `"aaa"` 之类的单词时，同一个位置可能会被反复递归到很多次。

记忆化的作用就是：

-   某个 `start` 一旦算过，以后直接拿答案
    
-   把指数级递归压到多项式级别
    

---

### 复杂度

设：

-   `n` 是字符串长度
    
-   `m` 是字典单词数
    
-   `L` 是单词平均长度
    

时间复杂度大致可以看成：

-   **O(n \* m \* L)**
    

因为每个 `start` 最多计算一次，每次会遍历字典并用 `startsWith` 匹配单词。

空间复杂度：

-   **O(n)**，来自记忆化数组和递归栈
    

---

### 适用场景

-   适合先建立直觉
    
-   适合面试时先从递归推导出解法
    
-   如果面试官想看你如何把暴力递归优化成可接受复杂度，这个解法很合适
    

---

### Java 代码

```java
import java.util.Arrays;
import java.util.List;

class Solution {
    private String s;
    private List<String> wordDict;
    private int[] memo; // -1: 未计算, 0: false, 1: true

    public boolean wordBreak(String s, List<String> wordDict) {
        this.s = s;
        this.wordDict = wordDict;
        this.memo = new int[s.length()];
        Arrays.fill(this.memo, -1);
        return dfs(0);
    }

    private boolean dfs(int start) {
        // 递归终点：已经成功拆分到字符串末尾
        if (start == s.length()) {
            return true;
        }

        // 如果这个位置之前算过，直接返回记忆结果
        if (memo[start] != -1) {
            return memo[start] == 1;
        }

        // 枚举所有单词，尝试从 start 开始匹配
        for (String word : wordDict) {
            int len = word.length();

            // 剪枝：如果越界，直接跳过
            if (start + len > s.length()) {
                continue;
            }

            // 如果当前单词能匹配上当前位置
            if (s.startsWith(word, start)) {
                // 继续判断后续部分能否拆分
                if (dfs(start + len)) {
                    memo[start] = 1; // 记录当前位置可拆分
                    return true;
                }
            }
        }

        memo[start] = 0; // 记录当前位置不可拆分
        return false;
    }
}
```

## 解法二：动态规划 + 哈希表 + 长度剪枝

这是这道题最经典、最标准、面试最推荐的解法。

---

### 核心定义

定义：

-   `dp[i]` 表示：**前** `i` **个字符**，也就是 `s[0..i-1]`，是否可以被拆分
    

最终答案就是：

-   `dp[n]`
    

其中 `n = s.length()`

---

### 状态转移

如果要让 `dp[i] = true`，那么必须存在某个位置 `j`，满足：

1.  `dp[j] == true`
    
2.  `s[j..i-1]` 是字典中的一个单词
    

所以状态转移方程是：

```text
dp[i] = 存在某个 j < i，使得 dp[j] == true 且 s[j..i-1] ∈ wordDict
```

---

### 初始化

-   `dp[0] = true`
    

为什么？

因为空字符串可以认为“已经被成功拆分”。

这个初始化是整个 DP 能推进下去的起点。

---

### 为什么要用 HashSet

如果字典是 `List`，每次判断一个字符串是否在字典中，复杂度会比较高。

所以先把字典转换成：

```java
Set<String> wordSet = new HashSet<>(wordDict);
```

这样查找复杂度平均就是 **O(1)**。

---

### 为什么要做长度剪枝

普通 DP 会枚举：

-   `i` 从 `1` 到 `n`
    
-   `j` 从 `0` 到 `i-1`
    

这会有很多无意义判断。

实际上，字典里的单词长度是有限的。

如果字典中：

-   最短单词长度是 `minLen`
    
-   最长单词长度是 `maxLen`
    

那么在计算 `dp[i]` 时，只需要检查长度在 `[minLen, maxLen]` 范围内的子串即可。

这样可以显著减少枚举次数。

---

### 复杂度

设：

-   `n` 为字符串长度
    
-   `R = maxLen - minLen + 1`
    

时间复杂度大致为：

-   **O(n \* R \* k)**
    

其中 `k` 是子串构造/哈希相关开销<br>如果从算法思想上简化描述，通常会说是：

-   **O(n²)**
    

空间复杂度：

-   **O(n + |wordDict|)**
    

---

### 适用场景

-   单次判断，最推荐
    
-   面试标准答案
    
-   代码短、结构清晰、容易解释
    
-   如果只需要通过 LeetCode，这个通常已经足够好了
    

---

### Java 代码

```java
import java.util.HashSet;
import java.util.List;
import java.util.Set;

class Solution {
    public boolean wordBreak(String s, List<String> wordDict) {
        if (wordDict == null || wordDict.isEmpty()) {
            return s == null || s.length() == 0;
        }

        Set<String> wordSet = new HashSet<>(wordDict);
        int n = s.length();

        // 统计字典中的最短和最长单词长度，用于剪枝
        int minLen = Integer.MAX_VALUE;
        int maxLen = 0;
        for (String word : wordDict) {
            int len = word.length();
            minLen = Math.min(minLen, len);
            maxLen = Math.max(maxLen, len);
        }

        boolean[] dp = new boolean[n + 1];
        dp[0] = true; // 空字符串可拆分

        // i 表示前 i 个字符
        for (int i = 1; i <= n; i++) {
            // 只枚举可能的单词长度，避免无效遍历
            for (int len = minLen; len <= maxLen && len <= i; len++) {
                int start = i - len;

                // 如果前缀本身不可拆分，没必要继续判断这个切分点
                if (!dp[start]) {
                    continue;
                }

                // 检查 s[start..i-1] 是否是一个合法单词
                if (wordSet.contains(s.substring(start, i))) {
                    dp[i] = true; // 找到一种可行拆分
                    break;
                }
            }
        }

        return dp[n];
    }
}
```

## 解法三：Trie（前缀树）+ 动态规划

这是更偏优化和工程化的写法。

---

### 为什么还要 Trie

在上面的 DP 里，我们会频繁做两件事：

1.  截取 `substring`
    
2.  去 `HashSet` 里查
    

虽然思路很标准，但在 Java 里：

-   `substring` 会创建新字符串对象
    
-   高频调用时会有额外开销
    

如果字典很大，或者你要处理很多次查询，那么可以把字典构造成 **Trie**：

-   从字符串某个位置出发，沿 Trie 一路往后匹配
    
-   走到某个节点是单词结尾，就说明找到一个合法切分点
    
-   不需要频繁构造子串
    

---

### 核心思路

还是定义：

-   `dp[i]` 表示前 `i` 个字符能否拆分
    

但这次转移方式变成：

1.  如果 `dp[i] == true`，说明位置 `i` 可达
    
2.  从 `s[i]` 开始沿 Trie 往后走
    
3.  如果走到某个节点 `isWord == true`
    
4.  就说明 `s[i..j]` 是字典单词，于是令 `dp[j + 1] = true`
    

---

### 这个解法的优势

-   不需要大量构造 `substring`
    
-   匹配过程是“按字符走”
    
-   如果前缀不在字典树中，可以很早停止
    
-   更适合字典大、查询多、追求性能的场景
    

---

### 复杂度

设：

-   `n` 是字符串长度
    
-   `maxLen` 是字典中的最大单词长度
    
-   `T` 是所有字典字符总数
    

构建 Trie：

-   **O(T)**
    

DP 过程中，每个可达位置最多向后走 `maxLen` 步：

-   **O(n \* maxLen)**
    

总复杂度：

-   **O(T + n \* maxLen)**
    

空间复杂度：

-   **O(T + n)**
    

---

### 适用场景

-   字典很大
    
-   有很多前缀共享
    
-   有多次查询需求
    
-   面试官追问“还能怎么优化”
    
-   实际业务里做分词、词典匹配、敏感词检测时，这类思路非常常见
    

---

### Java 代码

```java
import java.util.List;

class Solution {
    private static class TrieNode {
        TrieNode[] children = new TrieNode[26];
        boolean isWord; // 标记当前节点是否为一个单词结尾
    }

    public boolean wordBreak(String s, List<String> wordDict) {
        TrieNode root = new TrieNode();
        int maxLen = 0;

        // 构建前缀树
        for (String word : wordDict) {
            insert(root, word);
            maxLen = Math.max(maxLen, word.length());
        }

        int n = s.length();
        boolean[] dp = new boolean[n + 1];
        dp[0] = true; // 空前缀可达

        for (int i = 0; i < n; i++) {
            // 如果当前位置本身不可达，就没必要从这里往后匹配
            if (!dp[i]) {
                continue;
            }

            TrieNode node = root;

            // 从 i 开始往后尝试匹配字典中的单词
            for (int j = i; j < n && j - i + 1 <= maxLen; j++) {
                char c = s.charAt(j);
                int idx = c - 'a';

                // 当前路径不存在，说明从 i 出发不可能再匹配出单词
                if (node.children[idx] == null) {
                    break;
                }

                node = node.children[idx];

                // 如果走到一个单词结尾，说明 s[i..j] 是合法单词
                if (node.isWord) {
                    dp[j + 1] = true; // 标记新的可达位置
                }
            }

            // 已经能到终点，可以提前结束
            if (dp[n]) {
                return true;
            }
        }

        return dp[n];
    }

    private void insert(TrieNode root, String word) {
        TrieNode node = root;

        for (int i = 0; i < word.length(); i++) {
            int idx = word.charAt(i) - 'a';

            // 如果当前字符路径不存在，则创建新节点
            if (node.children[idx] == null) {
                node.children[idx] = new TrieNode();
            }

            node = node.children[idx];
        }

        node.isWord = true; // 单词插入结束，标记结尾
    }
}
```

## 三种解法如何选择

### 1\. 记忆化搜索

**适合：**

-   想从递归直觉推导
    
-   面试时先快速给出可行思路
    
-   讲解“从暴力递归到优化”的过程
    

**不太适合：**

-   非常强调性能的场景
    
-   代码风格要求更迭代化的场景
    

---

### 2\. 动态规划 + HashSet

**适合：**

-   面试主力写法
    
-   LeetCode 最稳妥提交
    
-   逻辑清晰、容易证明正确性
    

**推荐程度：最高**

如果面试只让你写一个版本，优先写这个。

---

### 3\. Trie + DP

**适合：**

-   面试官追问优化
    
-   字典很大
    
-   多轮重复查询
    
-   工程实践需要减少字符串对象创建开销
    

**推荐程度：高阶优化版**

---

## 正确性理解

以 `s = "leetcode"`，`wordDict = ["leet", "code"]` 为例：

### DP 视角

-   `dp[0] = true`
    
-   检查前 4 个字符 `"leet"` 在字典中，所以 `dp[4] = true`
    
-   再从 `dp[4]` 出发，检查 `"code"` 在字典中，所以 `dp[8] = true`
    
-   最终 `dp[n] = dp[8] = true`
    

这就说明整个字符串能被完整切分。

---

## 易错点总结

### 1\. `dp[i]` 的含义搞错

这题最容易把状态定义写乱。

建议固定成：

-   `dp[i]`：前 `i` 个字符能否拆分
    

这样：

-   `dp[0] = true`
    
-   最终返回 `dp[n]`
    

这是最自然、最不容易出错的写法。

---

### 2\. `substring` 的边界写错

Java 中：

```java
s.substring(l, r)
```

表示：

-   左闭右开 `[l, r)`
    

所以：

```java
s.substring(start, i)
```

实际表示的是：

-   `s[start..i-1]`
    

---

### 3\. 忘记 `dp[0] = true`

没有这个初始条件，后面的状态都推不出来。

---

### 4\. 把字典放在 `List` 里直接查

如果你每次都在 `List` 中做 `contains`，复杂度会上升很多。

应优先转成：

```java
Set<String> wordSet = new HashSet<>(wordDict);
```

---

### 5\. Java 中 `substring` 的性能问题

面试时很多人只说算法复杂度，却忽略语言细节。

在 Java 中，高频 `substring` 可能带来额外对象创建和复制开销。<br>因此：

-   普通面试：DP + HashSet 足够
    
-   性能敏感：考虑 `startsWith`、Trie，甚至进一步做自动机优化
    

这是一个很加分的细节。

## 面试场景建议

### 标准回答顺序

如果在面试中被问到这题，建议这样组织答案：

#### 第一步：先讲状态定义

直接说：

-   `dp[i]` 表示前 `i` 个字符能否由字典中的单词拼成
    

#### 第二步：给出转移

如果存在 `j < i`，使得：

-   `dp[j] == true`
    
-   `s[j..i-1]` 在字典中
    

则：

-   `dp[i] = true`
    

#### 第三步：说优化点

-   用 `HashSet` 优化查词
    
-   用 `minLen / maxLen` 做长度剪枝
    
-   如果继续优化，可以用 Trie 减少子串构造开销
    

这套回答路线非常完整，能体现：

-   你会建模
    
-   你会写标准解
    
-   你还能做性能优化
    

---

### 面试中最推荐写哪一个

首选写：

-   **动态规划 + HashSet + 长度剪枝**
    

原因：

1.  足够标准
    
2.  不容易写错
    
3.  面试官最容易接受
    
4.  容易在白板上解释正确性
    

---

### 如果面试官追问“还能优化吗”

你可以继续补充：

1.  **字典固定、查询很多次**：预建 Trie
    
2.  **进一步做字符串匹配优化**：Trie / 自动机
    
3.  **减少无意义枚举**：记录最短和最长单词长度
    
4.  **减少对象创建**：尽量避免高频 `substring`
    

这类追问回答非常加分。

## 实际工作场景建议

这题在业务中经常会映射为以下场景：

-   文本分词
    
-   命令词识别
    
-   敏感词过滤
    
-   拼写切分
    
-   词典匹配
    
-   搜索召回前的 query 预处理
    

### 如果是一次性小规模判断

直接用：

-   **DP + HashSet**
    

最简单直接。

---

### 如果是固定字典、频繁查询

比如：

-   词典加载一次
    
-   每秒很多字符串要判断
    

此时建议：

-   预构建 **Trie**
    
-   多次复用
    

这样整体吞吐量更高。

---

### 如果不只是判断 true/false，而是要输出方案

比如要输出：

-   一种拆分结果
    
-   所有拆分结果
    
-   拆分路径数
    

那么就需要在这题基础上继续扩展：

-   记录路径前驱
    
-   回溯恢复方案
    
-   或做 `Word Break II` 的 DFS + 记忆化
    

也就是说，**139 是“可不可拆”版本，140 是“输出所有拆法”版本**。<br>面试中如果你能顺带提到这一点，会显得对题族理解比较完整。

---

## 总结

这题最核心的一句话是：

> 只要某个前缀已经可以拆分，并且后面接上的一段也是字典单词，那么更长的前缀也可以拆分。

按照解题优先级：

1.  **记忆化搜索**：帮助建立递归直觉
    
2.  **动态规划 + HashSet + 长度剪枝**：最标准、最推荐
    
3.  **Trie + DP**：更偏性能优化和工程实践
    

如果你只准备一个最稳的版本，优先掌握第二种。<br>如果你想把这题吃透，并应对追问，把第三种也理解掉。
