# 力扣 287 题：寻找重复数

## 题目要点

给定一个长度为 `n + 1` 的数组 `nums`，其中每个数都在 `[1, n]` 范围内。<br>已知只有 **一个重复的整数**，但它可能 **重复出现两次或多次**，要求找出这个重复数。

这题最关键的点有 4 个：

1.  **数组长度是** `n + 1`**，但取值只有** `1 ~ n`
    
    -   这意味着根据 **抽屉原理**，一定存在重复数。
        
2.  **重复的值只有一个**
    
    -   不是说只重复两次，而是“只有一个值重复”，这个值可能出现很多次。
        
3.  **进阶要求**
    
    -   **不能修改原数组**
        
    -   **只使用 O(1) 额外空间**
        
4.  这题最经典的最优解不是排序，也不是哈希，而是：
    
    -   **值域二分**
        
    -   **Floyd 判圈（快慢指针）**
        

---

## 解法对比

| 解法  | 时间复杂度 | 额外空间 | 是否修改数组 | 是否满足进阶要求 | 适用场景 |
| --- | --- | --- | --- | --- | --- |
| 排序法 | `O(n log n)` | 实现相关，通常 `O(log n)` | 是   | 否   | 快速想到、约束不严格 |
| 哈希集合 | `O(n)` | `O(n)` | 否   | 否   | 工程里最容易写对 |
| 值域二分 | `O(n log n)` | `O(1)` | 否   | 是   | 进阶可过、证明相对容易 |
| Floyd 判圈 | `O(n)` | `O(1)` | 否   | 是   | 面试最优解，必须掌握 |

---

## 解法一：排序法

### 思路

如果把数组排好序，那么重复元素一定会出现在相邻位置。

例如：

```text
[1, 3, 4, 2, 2] -> 排序后 [1, 2, 2, 3, 4]
```

只要遍历一遍，找到相邻相等的元素即可。

### 适用场景

-   你只是想先快速做出来
    
-   不在意是否修改数组
    
-   不追求满足题目进阶限制
    

### 缺点

-   会修改原数组
    
-   时间复杂度不是最优
    
-   不符合这题进阶要求
    

### 复杂度

-   时间复杂度：`O(n log n)`
    
-   空间复杂度：取决于排序实现，通常可认为是实现相关；但**核心问题是它修改了原数组**
    

### Java 代码

```java
import java.util.Arrays;

class Solution {
    public int findDuplicate(int[] nums) {
        Arrays.sort(nums); // 排序后，重复元素一定相邻

        for (int i = 1; i < nums.length; i++) {
            if (nums[i] == nums[i - 1]) {
                return nums[i]; // 找到相邻相等的元素
            }
        }

        return -1; // 理论上不会走到这里
    }
}
```

### 细节技巧

-   这题在面试里，如果你先写排序法，最好主动补一句：
    
    -   “这个方法能做出来，但会修改数组，不满足进阶要求。”
        
-   这样能体现你知道题目的重点不在这里。

## 解法二：哈希集合

### 思路

遍历数组，用 `HashSet` 记录已经见过的数字：

-   如果当前数字第一次出现，就放进集合
    
-   如果当前数字已经存在，说明它就是重复数
    

### 适用场景

-   实际工作里，如果没有严格空间限制，这是最稳妥、最清晰的写法
    
-   面试里可以作为第一版 baseline 解法，先保证正确
    

### 复杂度

-   时间复杂度：`O(n)`
    
-   空间复杂度：`O(n)`
    

### Java 代码

```java
import java.util.HashSet;
import java.util.Set;

class Solution {
    public int findDuplicate(int[] nums) {
        Set<Integer> seen = new HashSet<>();

        for (int num : nums) {
            if (!seen.add(num)) { // add 返回 false，说明这个数已经存在
                return num;
            }
        }

        return -1; // 理论上不会走到这里
    }
}
```

### 细节技巧

-   因为题目保证只有一个重复值，所以一旦发现重复，直接返回即可
    
-   如果你想进一步利用题目“取值范围在 `[1, n]`”这个条件，也可以用 `boolean[] visited`，本质还是 `O(n)` 额外空间
    

### 面试表达建议

这一步适合这样说：

> “先给一个最直接的 O(n) 时间解法，用 HashSet 判重。<br>但它用了 O(n) 额外空间，不满足进阶条件，所以我再继续优化。”

这种表达很加分，因为你不是一上来就硬背最优解，而是有完整优化过程。

## 解法三：值域二分

### 核心思想

这题的二分，不是对数组下标二分，而是对 **数值范围** `[1, n]` 二分。

设 `mid` 为中间值，统计数组中有多少个数 `<= mid`。

-   如果 `count > mid`
    
    -   说明区间 `[1, mid]` 中最多只能容纳 `mid` 个不同的数字
        
    -   但现在居然有 `count` 个数落进来了，而且 `count > mid`
        
    -   根据抽屉原理，重复数一定在 `[1, mid]`
        
-   否则
    
    -   重复数一定在 `[mid + 1, n]`
        

### 为什么这招可行

因为 `[1, mid]` 这个区间本来只有 `mid` 个不同取值位置：

```text
1, 2, 3, ..., mid
```

如果数组中小于等于 `mid` 的数出现了超过 `mid` 次，那这些数里必然有重复。

注意这里的判断条件是：

```java
count > mid
```

不是 `count >= mid`。

### 示例推演

以 `nums = [1, 3, 4, 2, 2]` 为例：

#### 第 1 轮

-   `left = 1, right = 4`
    
-   `mid = 2`
    
-   `<= 2` 的数有：`1, 2, 2`
    
-   `count = 3`
    

因为 `3 > 2`，说明重复数在 `[1, 2]`

#### 第 2 轮

-   `left = 1, right = 2`
    
-   `mid = 1`
    
-   `<= 1` 的数有：`1`
    
-   `count = 1`
    

因为 `1 <= 1`，说明重复数不在 `[1, 1]`，而在 `[2, 2]`

最后答案是 `2`

### 适用场景

-   不能修改数组
    
-   只能用 O(1) 空间
    
-   你想给出一个比 Floyd 更容易解释和证明的解法
    

### 复杂度

-   时间复杂度：`O(n log n)`
    
    -   二分值域要做 `log n` 次
        
    -   每次需要遍历整个数组统计数量，代价是 `O(n)`
        
-   空间复杂度：`O(1)`
    

### Java 代码

```java
class Solution {
    public int findDuplicate(int[] nums) {
        int left = 1;
        int right = nums.length - 1; // 数值范围是 [1, n]

        while (left < right) {
            int mid = left + (right - left) / 2;
            int count = 0;

            for (int num : nums) {
                if (num <= mid) {
                    count++; // 统计有多少个数落在 [1, mid]
                }
            }

            if (count > mid) {
                right = mid; // 左侧“超载”，重复数一定在左边
            } else {
                left = mid + 1; // 左侧数量正常，重复数在右边
            }
        }

        return left; // left == right，即重复数
    }
}
```

### 细节技巧

1.  **二分的是值，不是索引**
    
    -   这是这题最容易写错的地方
        
2.  **判断是** `count > mid`
    
    -   不是 `>=`
        
    -   因为 `[1, mid]` 最多正好容纳 `mid` 个不同数字
        
3.  **边界从** `1` **到** `n`
    
    -   不是从 `0` 开始
        
    -   因为题目保证所有数都在 `[1, n]`
        

### 常见错误

-   把这题误写成“先排序再二分”
    
-   写成统计 `[left, mid]` 区间元素个数，导致逻辑混乱
    
-   使用 `count >= mid`，这会把边界判断写错

## 解法四：Floyd 判圈（快慢指针，最优解）

这是这题最经典、最值得掌握的解法。

---

### 第一步：把数组看成链表

定义映射：

```text
i -> nums[i]
```

因为 `nums[i]` 的取值范围一定在 `[1, n]`，所以从任意下标出发，下一步总能跳到一个合法位置。

例如数组：

```text
nums = [1, 3, 4, 2, 2]
```

可以看成：

```text
0 -> 1
1 -> 3
3 -> 2
2 -> 4
4 -> 2
```

连起来就是：

```text
0 -> 1 -> 3 -> 2 -> 4 -> 2 -> 4 -> 2 ...
```

很明显，出现了环，环入口是 `2`，而 `2` 正好就是重复数。

---

### 为什么一定有环

因为：

-   下标有 `0 ~ n`，一共 `n + 1` 个位置
    
-   但数值只能落在 `1 ~ n`
    
-   从 `0` 出发不断根据 `nums[i]` 跳转，最终一定会重复访问某个位置
    
-   一旦重复访问，就形成环
    

更关键的一点：

-   `0` 不可能被任何节点指向，因为数组里的值不会是 `0`
    
-   所以从 `0` 出发，会先走一段“链”，然后进入环
    

---

### 为什么“环入口”就是“重复数”

这是这题最核心的理解。

如果某个数 `d` 重复了，那么就意味着至少有两个不同下标指向它：

```text
nums[a] = d
nums[b] = d
```

也就是说，在这张“函数图”里，节点 `d` 的入度至少为 2。

而从 `0` 出发走向环时，**第一个进入环的节点**，也就是环入口，恰好就是那个“被多个位置指向”的节点。<br>这个节点对应的值，就是重复数。

可以把它理解为：

-   普通节点只有一条路走过来
    
-   环入口既可以从环外走进来，也可以从环内绕回来
    
-   所以它就是那个“重复汇入”的位置
    

---

### Floyd 判圈分两步

#### 阶段 1：找到环中的相遇点

-   慢指针每次走 1 步
    
-   快指针每次走 2 步
    
-   如果有环，它们一定会在环内相遇
    

#### 阶段 2：找到环入口

-   让一个指针回到起点 `0`
    
-   另一个指针留在相遇点
    
-   两个指针都每次走 1 步
    
-   它们再次相遇的位置，就是环入口，也就是重复数
    

---

### 为什么第二次相遇就是环入口

设：

-   从起点 `0` 到环入口的距离为 `a`
    
-   从环入口到第一次相遇点的距离为 `b`
    
-   环长为 `c`
    

第一次相遇时：

-   慢指针走了：`a + b`
    
-   快指针走了：`a + b + k * c`
    

又因为快指针速度是慢指针 2 倍，所以：

```text
2(a + b) = a + b + k * c
```

整理得：

```text
a = k * c - b
```

这说明：

-   从相遇点再走 `c - b` 步可以到环入口
    
-   从起点走 `a` 步也能到环入口
    
-   且 `a` 与“从相遇点继续前进到入口”的步数在环意义下是等价的
    

所以，一个指针从 `0` 开始走，另一个指针从相遇点开始走，它们都会在环入口相遇。

---

### 适用场景

-   面试最优答案
    
-   需要满足：
    
    -   不修改数组
        
    -   O(1) 额外空间
        
    -   O(n) 时间
        

### 复杂度

-   时间复杂度：`O(n)`
    
-   空间复杂度：`O(1)`
    

### Java 代码

```java
class Solution {
    public int findDuplicate(int[] nums) {
        int slow = 0;
        int fast = 0;

        // 第一阶段：找到环中的相遇点
        do {
            slow = nums[slow]; // 慢指针每次走一步
            fast = nums[nums[fast]]; // 快指针每次走两步
        } while (slow != fast);

        // 第二阶段：从起点和相遇点同时出发，寻找环入口
        int finder = 0;
        while (finder != slow) {
            finder = nums[finder]; // 从起点出发的指针每次走一步
            slow = nums[slow]; // 相遇点指针也每次走一步
        }

        return finder; // 环入口就是重复数
    }
}
```

### 细节技巧

1.  **推荐用** `do...while`
    
    -   因为一开始 `slow = fast = 0`
        
    -   如果直接 `while (slow != fast)`，循环体根本不会执行
        
2.  **第二阶段重新出发的是** `0`
    
    -   不是 `1`
        
    -   因为我们整个“链表”的起点是下标 `0`
        
3.  **返回的是** `finder` **或** `slow`
    
    -   不是 `nums[finder]`
        
    -   因为“节点编号本身”就是那个重复值
        
4.  **这题不是链表题，但本质是函数图判环**
    
    -   面试里如果能说出“把数组映射成函数图”，会显得很扎实

## 常见错误汇总

### 1\. 把值域二分理解成下标二分

这题二分的是：

```text
重复数可能落在哪个值域区间
```

不是数组位置。

---

### 2\. Floyd 第一阶段写错移动方式

正确写法：

```java
slow = nums[slow];
fast = nums[nums[fast]];
```

不是：

```java
fast = nums[fast];
```

否则快慢指针速度一样，失去判圈意义。

---

### 3\. Floyd 第二阶段返回错了

正确返回：

```java
return finder;
```

或 `return slow;`

错误写法：

```java
return nums[finder];
```

因为环入口节点本身就是重复数。

---

### 4\. 使用原地交换、符号标记等技巧

这些做法在很多数组题里很常见，但这题的进阶要求是：

-   **不能修改原数组**
    

所以这类方法即使能做出来，也不符合题意重点。

---

## 面试场景建议

### 建议回答顺序

如果是面试，推荐你按这个顺序讲：

1.  **先给 HashSet**
    
    -   `O(n)` 时间，`O(n)` 空间
        
    -   最容易想到，也最容易写对
        
2.  **然后指出不满足进阶要求**
    
    -   不能做到 `O(1)` 额外空间
        
3.  **再给值域二分**
    
    -   满足“不修改数组 + O(1) 空间”
        
    -   但时间是 `O(n log n)`
        
4.  **最后给 Floyd 判圈**
    
    -   `O(n)` 时间
        
    -   `O(1)` 空间
        
    -   最优解
        

这样的表达会显得你不是在“背答案”，而是在逐步优化。

## 实际工作场景建议

### 1\. 如果没有严格空间限制，优先写 HashSet

原因很现实：

-   可读性最好
    
-   最容易维护
    
-   最不容易出错
    
-   同事一眼就能看懂
    

工程里很多时候，**可维护性比极限优化更重要**。

---

### 2\. 如果输入数组不能改，且内存敏感，再选 Floyd

例如：

-   大数据量场景
    
-   输入数组来自共享缓存，不允许修改
    
-   面试题明确要求 `O(1)` 额外空间
    

这时 Floyd 是最佳选择。

---

### 3\. 如果面试官更关注“证明过程”，值域二分很稳

Floyd 的思维跳跃稍大，有些面试官会追问：

-   为什么能成环？
    
-   为什么入口就是重复值？
    
-   为什么第二次相遇是入口？
    

如果你对这些证明不够熟，值域二分反而更好讲清楚。

---

## 一句话总结

这题真正要掌握的不是“找重复数”本身，而是三种能力：

1.  **抽屉原理**
    
2.  **二分答案（值域二分）**
    
3.  **把数组映射成函数图，用 Floyd 判圈找环入口**
    

如果只记一个最优模板，就记 **Floyd 判圈**；<br>如果要兼顾“容易证明”，就把 **值域二分** 也一起掌握。
