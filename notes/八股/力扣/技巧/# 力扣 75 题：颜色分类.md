# 力扣 75 题：颜色分类

## 题目理解

给定一个只包含 `0`、`1`、`2` 的数组 `nums`，要求**原地**将它们排序，使得相同颜色相邻，并按照 `0 -> 1 -> 2` 的顺序排列。

这道题的本质不是“排序”本身，而是利用：

-   元素种类非常少，只有 3 种
    
-   要求原地修改
    
-   进阶要求通常是：**一趟遍历 + 常数空间**
    

这题是非常经典的：

-   入门思路：**计数排序**
    
-   最优思路：**荷兰国旗问题 / 三路划分**
    

---

## 解法一：计数排序（两次遍历，思路最稳）

### 适用场景

-   面试时先快速给出正确解
    
-   数据值域很小、种类固定
    
-   不强求“一趟遍历”
    

### 核心思路

因为数组里只有 `0`、`1`、`2` 三种数字，所以根本没必要做通用排序。

先统计：

-   `0` 出现了多少次
    
-   `1` 出现了多少次
    
-   `2` 出现了多少次
    

然后再按次数回填数组：

-   前面填若干个 `0`
    
-   中间填若干个 `1`
    
-   后面填若干个 `2`
    

### 算法步骤

1.  创建长度为 3 的计数数组 `count`
    
2.  遍历 `nums`，统计每个数字出现次数
    
3.  再次遍历 `count`，按顺序把 `0/1/2` 回写到 `nums`
    

### 复杂度分析

-   时间复杂度：`O(n)`
    
-   空间复杂度：`O(1)`
    

这里空间虽然用了 `count[3]`，但它是固定常数级，不随 `n` 增长。

### Java 代码

```java
class Solution {
    public void sortColors(int[] nums) {
        int[] count = new int[3];

        // 第一次遍历：统计 0、1、2 的出现次数
        for (int num : nums) {
            count[num]++;
        }

        int index = 0;

        // 第二次遍历：按 0、1、2 的顺序回填数组
        for (int color = 0; color < 3; color++) {
            while (count[color] > 0) {
                nums[index++] = color; // 将当前颜色写回原数组
                count[color]--;
            }
        }
    }
}
```

### 这一解法的优缺点

#### 优点

-   非常容易想到
    
-   代码短，几乎不容易写错
    
-   面试中可以作为“第一版正确答案”
    

#### 缺点

-   不是一趟遍历
    
-   没有充分利用“原地三路划分”的技巧
    
-   如果面试官追问更优解，通常还要继续优化

## 解法二：双指针三路划分 / 荷兰国旗（一次遍历，最优）

### 适用场景

-   面试官要求：
    
    -   原地
        
    -   常数空间
        
    -   一趟扫描
        
-   需要掌握经典 partition 思想
    
-   想真正理解这道题的核心技巧
    

### 核心思路

我们不再“统计后回填”，而是在遍历过程中直接把元素放到正确区域。

维护三个区域：

-   左边区域：全是 `0`
    
-   中间区域：全是 `1`
    
-   右边区域：全是 `2`
    

再加一个当前扫描指针 `i`，不断处理当前位置的元素。

### 指针定义

-   `left`：下一个应该放 `0` 的位置
    
-   `right`：下一个应该放 `2` 的位置
    
-   `i`：当前扫描位置
    

### 维护的不变量

在任意时刻，数组都满足：

-   `[0, left - 1]` 全是 `0`
    
-   `[left, i - 1]` 全是 `1`
    
-   `[i, right]` 是尚未处理的区域
    
-   `[right + 1, n - 1]` 全是 `2`
    

这个不变量是整道题的灵魂。

---

## 为什么这样做是对的

当扫描到 `nums[i]` 时：

### 情况 1：`nums[i] == 0`

说明它应该被放到左边区域，于是：

-   交换 `nums[i]` 和 `nums[left]`
    
-   `left++`
    
-   `i++`
    

为什么这里可以直接 `i++`？

因为交换前：

-   `[left, i - 1]` 区域全是 `1`
    

所以交换到 `i` 位置上的元素，要么是：

-   `1`
    
-   或者当 `left == i` 时，其实是自己和自己交换
    

无论哪种情况，都不需要重新检查，所以可以直接 `i++`。

---

### 情况 2：`nums[i] == 1`

它本来就应该在中间区域，不需要交换：

-   `i++`
    

---

### 情况 3：`nums[i] == 2`

说明它应该被放到右边区域，于是：

-   交换 `nums[i]` 和 `nums[right]`
    
-   `right--`
    

**注意：这里不能** `i++`

因为从 `right` 交换过来的那个元素还没有检查过，它可能是 `0`、`1`、`2` 中任意一个，必须留在当前位置继续判断。

这是这道题最容易写错的地方。

---

## 动态推演一遍

以数组：

```text
[2, 0, 2, 1, 1, 0]
```

为例。

初始：

```text
left = 0, i = 0, right = 5
```

### 第 1 步

`nums[i] = 2`

和 `right` 交换：

```text
[0, 0, 2, 1, 1, 2]
right = 4, i = 0
```

这里 `i` 不动，因为新换过来的 `0` 还要继续判断。

---

### 第 2 步

`nums[i] = 0`

和 `left` 交换：

```text
[0, 0, 2, 1, 1, 2]
left = 1, i = 1
```

---

### 第 3 步

`nums[i] = 0`

继续和 `left` 交换：

```text
[0, 0, 2, 1, 1, 2]
left = 2, i = 2
```

---

### 第 4 步

`nums[i] = 2`

和 `right` 交换：

```text
[0, 0, 1, 1, 2, 2]
right = 3, i = 2
```

---

### 第 5 步

`nums[i] = 1`

```text
i = 3
```

---

### 第 6 步

`nums[i] = 1`

```text
i = 4
```

此时 `i > right`，遍历结束。

最终数组：

```text
[0, 0, 1, 1, 2, 2]
```

---

## Java 代码

```java
class Solution {
    public void sortColors(int[] nums) {
        int left = 0;                 // [0, left - 1] 都是 0
        int i = 0;                    // 当前扫描位置
        int right = nums.length - 1;  // [right + 1, n - 1] 都是 2

        while (i <= right) {
            if (nums[i] == 0) {
                swap(nums, left, i);  // 把 0 放到左边区域
                left++;
                i++;
            } else if (nums[i] == 1) {
                i++;                  // 1 留在中间区域，直接跳过
            } else { // nums[i] == 2
                swap(nums, i, right); // 把 2 放到右边区域
                right--;
                // 这里不能 i++，因为交换过来的元素还没有检查
            }
        }
    }

    private void swap(int[] nums, int i, int j) {
        int temp = nums[i];
        nums[i] = nums[j];
        nums[j] = temp;
    }
}
```

---

## 复杂度分析

-   时间复杂度：`O(n)`
    
-   空间复杂度：`O(1)`
    

虽然有交换操作，但每个元素最多被处理有限次，总体仍然是线性时间。

---

## 两种解法对比

| 解法  | 时间复杂度 | 空间复杂度 | 是否一趟遍历 | 难度  | 适用场景 |
| --- | --- | --- | --- | --- | --- |
| 计数排序 | `O(n)` | `O(1)` | 否   | 简单  | 快速写出正确解、值域很小 |
| 荷兰国旗 | `O(n)` | `O(1)` | 是   | 中等  | 面试最优解、考察双指针与 partition |

## 易错点总结

### 1\. `nums[i] == 2` 时，交换后不能 `i++`

错误写法会漏掉新交换过来的元素。

错误原因：

-   `right` 位置的元素之前没有处理过
    
-   它换到 `i` 后必须重新判断
    

这是最常见的 bug。

---

### 2\. 循环条件要写成 `i <= right`

不是 `i < right`。

因为当 `i == right` 时，这个位置仍然是未处理区域，不能漏掉。

---

### 3\. `nums[i] == 0` 时，交换后可以 `i++`

很多人会和 `2` 的情况混淆。

这里之所以可以前进，是因为从 `left` 换过来的元素一定已经是处理过的中间区域元素，通常为 `1`，不会有问题。

---

### 4\. 自己和自己交换是允许的

例如：

-   `left == i`
    
-   或者 `i == right`
    

这种交换不会出错，不需要额外特判，代码反而更简洁。

## 面试中的推荐回答顺序

这题在面试里非常适合分层回答。

### 第一层：先给出计数排序

先快速说明：

-   因为只有 0、1、2 三种值
    
-   可以统计次数后回填
    
-   时间 `O(n)`，空间 `O(1)`
    

这样能先证明你有稳定、正确的解题能力。

---

### 第二层：主动升级到荷兰国旗

然后补一句：

-   如果要求一趟遍历
    
-   或者更强调“原地划分”
    
-   可以使用三指针做荷兰国旗问题
    

这样会显得你不是“只会 AC”，而是理解题目的最优结构。

---

### 第三层：讲清楚不变量

如果能把这个不变量说出来，面试印象会很好：

-   左边全是 0
    
-   中间全是 1
    
-   右边全是 2
    
-   当前指针负责压缩未知区间
    

这比只会背代码更重要。

## 实际工作中的建议

### 1\. 工程里优先考虑可读性与稳定性

如果这是业务代码，并不是算法面试：

-   数据规模不大
    
-   性能不是瓶颈
    
-   逻辑可读性更重要
    

那么计数排序往往更稳、更容易维护，出 bug 的概率更低。

---

### 2\. 荷兰国旗思想非常像实际中的“分类分区”

这类思想在工作中经常以变形形式出现，例如：

-   将状态分成三类，原地重排
    
-   日志/任务按优先级做原地分区
    
-   类似快速排序中的 partition 过程
    
-   按阈值将数据分成“小于 / 等于 / 大于”三段
    

所以这题不是单独一题，而是很多“原地分类”问题的模板。

---

### 3\. 如果问题扩展为 `k` 种颜色怎么办

这时就要分情况：

-   `k` 很小、取值范围固定：**计数排序**更合适
    
-   只有三类并要求原地一次遍历：**荷兰国旗**
    
-   `k` 较大或是普通排序需求：考虑通用排序算法
    

也就是说，75 题的关键不是机械记忆，而是学会根据“值域大小”和“空间/遍历约束”来选策略。

---

## 最终结论

这题最推荐掌握两个层次：

1.  **计数排序**
    
    -   简单直接
        
    -   容易写对
        
    -   适合快速得到正确答案
        
2.  **荷兰国旗**
    
    -   一趟遍历
        
    -   常数空间
        
    -   面试最优解
        
    -   本质是经典三路划分思想
        

如果只准备一个最强版本，就记住荷兰国旗这套写法，尤其要记住一句：

> 当 `nums[i] == 2` 时，和 `right` 交换后，`i` 不能前进。
