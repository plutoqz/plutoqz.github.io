# 230\. 二叉搜索树中第 K 小的元素

**题意**：给定一棵二叉搜索树 `root` 和整数 `k`，返回这棵树中第 `k` 小的元素。

---

## 核心性质

二叉搜索树（BST）的中序遍历结果是**严格递增**的。

所以这道题本质上就是：

-   找到 BST 中序遍历序列中的第 `k` 个数。
    

---

## 复杂度对比

| 解法  | 时间复杂度 | 空间复杂度 | 适用场景 |
| --- | --- | --- | --- |
| 解法一：完整中序收集 | `O(n)` | `O(n)` | 最容易写，最容易验证 |
| 解法二：迭代中序 + 提前停止 | `O(h + k)`，最坏 `O(n)` | `O(h)` | **单次查询最推荐，面试首选** |
| 解法三：Morris 中序 | `O(n)` | `O(1)` | 追求额外空间最优 |
| 解法四：子树节点数统计 | 预处理 `O(n)`，单次查询 `O(h)` | `O(n)` | **同一棵树要频繁查询第 k 小** |

> `n` 是节点总数，`h` 是树高。

---

# 解法一：完整中序遍历后取第 k 个

## 思路

既然中序遍历结果有序，那就：

1.  中序遍历整棵树
    
2.  把所有节点值放进 `List`
    
3.  返回 `list.get(k - 1)`
    

这是最直观的写法，但不是最省空间的。

## 适用场景

-   刚开始做题
    
-   想先写一个最稳妥、最不容易错的版本
    
-   面试时可以先快速给出，再优化到解法二
    

## 代码

```java
import java.util.ArrayList;
import java.util.List;

class Solution {
    public int kthSmallest(TreeNode root, int k) {
        List<Integer> list = new ArrayList<>();
        inorder(root, list);
        return list.get(k - 1); // 第 k 小，对应下标 k - 1
    }

    private void inorder(TreeNode node, List<Integer> list) {
        if (node == null) {
            return;
        }

        inorder(node.left, list);  // 先遍历左子树
        list.add(node.val);        // 再访问当前节点
        inorder(node.right, list); // 最后遍历右子树
    }
}
```

## 复杂度

-   时间复杂度：`O(n)`
    
-   空间复杂度：`O(n)`<br>`List` 存了所有节点，递归还会有栈空间

# 解法二：迭代中序遍历 + 提前停止

## 思路

既然中序遍历是从小到大，那其实没必要把所有节点都遍历完。

每访问到一个节点，就说明拿到了当前第几个最小值：

-   访问 1 次：第 1 小
    
-   访问 2 次：第 2 小
    
-   ...
    
-   访问到第 `k` 次时，直接返回
    

为了避免递归，用栈来模拟中序遍历：

1.  一路往左走，把路径节点都压栈
    
2.  弹出栈顶，说明当前是下一个最小值
    
3.  `k--`
    
4.  当 `k == 0` 时返回
    
5.  然后转向当前节点的右子树继续
    

## 为什么这是面试首选

-   比“全部存下来”更优
    
-   比递归更稳，避免极深树导致栈溢出
    
-   写法标准、可控、容易讲清楚
    

## 适用场景

-   **单次查询最优解**
    
-   面试默认优先写这个版本
    

## 代码

```java
import java.util.ArrayDeque;
import java.util.Deque;

class Solution {
    public int kthSmallest(TreeNode root, int k) {
        Deque<TreeNode> stack = new ArrayDeque<>();
        TreeNode cur = root;

        while (cur != null || !stack.isEmpty()) {
            // 不断向左走，把路径上的节点压栈
            while (cur != null) {
                stack.push(cur);
                cur = cur.left;
            }

            // 栈顶就是当前还未访问的最小节点
            cur = stack.pop();
            k--; // 访问到一个更小的元素

            if (k == 0) {
                return cur.val; // 找到第 k 小
            }

            // 转向右子树，继续中序遍历
            cur = cur.right;
        }

        return -1; // 题目保证 k 合法，理论上不会走到这里
    }
}
```

## 复杂度

-   时间复杂度：`O(h + k)`，最坏 `O(n)`
    
-   空间复杂度：`O(h)`
    

## 细节技巧

-   Java 里栈建议用 `Deque`，不要用老旧的 `Stack`
    
-   这段代码本质是在“按从小到大”一个个取数
    
-   `k` 是从 1 开始计数，不要写成 `k - 1 == 0` 之类绕的逻辑

# 解法三：Morris 中序遍历（额外空间 O(1)）

## 思路

如果面试官继续追问：

> 能不能不用递归，也不用显式栈？

就可以用 **Morris Traversal**。

它的核心是：

-   利用树中原本为空的右指针，临时建立“线索”
    
-   遍历完左子树后，再通过这条线索回到当前节点
    
-   遍历结束时再把树结构恢复
    

这样就不需要栈了。

## 关键点

对于当前节点 `cur`：

-   如果 `cur.left == null`
    
    -   直接访问 `cur`
        
    -   然后去 `cur.right`
        
-   如果 `cur.left != null`
    
    -   找到左子树中最右侧节点 `predecessor`
        
    -   如果 `predecessor.right == null`
        
        -   建立线索：`predecessor.right = cur`
            
        -   然后 `cur = cur.left`
            
    -   否则说明左子树已经处理完
        
        -   恢复：`predecessor.right = null`
            
        -   访问 `cur`
            
        -   然后 `cur = cur.right`
            

## 适用场景

-   对空间要求极高
    
-   面试官追问 `O(1)` 额外空间
    
-   你对树的线索化遍历比较熟
    

## 代码

```java
class Solution {
    public int kthSmallest(TreeNode root, int k) {
        TreeNode cur = root;
        int count = 0;
        int ans = -1;

        while (cur != null) {
            if (cur.left == null) {
                // 没有左子树，当前节点可以直接访问
                count++;
                if (count == k) {
                    ans = cur.val; // 记录答案
                }
                cur = cur.right;
            } else {
                // 找到当前节点左子树中的最右节点（中序前驱）
                TreeNode predecessor = cur.left;
                while (predecessor.right != null && predecessor.right != cur) {
                    predecessor = predecessor.right;
                }

                if (predecessor.right == null) {
                    // 第一次到达当前节点，建立线索，继续走左边
                    predecessor.right = cur;
                    cur = cur.left;
                } else {
                    // 第二次回到当前节点，说明左子树已遍历完，先恢复树结构
                    predecessor.right = null;

                    count++;
                    if (count == k) {
                        ans = cur.val; // 记录答案
                    }

                    cur = cur.right;
                }
            }
        }

        return ans;
    }
}
```

## 复杂度

-   时间复杂度：`O(n)`
    
-   空间复杂度：`O(1)`
    

## 易错点

### 1）不要在找到答案后立刻 `return`

Morris 会临时改树的指针。

如果你一找到第 `k` 小就直接返回，可能还有一些线索边没恢复，树结构会被破坏。

上面的写法是：

-   先记录 `ans`
    
-   继续遍历，把树恢复干净
    
-   最后返回
    

这是更稳妥的写法。

# 解法四：子树节点数统计（频繁查询场景最优）

## 思路

这属于题目的 follow-up 方向：

> 如果二叉搜索树经常被修改，并且需要频繁查第 k 小，怎么办？

单次查找靠中序遍历没问题，但如果要查很多次，就会重复做大量无效工作。

更好的办法是：

-   预处理每个节点的子树大小
    
-   查询时看左子树节点数 `leftSize`
    

对于当前节点 `node`：

-   如果 `k <= leftSize`<br>说明第 `k` 小在左子树
    
-   如果 `k == leftSize + 1`<br>说明当前节点就是答案
    
-   如果 `k > leftSize + 1`<br>说明去右子树继续找，新的 `k` 变成<br>`k - leftSize - 1`
    

这就是“按排名查找”。

## 适用场景

-   同一棵树要查很多次第 `k` 小
    
-   面试 follow-up
    
-   实际工程里可以扩展成 **Order Statistic Tree（带 size 的平衡树）**
    

## 代码

```java
import java.util.HashMap;
import java.util.Map;

class Solution {
    private final Map<TreeNode, Integer> sizeMap = new HashMap<>();

    public int kthSmallest(TreeNode root, int k) {
        countNodes(root);       // 先统计每个节点的子树大小
        return select(root, k); // 再按排名查找
    }

    private int countNodes(TreeNode node) {
        if (node == null) {
            return 0;
        }

        int leftSize = countNodes(node.left);   // 统计左子树大小
        int rightSize = countNodes(node.right); // 统计右子树大小
        int totalSize = leftSize + rightSize + 1;

        sizeMap.put(node, totalSize); // 记录当前节点的子树大小
        return totalSize;
    }

    private int select(TreeNode node, int k) {
        int leftSize = (node.left == null) ? 0 : sizeMap.get(node.left);

        if (k <= leftSize) {
            return select(node.left, k); // 第 k 小在左子树
        }

        if (k == leftSize + 1) {
            return node.val; // 当前节点正好是第 k 小
        }

        return select(node.right, k - leftSize - 1); // 去右子树继续找
    }
}
```

## 复杂度

-   预处理：`O(n)`
    
-   单次查询：`O(h)`
    
-   总空间：`O(n)`
    

## 如果树还会频繁修改怎么办

这时不能每次查询前都重新统计一遍。

正确工程思路是：

-   在节点结构里直接维护 `size`
    
-   插入、删除时同步更新 `size`
    
-   用平衡树保证 `h` 不会退化
    

这就是典型的：

-   Treap
    
-   AVL
    
-   Red-Black Tree + size
    
-   Order Statistic Tree

# 面试该怎么答最稳

## 推荐回答顺序

### 第一步：先说核心结论

> BST 的中序遍历结果是递增序列，所以第 k 小元素就是中序遍历中的第 k 个节点。

### 第二步：先给面试官一个稳定解

直接写 **解法二：迭代中序 + 提前停止**

原因：

-   不需要遍历全部节点
    
-   比递归更稳
    
-   代码不长
    
-   不容易出 bug
    

### 第三步：如果被追问，再补充优化

-   如果要求 `O(1)` 额外空间：说 **Morris**
    
-   如果要求频繁查询：说 **维护子树大小**

# 实际工作中的建议

## 1）一次性查询：直接用迭代中序

如果只是查一次第 `k` 小，没必要搞复杂结构，**解法二**就足够了。

## 2）高并发 / 高频排名查询：不要每次现算

如果业务是：

-   排名系统
    
-   有序集合
    
-   频繁求第 k 大 / 第 k 小
    
-   数据还会增删
    

就不应该每次都中序遍历，而应该维护：

-   平衡树 + 子树大小
    
-   或值域可压缩时用树状数组 / 线段树
    

## 3）Java 中避免递归过深

线上数据如果极端退化成链表，递归可能触发 `StackOverflowError`。<br>所以工程代码里，**迭代通常比递归更稳**。

---

# 易错点总结

1.  `k` **是从 1 开始，不是从 0 开始**
    
2.  **中序遍历顺序是：左 -> 根 -> 右**
    
3.  **Java 里优先用** `Deque` **代替** `Stack`
    
4.  **Morris 遍历别忘了恢复树结构**
    
5.  **频繁查询场景下，不要每次都重新中序整棵树**
