# 动态规划

---

# 1\. 先给你一个最核心的理解

**动态规划的本质：**

> **把一个大问题拆成有重叠的子问题，通过记录子问题答案，避免重复计算，从而以更低代价求最优解、计数解、可行性解等。**

它不是一个单一算法，而是一类\*\*“设计问题求解过程的方法论”\*\*。

如果只记一句：

> **DP = 状态定义 + 状态转移 + 初始条件 + 计算顺序**

---

# 2\. 动态规划到底在解决什么问题？

动态规划最适合这类问题：

## 2.1 两个关键特征

### 1）最优子结构（Optimal Substructure）

原问题的最优解，可以由子问题的最优解组合得到。

例如：

-   最短路径
    
-   最长上升子序列
    
-   背包最大价值
    
-   编辑距离
    

### 2）重叠子问题（Overlapping Subproblems）

在递归拆分过程中，会反复遇到同样的子问题。

例如斐波那契：

-   `f(5)` 需要 `f(4)` 和 `f(3)`
    
-   `f(4)` 又需要 `f(3)` 和 `f(2)`
    
-   所以 `f(3)` 被重复算了很多次
    

**如果只有最优子结构，没有重叠子问题，通常更像分治。**<br>**如果有重叠子问题，但没有合适的最优子结构，就不一定能用标准 DP。**

---

# 3\. 从递归到记忆化，再到底向上：DP 的自然来源

理解 DP 最好的方式，不是上来背表格，而是看它如何从暴力递归演化而来。

---

## 3.1 例子：斐波那契数列

定义：

[ f(n)=f(n-1)+f(n-2) ]

### 暴力递归

```python
def fib(n):
    if n <= 1:
        return n
    return fib(n-1) + fib(n-2)
```

问题：

-   重复计算严重
    
-   时间复杂度接近 `O(2^n)`
    

---

## 3.2 加记忆化搜索（Top-down）

```python
def fib(n, memo={}):
    if n in memo:
        return memo[n]
    if n <= 1:
        return n
    memo[n] = fib(n-1, memo) + fib(n-2, memo)
    return memo[n]
```

特点：

-   还是递归思维
    
-   只算需要的状态
    
-   每个状态只算一次
    
-   时间复杂度降为 `O(n)`
    

---

## 3.3 改成递推（Bottom-up）

```python
def fib(n):
    if n <= 1:
        return n
    dp = [0] * (n + 1)
    dp[1] = 1
    for i in range(2, n + 1):
        dp[i] = dp[i-1] + dp[i-2]
    return dp[n]
```

进一步优化空间：

```python
def fib(n):
    if n <= 1:
        return n
    a, b = 0, 1
    for _ in range(2, n + 1):
        a, b = b, a + b
    return b
```

---

## 3.4 这三者的关系

-   **暴力递归**：定义问题结构，但重复多
    
-   **记忆化搜索**：递归 + 缓存，属于 DP
    
-   **递推 DP**：显式按顺序填表，属于 DP
    

所以你可以把 DP 看成：

> **“对递归子问题求解过程的缓存与组织”**

---

# 4\. 动态规划的标准思考框架

这是最重要的一部分。以后做题几乎都按这个顺序思考。

---

## 第一步：定义状态（State）

状态就是“**一个子问题的完整描述**”。

状态定义要回答：

> “我要求的答案，如果缩小成一个子问题，需要用哪些变量唯一描述？”

比如：

-   `dp[i]`：表示前 `i` 个元素/到第 `i` 个位置时的答案
    
-   `dp[i][j]`：表示考虑前 `i` 个物品、容量为 `j` 时的答案
    
-   `dp[i][j]`：表示区间 `[i, j]` 上的答案
    
-   `dp[i][k]`：表示到位置 `i` 且状态为 `k` 的答案
    

### 好状态的标准

一个好的状态定义应满足：

1.  **能够表示子问题**
    
2.  **能够由更小状态转移而来**
    
3.  **不遗漏信息**
    
4.  **尽量不冗余**
    

---

## 第二步：写出状态转移方程（Transition）

状态转移的核心问题：

> “最后一步发生了什么？”

或者：

> “当前状态是由哪些更小状态转移来的？”

这是 DP 的灵魂。

例如爬楼梯：

-   到第 `i` 级台阶，最后一步可能来自 `i-1` 或 `i-2`
    
-   所以： [ dp[i] = dp[i-1] + dp[i-2] ]
    

例如 0/1 背包：

-   对第 `i` 个物品，要么不选，要么选
    
-   所以： [ dp[i][j] = \\max(dp[i-1][j],\\ dp[i-1][j-w\_i]+v\_i) ]
    

---

## 第三步：确定初始条件（Base Case）

递推总要有起点。

例如：

-   `dp[0] = 1` 还是 `0`？
    
-   `dp[0][0] = 0`？
    
-   空串与空串的编辑距离是多少？
    

初始条件不是“随便填”，它必须和状态定义严格一致。

---

## 第四步：确定计算顺序

因为 `dp[state]` 依赖更小状态，所以你必须保证：

> **在计算当前状态前，所有依赖状态已经算出来。**

例如：

-   线性 DP：从小到大
    
-   区间 DP：按区间长度从短到长
    
-   背包 DP：看是 0/1 还是完全背包，循环方向不同
    
-   DAG 上 DP：按拓扑顺序
    

---

## 第五步：看是否能优化

优化方向包括：

-   **空间压缩**
    
-   **去掉无效维度**
    
-   **单调队列优化**
    
-   **斜率优化**
    
-   **状态机简化**
    
-   **前缀和辅助**
    
-   **滚动数组**
    

但注意：

> **先把对的 DP 写出来，再谈优化。**

---

# 5\. 你必须真正理解的“状态”是什么

很多人学 DP 最大障碍就是“不会定义状态”。

## 5.1 状态不是程序变量，而是“子问题编号”

比如背包问题中的 `dp[i][j]`，不是“数组第 i 行 j 列”这么机械地理解，而是：

> **前 i 个物品中选，容量不超过 j 时，能得到的最大价值**

这其实是一个子问题。

所以本质上：

> **DP 数组只是存储子问题答案的容器。**

---

## 5.2 常见状态设计思路

### 1）按前缀设计

最常见

-   `dp[i]`：前 `i` 个字符/数处理完后的答案
    
-   适合字符串、数组、序列
    

### 2）按位置设计

-   `dp[i]`：以 `i` 结尾的答案
    
-   比如最长上升子序列
    

### 3）按区间设计

-   `dp[i][j]`：区间 `[i, j]` 的答案
    
-   适合括号匹配、回文、矩阵连乘、戳气球
    

### 4）按容量/资源设计

-   `dp[i][j]`：资源限制为 `j` 时的最优值
    
-   背包类问题
    

### 5）按“决策阶段 + 当前条件”设计

-   股票问题：第 `i` 天、是否持股、交易次数
    
-   状态机 DP
    

### 6）按集合设计

-   `dp[mask][i]`
    
-   表示访问过一组点、当前在 `i`
    
-   适合状压 DP，如旅行商问题
    

### 7）按树结构设计

-   `dp[u][...]`
    
-   以节点 `u` 为根的子树上的答案
    
-   树形 DP
    

---

# 6\. 动态规划和递归、回溯、分治的区别与联系

这部分非常关键，因为很多人会混淆。

---

# 6.1 DP 和回溯的关系

## 回溯是什么？

回溯本质是：

> **通过枚举所有可能决策路径，在搜索树中寻找满足条件的解。**

典型问题：

-   全排列
    
-   子集
    
-   N 皇后
    
-   数独
    
-   组合总和
    

回溯通常长这样：

```python
def dfs(path, choices):
    if 终止条件:
        记录答案
        return
    for 每个可选决策:
        做选择
        dfs(...)
        撤销选择
```

---

## DP 和回溯的联系

### 联系 1：都在处理“状态”

-   回溯：搜索状态空间树
    
-   DP：在状态空间图上去重求值
    

### 联系 2：都可以从“递归定义”出发

很多 DP 可以先写成递归，再记忆化。

### 联系 3：本质都涉及“决策”

DP 的转移方程，本质也是在“枚举最后一步或下一步决策”。

---

## 区别

### 回溯：

-   目标常常是**找所有解**、判断是否存在解
    
-   枚举路径
    
-   即使状态重复，很多时候也会重复搜索
    
-   强调“试探、撤销”
    
-   典型复杂度指数级
    

### DP：

-   目标通常是**最优值、方案数、可行性**
    
-   不关心具体枚举所有路径（除非额外恢复）
    
-   对重复子问题进行缓存
    
-   强调“子问题答案复用”
    
-   常能把指数级降到多项式级
    

---

## 一个最关键的区别

> **回溯是“遍历解空间树”**<br>**DP是“压缩重复子问题后的状态图求值”**

如果回溯搜索树中不同路径会抵达同一个状态，那就有可能转成 DP 或记忆化搜索。

---

## 举例：爬楼梯

### 回溯思路

每次走 1 步或 2 步，枚举所有走法：

```python
def dfs(remain):
    if remain == 0:
        return 1
    if remain < 0:
        return 0
    return dfs(remain-1) + dfs(remain-2)
```

这其实已经是递归了。因为 `dfs(remain)` 会重复出现，所以可记忆化，变成 DP。

所以：

> **很多 DP 问题，最原始的暴力形式就是回溯/递归。**

---

## 什么情况下更像回溯而不是 DP？

当问题要求：

-   输出所有方案
    
-   枚举所有排列/组合
    
-   状态重复不明显
    
-   剪枝比记忆化更关键
    

例如：

-   全排列
    
-   N 皇后所有解
    
-   子集枚举
    

---

# 6.2 DP 和分治的关系

## 分治是什么？

分治（Divide and Conquer）：

> **把问题分成若干相互独立的子问题，分别解决后再合并。**

经典例子：

-   归并排序
    
-   快速排序
    
-   二分查找
    
-   线段树构建
    
-   最近点对
    

---

## DP 和分治的联系

它们都在做：

-   大问题拆成小问题
    
-   利用子问题结果构造原问题结果
    

---

## 核心区别

### 分治：

-   子问题通常**相互独立**
    
-   一般不需要缓存重复子问题
    
-   合并过程是关键
    

### DP：

-   子问题通常**有重叠**
    
-   需要复用子问题答案
    
-   转移关系是关键
    

---

## 举例对比

### 归并排序

-   排左半边、排右半边
    
-   左右子问题彼此独立
    
-   合并有序数组
    
-   这是分治，不是 DP
    

### Fibonacci

-   `f(n-1)` 和 `f(n-2)` 有大量重叠
    
-   需要缓存
    
-   这是 DP
    

---

## 一个重要认识

> **DP 可以看作“带记忆的递归拆分”**<br>**分治可以看作“无重叠子问题的递归拆分”**

---

# 6.3 DP、回溯、分治三者统一理解

你可以建立下面这个总框架：

## 1）回溯

-   搜索“决策树”
    
-   关注枚举路径与可行解
    
-   适合组合爆搜问题
    

## 2）分治

-   拆成“独立子问题”
    
-   关注合并结果
    
-   适合排序、查找、结构化递归
    

## 3）DP

-   拆成“重叠子问题”
    
-   关注状态复用与转移
    
-   适合最优值、计数、可行性问题
    

---

## 一句话区分

-   **回溯：搜所有可能**
    
-   **分治：拆开分别算再合并**
    
-   **DP：子问题会重复，所以记住答案再用**
    

---

# 7\. 动态规划的几种实现形态

---

## 7.1 记忆化搜索（自顶向下）

特点：

-   代码更贴近递归定义
    
-   好想、好写
    
-   有时只会访问一部分状态
    
-   但递归深度可能有限
    

适合：

-   状态转移自然写成递归
    
-   图 DP、树 DP、区间 DP 的某些题
    

---

## 7.2 递推（自底向上）

特点：

-   可控、效率高
    
-   容易做空间优化
    
-   更适合竞赛和工程实现
    

适合：

-   能明确排序状态的依赖关系
    
-   线性 DP、背包 DP、二维 DP
    

---

## 7.3 状态压缩 DP

当状态是某个集合时，用二进制掩码表示。

例如：

-   `mask` 表示哪些元素已选
    
-   `dp[mask]`
    
-   `dp[mask][i]`
    

典型：

-   TSP
    
-   集合覆盖类小规模问题
    

---

## 7.4 树形 DP

子问题定义在树上。

比如：

-   选或不选当前节点
    
-   子树内最优值
    
-   树上最大独立集
    

---

## 7.5 区间 DP

子问题定义在区间上：

-   `dp[i][j]`
    

典型：

-   最长回文子序列
    
-   矩阵连乘
    
-   戳气球
    
-   合并石子
    

---

## 7.6 状态机 DP

适合“过程状态有限”的问题。

典型：

-   股票买卖
    
-   某些字符串匹配
    
-   限制相邻选择问题
    

---

# 8\. 如何判断一道题能不能用 DP？

你可以用下面这套判定清单。

---

## 问题一：是否在求“最优值 / 方案数 / 是否可行”？

DP 最常见输出就是这三类：

-   最大/最小值
    
-   方案总数
    
-   能否做到
    

如果是“输出所有方案”，通常更偏回溯。

---

## 问题二：能否定义子问题？

例如：

-   前 `i` 个元素
    
-   到位置 `i`
    
-   区间 `[i, j]`
    
-   已用容量 `j`
    
-   子树 `u`
    
-   已访问集合 `mask`
    

如果无法稳定定义子问题，DP 就难做。

---

## 问题三：子问题是否重复？

如果暴力递归时会多次计算同一个状态，就是强烈信号。

---

## 问题四：当前决策是否只依赖有限个更小状态？

如果一个状态要依赖“整个历史路径细节”，就可能不适合普通 DP，除非把这些信息也纳入状态。

---

## 问题五：状态数是否可承受？

即使理论能做 DP，状态太多也不行。

例如：

-   状态数 `10^8` 可能就危险
    
-   要考虑时间和空间复杂度
    

---

# 9\. 动态规划最核心的“建模方法”

这里是从“会做”到“真正理解”的关键。

---

# 9.1 方法一：从“最后一步”倒推

这是最经典的方法。

问自己：

> “要求当前问题的答案，最后一次决策是什么？”

例如：

### 爬楼梯

最后一步：

-   从 `i-1` 来
    
-   从 `i-2` 来
    

### 0/1 背包

看第 `i` 个物品：

-   不选
    
-   选
    

### 编辑距离

最后一步：

-   插入
    
-   删除
    
-   替换
    
-   或字符相同，不操作
    

### 最长上升子序列

如果定义 `dp[i]` 为“以 `i` 结尾的 LIS 长度”，那最后一个前驱是谁？

---

# 9.2 方法二：从“枚举选/不选”建模

非常适合背包、打家劫舍、股票、序列选择类问题。

例如打家劫舍：

-   偷当前房子，则前一个不能偷
    
-   不偷当前房子，则继承前面最优
    

---

# 9.3 方法三：从“前缀划分”建模

常见于字符串和数组切分问题。

例如单词拆分：

-   `dp[i]` 表示前 `i` 个字符能否被字典拆分
    
-   枚举最后一个单词起点 `j`
    
-   如果 `dp[j] = True` 且 `s[j:i]` 在字典里，则 `dp[i]=True`
    

---

# 9.4 方法四：从“区间合并/断点划分”建模

区间 DP 常这样思考：

> `[i, j]` 的答案，枚举某个分界点 `k`

例如：

-   矩阵链乘法
    
-   戳气球
    
-   合并石子
    
-   最长回文子序列（看首尾关系）
    

---

# 9.5 方法五：把约束变成状态

例如股票问题：

-   第 `i` 天
    
-   是否持股
    
-   已交易次数
    

约束“最多交易 k 次”不能忽略，所以必须进状态。

---

# 10\. 经典 DP 例题串讲

下面用几个典型例题建立模型感。

---

# 10.1 爬楼梯：最基础计数型 DP

题意： 每次走 1 或 2 步，问到第 `n` 级有多少种走法。

## 状态

`dp[i]`：到第 `i` 级的方法数

## 转移

[ dp[i]=dp[i-1]+dp[i-2] ]

## 初始

-   `dp[0]=1`：到第 0 级有 1 种方式（什么都不做）
    
-   `dp[1]=1`
    

## 代码

```python
def climbStairs(n):
    dp = [0] * (n + 1)
    dp[0] = 1
    for i in range(1, n + 1):
        dp[i] += dp[i - 1]
        if i >= 2:
            dp[i] += dp[i - 2]
    return dp[n]
```

### 易错点

-   `dp[0]` 设多少要和定义一致
    
-   计数问题和最优问题不要混
    

---

# 10.2 打家劫舍：最优型 DP

题意： 不能偷相邻房子，求最大收益。

## 状态

`dp[i]`：前 `i` 个房子能偷到的最大值

## 转移

第 `i` 个房子（下标从 1）：

-   不偷：`dp[i-1]`
    
-   偷：`dp[i-2] + nums[i-1]`
    

[ dp[i] = \\max(dp[i-1], dp[i-2]+nums[i-1]) ]

## 初始

-   `dp[0]=0`
    
-   `dp[1]=nums[0]`
    

```python
def rob(nums):
    n = len(nums)
    if n == 0:
        return 0
    dp = [0] * (n + 1)
    dp[1] = nums[0]
    for i in range(2, n + 1):
        dp[i] = max(dp[i-1], dp[i-2] + nums[i-1])
    return dp[n]
```

### 本质

这是“选/不选当前元素”的典型模型。

---

# 10.3 最长上升子序列（LIS）：以某位置结尾

## 状态

`dp[i]`：**以** `nums[i]` **结尾** 的最长上升子序列长度

注意不是“前 i 个元素”的 LIS，而是“必须以 i 结尾”。

## 转移

枚举前面位置 `j < i`，如果 `nums[j] < nums[i]`： [ dp[i] = \\max(dp[i], dp[j] + 1) ]

## 初始

每个元素单独成序列： [ dp[i]=1 ]

```python
def lengthOfLIS(nums):
    n = len(nums)
    dp = [1] * n
    for i in range(n):
        for j in range(i):
            if nums[j] < nums[i]:
                dp[i] = max(dp[i], dp[j] + 1)
    return max(dp)
```

### 为什么这样定义状态？

因为“以 i 结尾”才方便找前驱并转移。<br>这体现了：**状态定义是为了方便转移。**

---

# 10.4 0/1 背包：资源约束型 DP

题意： 有 `n` 个物品，每个物品只能选一次，重量 `w[i]`，价值 `v[i]`，背包容量 `W`，求最大总价值。

## 二维状态

`dp[i][j]`：前 `i` 个物品中选，容量不超过 `j` 时最大价值

## 转移

不选第 `i` 个： [ dp[i][j] = dp[i-1][j] ]

选第 `i` 个（如果能放）： [ dp[i][j] = \\max(dp[i][j], dp[i-1][j-w\_i]+v\_i) ]

完整： [ dp[i][j]= \\begin{cases} dp[i-1][j], & j<w\_i \\ \\max(dp[i-1][j], dp[i-1][j-w\_i]+v\_i), & j\\ge w\_i \\end{cases} ]

---

## 一维优化

```python
def knapsack_01(weights, values, W):
    dp = [0] * (W + 1)
    for w, v in zip(weights, values):
        for j in range(W, w - 1, -1):
            dp[j] = max(dp[j], dp[j - w] + v)
    return dp[W]
```

### 为什么倒序？

为了保证每个物品只用一次。<br>如果正序，当前轮更新后的 `dp[j-w]` 可能又被当前物品再次利用，相当于重复选。

这个循环方向是背包 DP 的关键细节。

---

# 10.5 完全背包：和 0/1 背包的本质区别

每个物品可选无限次。

转移变成： [ dp[j] = \\max(dp[j], dp[j-w]+v) ]

但循环要**正序**：

```python
def complete_knapsack(weights, values, W):
    dp = [0] * (W + 1)
    for w, v in zip(weights, values):
        for j in range(w, W + 1):
            dp[j] = max(dp[j], dp[j - w] + v)
    return dp[W]
```

### 为什么正序？

因为允许同一物品重复选，所以当前物品更新出的 `dp[j-w]` 可以继续用于更新 `dp[j]`。

---

# 10.6 编辑距离：二维字符串 DP

给两个字符串 `word1`, `word2`，允许插入、删除、替换，求最少操作数。

## 状态

`dp[i][j]`：`word1` 前 `i` 个字符变成 `word2` 前 `j` 个字符的最小操作数

## 转移

若最后字符相同： [ dp[i][j] = dp[i-1][j-1] ]

否则：

-   插入：`dp[i][j-1] + 1`
    
-   删除：`dp[i-1][j] + 1`
    
-   替换：`dp[i-1][j-1] + 1`
    

[ dp[i][j] = \\min(dp[i][j-1]+1,\\ dp[i-1][j]+1,\\ dp[i-1][j-1]+1) ]

## 初始

-   `dp[i][0]=i`
    
-   `dp[0][j]=j`
    

这类题最能训练你对“最后一步”的理解。

# 11\. 动态规划中的关键细节与常见坑

这部分非常重要，很多题不是不会，而是细节错。

---

## 11.1 状态定义不清，导致转移混乱

比如 LIS：

-   `dp[i]` 定义成“前 i 个数的 LIS”就不好转移
    
-   定义成“以 i 结尾的 LIS”就好转移
    

**原则：状态定义是为转移服务的。**

---

## 11.2 初始条件不匹配

例如：

-   计数题 `dp[0]` 到底是 0 还是 1？
    
-   背包初始化是否允许“不选任何物品”？
    
-   不可达状态是否应该设为 `-inf`？
    

---

## 11.3 计算顺序错误

比如：

-   区间 DP 没按长度枚举
    
-   背包方向错
    
-   一个状态依赖未来状态但你提前算了
    

---

## 11.4 把“最大值问题”和“方案数问题”混淆

例如：

-   求最大价值：用 `max`
    
-   求方案数：用加法
    
-   求是否可行：用布尔或
    

这三类问题转移形式完全不同。

---

## 11.5 没处理“不可能状态”

例如某些题中：

-   某状态根本达不到
    
-   这时不能默认初始化为 0
    
-   可能应设为负无穷/正无穷/False
    

例如最大值题中，如果初始化为 0，会把不可达状态误当成可达。

---

## 11.6 空间压缩后覆盖了还没用的状态

尤其是一维背包，循环顺序非常关键。

---

## 11.7 过度套模板

DP 不是“看到数组就 `dp[i]`，看到两个数组就 `dp[i][j]`”。

要先回答：

-   子问题是什么？
    
-   转移依据是什么？
    
-   为什么不会漏？
    
-   为什么不会重？
    

---

# 13\. 如何把一道题从暴力变成 DP？

这是你真正掌握 DP 的关键流程。

---

## 通用流程

### Step 1：先写暴力递归

问：

-   每一步能做什么选择？
    
-   终止条件是什么？
    

### Step 2：看递归参数

这些参数如果唯一确定一个子问题，那么它们就是状态。

### Step 3：看是否重复

如果同样参数组合被反复调用，就有重叠子问题。

### Step 4：加记忆化

先让递归正确。

### Step 5：改成递推

如果需要更高效率或更好控制。

---

## 非常重要的一句话

> **不会 DP 时，先写递归；递归参数就是状态候选；递归分支就是转移候选。**

这是一条非常实用的学习路径。

# 14\. 学动态规划最难的地方到底是什么？

真正难点不是代码，而是：

## 1）抽象子问题

把原问题压缩成合适状态

## 2）找到无后效性

也就是：

> 当前状态一旦确定，未来决策不需要知道“是怎么走到这个状态的”。

如果未来还依赖更早的具体历史，那你的状态信息不够，需要补充。

---

## 3）设计合适维度

状态过少会丢信息，状态过多会爆炸。

这是 DP 的核心建模能力。

---

# 15\. “无后效性”一定要懂

这是 DP 理论中的关键概念。

## 定义直观理解

当你已经知道当前状态时，后续如何决策只依赖当前状态，而不依赖到达当前状态的具体路径。

比如最短路径中：

-   到达某点的最小代价已知
    
-   后续只关心“当前在这个点，代价是多少”
    
-   不关心你是怎么来的
    

这就适合 DP/最短路。

---

## 反例思考

如果问题要求：

-   “不能和过去某个复杂模式冲突”
    
-   而这个复杂历史没放进状态 那就不满足无后效性。
    

所以：

> **状态必须携带所有会影响未来决策的信息。**

---

# 16\. 动态规划和贪心的区别

虽然你没直接问，但这和 DP 理解很相关。

## 贪心

每一步做局部最优，希望得到全局最优

## DP

系统比较所有可能子决策的最优结果，再组合出全局最优

### 区别

-   贪心不回头、不保存大量状态
    
-   DP会保留多个子问题答案
    

### 什么时候贪心可行？

当问题具有“贪心选择性质”时。<br>否则通常需要 DP。

例如：

-   活动选择问题：贪心可做
    
-   0/1 背包：贪心不行，通常要 DP

2.  **暴力怎么做？**
    
    -   最后一步/枚举选择是什么？
        
3.  **子问题怎么定义？**
    
    -   `dp[...]` 具体含义是什么？
        
4.  **状态转移是什么？**
    
    -   为什么完整？为什么不重不漏？
        
5.  **初始条件是什么？**
    
    -   与状态定义是否一致？
        
6.  **计算顺序是什么？**
    
    -   依赖状态是否已知？
        
7.  **复杂度是多少？**
    
    -   状态数 × 每个状态转移代价
        
8.  **能否优化？**
    
    -   空间、转移、预处理

# 19\. 一个很容易混淆的问题：DP 是否一定比回溯好？

不一定。

## DP 更适合：

-   最优值
    
-   方案数
    
-   存在性
    
-   状态重复明显
    

## 回溯更适合：

-   输出所有解
    
-   解空间需要遍历
    
-   约束复杂但重复状态少
    
-   剪枝效果显著
    

例如 N 皇后：

-   求所有方案：回溯很自然
    
-   求方案数：也可以回溯计数
    
-   如果规模和状态重复有结构，也可以设计位运算优化，但本质还是搜索
    

所以不要机械认为“能搜就一定转 DP”。

# 20\. 一个更高层的统一视角：状态空间图

这是帮助你彻底打通理解的高级视角。

把每个“状态”看作图上的一个节点：

-   回溯：在图/树上深搜路径
    
-   分治：递归拆到独立分支再合并
    
-   DP：在有向无环状态图上，对每个节点值进行复用计算
    

所以 DP 本质上是在做：

> **对状态图进行动态求值**

这也是为什么：

-   记忆化搜索 = DFS + 缓存
    
-   递推 DP = 按拓扑顺序算状态图
    

这个视角一旦建立，很多方法就统一了。

错。真正懂是你能解释：

-   为什么这么定义？
    
-   为什么转移完整？
    
-   为什么顺序正确？
    
-   为什么复杂度是这样？
