继续回答 **Q8 ~ Q12**。

# Q8：检索结果相关性不高，怎么优化召回？

先说一个关键判断：

> **“相关性不高”不一定是召回问题，也可能是排序问题、切块问题、Query 理解问题。**

## 1）先定位是哪一层出了问题

建议先做分层诊断：

-   **Top50 里有正确文档，但 Top5 没有**<br>→ 主要是 **排序 / rerank** 问题
    
-   **Top50 里都没有正确文档**<br>→ 主要是 **召回** 问题
    
-   **召回到了，但答案仍然不对**<br>→ 可能是 **chunk 过碎/过大、上下文拼装、生成阶段** 问题
    

常用诊断指标：

-   Recall@K
    
-   HitRate@K
    
-   MRR / nDCG
    
-   Empty Recall Rate（空召回率）
    
-   Answer Supported Rate（回答是否被证据支持）
    

---

## 2）优化召回的核心手段

## A. Query 侧优化

### ① Query Rewrite

把用户口语问题改写成检索友好的查询：

-   同义词归一：退款/退货/售后
    
-   别名归一：88VIP / 88会员
    
-   纠错：错别字、简称、口语化表达
    
-   指代消解：这单/那个商品/上周买的鞋
    

### ② Multi-Query Expansion

对一个问题生成多个检索子问题：

-   原问题
    
-   关键词版
    
-   实体版
    
-   条件版
    

比如：

> “88VIP 的退货运费券可以提现吗？”

可拆成：

-   88VIP 运费权益 提现规则
    
-   退货运费券 使用限制
    
-   88VIP 权益 退款 现金提现
    

### ③ Query Decomposition

复杂问题拆成多个检索子任务：

-   查规则
    
-   查活动资格
    
-   查订单上下文
    

---

## B. 文档与索引侧优化

### ① Chunk 重做

很多“相关性差”其实是 chunk 切坏了：

-   FAQ：按问答对切
    
-   规则：按条款/标题层级切
    
-   商品：按字段化切，不要整页一起切
    
-   SOP：按步骤切
    

### ② 元数据补强

给 chunk 增加 metadata：

-   类目
    
-   业务域
    
-   店铺
    
-   品牌
    
-   时间
    
-   适用人群
    
-   规则生效区间
    

这样可以做 **过滤召回**，比纯向量更稳。

### ③ 多粒度索引

同一份知识建立多套索引：

-   小 chunk：提升命中率
    
-   大 chunk：提升可读性
    
-   标题索引：提升精确定位
    
-   FAQ 索引 / 规则索引 / 商品索引分开管理
    

---

## C. 召回策略优化

### ① Hybrid Search

生产上不要只靠向量检索，建议：

> **Dense Vector + BM25 + Metadata Filter + Rerank**

适用原因：

-   电商/规则类问题里，**关键词精确匹配**很重要
    
-   实体词、品牌词、规则编号、活动名，BM25 往往比纯向量更稳
    

### ② 分路召回

按意图路由到不同知识源：

-   商品咨询 → 商品知识索引
    
-   物流/售后 → 订单/售后规则索引
    
-   平台政策 → 规则库
    
-   商家问题 → 商家后台知识库
    

### ③ 先过滤再召回

淘天场景特别重要：

-   先确定类目、店铺、租户、时间范围
    
-   再做向量召回<br>否则会出现“语义像，但业务域不对”的问题
    

---

## D. 排序与模型优化

### ① 上 Reranker

如果 top50 里已经有正确文档，但排序靠后，优先上：

-   Cross-Encoder Reranker
    
-   Domain Reranker
    

### ② 领域化 Embedding

通用 embedding 对电商术语、规则术语不一定好，建议：

-   用业务日志做 hard negative mining
    
-   训练/微调领域 embedding 或 reranker
    

---

## E. 反馈闭环

建立“检索优化飞轮”：

-   收集未命中 query
    
-   收集低满意答案
    
-   收集人工标注正负样本
    
-   周期性重训 reranker / 同义词词典 / query rewrite 模板
    

---

## 3）淘天场景建议

如果是淘天这种复杂场景，我推荐：

```text
Query理解
  → 意图识别 + 实体抽取 + Query Rewrite
  → 分业务域路由
  → BM25 + Dense 混合召回
  → Metadata过滤
  → Rerank
  → Parent文档回填
  → 再交给LLM生成
```

### 一句话总结

> 召回优化的重点不是“换个向量库”，而是 **Query 理解、混合检索、元数据过滤、rerank、反馈闭环**。

# Q9：Agent 推理链路包含多个串行工具，响应延迟高，该如何优化？

核心原则就四个字：

> **减、并、缓存、降级**

---

## 1）先做链路拆解，找关键路径

不要一上来就“换模型”，先看时间花在哪：

-   Planner LLM：多少 ms
    
-   每个工具：多少 ms
    
-   检索：多少 ms
    
-   反思：多少 ms
    
-   最终生成：多少 ms
    

重点看：

-   P50 / P95 / P99
    
-   最慢工具 TopN
    
-   串行依赖是否真的必要
    

---

## 2）减少不必要的串行步骤

## A. 把“Thought Loop”变成“DAG”

很多 Agent 慢，是因为模型一边想一边临时决定下一步。<br>生产上建议改成：

-   先产出结构化计划
    
-   能并行的步骤并行
    
-   只保留必要的依赖关系
    

例如：

原链路：

```text
查订单 → 查物流 → 查退款规则 → 查优惠资格 → 最终回答
```

优化后：

```text
并行(查订单, 查退款规则, 查优惠资格) → 若命中订单再查物流 → 最终回答
```

---

## B. 合并工具

如果两个工具总是一起调，最好在后端做聚合 API：

-   `getOrder`
    
-   `getLogistics`
    
-   `getRefundEligibility`
    

可聚成：

-   `getOrderServiceSnapshot`
    

这样比 Agent 自己串行调三个 RPC 更快、更稳。

---

## 3）把真正能并行的地方并行化

适合并行的场景：

-   多知识源召回
    
-   多候选订单查询
    
-   规则查询 + 商品信息查询
    
-   多个只读工具
    

不适合并行：

-   明显有前置依赖的写操作
    
-   高副作用工具
    

---

## 4）缓存与预取

## A. 工具结果缓存

-   订单基础信息：短 TTL
    
-   商品属性：中 TTL
    
-   平台规则：长 TTL
    
-   高频 FAQ：语义缓存
    

## B. 检索缓存

-   query rewrite 结果缓存
    
-   retrieval topK 缓存
    
-   rerank 结果缓存
    

## C. 预测性预取

如果意图已识别为“物流咨询”，可以提前并发预取：

-   最近订单
    
-   物流状态
    
-   售后规则
    

---

## 5）减少模型调用次数

这是最直接的延迟优化手段。

### 典型做法：

-   小模型做意图识别 / 工具路由
    
-   大模型只做复杂规划和最终回答
    
-   简单 FAQ 不走完整 Agent 闭环
    
-   反思模块只在失败、高风险场景触发
    

---

## 6）控制上下文大小

上下文越大，模型越慢、越贵。

优化方式：

-   会话摘要代替全量历史
    
-   工具只返回必要字段
    
-   检索只回 topK 精简片段
    
-   去重相似 chunk
    
-   长结果先压缩再喂给 LLM
    

---

## 7）工程层优化

-   HTTP Keep-Alive / 连接池 / gRPC
    
-   请求超时与 deadline 透传
    
-   熔断、重试、hedged request（尾延迟优化）
    
-   批量查询代替 N 次单查
    
-   本地热缓存 + Redis
    

---

## 8）用户体验优化

即使总耗时没法极致压缩，也可以优化“感知延迟”：

-   流式输出
    
-   先返回阶段性结果
    
-   长耗时操作异步化
    
-   显示“正在查询订单/物流”
    

---

## 9）实战建议

生产上建议优先做这 5 件事：

1.  先量化关键路径
    
2.  把串行改 DAG
    
3.  合并高频串行工具
    
4.  缓存只读查询
    
5.  简单场景绕过复杂 Agent
    

---

# Q10：高并发场景下，如何设计 Agent 服务的弹性伸缩策略？

Agent 系统的弹性，不是只看 CPU 扩容。

> **真正要扩的是：并发、队列、tokens/s、外部依赖容量、模型配额。**

---

## 1）先把系统拆成可独立伸缩的层

建议拆成：

-   **Gateway 层**：接入、限流、鉴权
    
-   **Orchestrator 层**：Agent 编排，尽量无状态
    
-   **Retrieval 层**：检索、rerank
    
-   **Tool Adapter 层**：对接订单/商品/物流等服务
    
-   **Model Gateway 层**：模型路由、限流、熔断
    
-   **Async Worker 层**：异步任务、记忆写回、回放评测
    

这样每一层可以分别扩缩。

---

## 2）Orchestrator 要尽量无状态

会话状态不要放在 Pod 本地：

-   Session → Redis / DB
    
-   Memory → 独立服务
    
-   Trace / Logs → 观测平台
    

这样扩容时实例可随时加减，不会丢会话。

---

## 3）弹性伸缩指标要看“组合信号”

建议用 HPA/KEDA + 自定义指标：

### 核心指标

-   QPS
    
-   In-flight requests
    
-   队列长度 / queue lag
    
-   P95/P99 延迟
    
-   Tokens per second
    
-   模型 RPM/TPM 利用率
    
-   工具服务并发占用
    
-   错误率 / 超时率
    

### 为什么不能只看 CPU？

因为 Agent 常见瓶颈不是 CPU，而是：

-   外部模型配额
    
-   下游工具服务
    
-   向量检索吞吐
    
-   长尾请求堆积
    

---

## 4）做分级限流和优先级队列

高并发时一定要“保核心链路”。

建议按场景分级：

-   P0：支付、退款、物流、售后
    
-   P1：客服问答
    
-   P2：导购建议、内容生成
    

当系统吃紧时：

-   优先保 P0/P1
    
-   降级 P2
    
-   限制低优先级用户/租户流量
    

---

## 5）降级策略要预先设计好

高峰时不可能所有请求都走“完整版 Agent”。

可降级项：

-   大模型 → 小模型
    
-   多轮规划 → 单轮直接回答
    
-   关闭反思
    
-   topK 从 20 降到 8
    
-   关闭 rerank 或切轻量版
    
-   优先使用缓存答案
    
-   长任务异步化
    

---

## 6）削峰填谷：队列 + 背压

对于非强实时任务：

-   先进 MQ / 任务队列
    
-   Worker 按能力消费
    
-   对上游返回“已受理，稍后通知”
    

同时要做背压：

-   队列过长时拒绝低优先请求
    
-   超预算请求直接 fail-fast
    
-   下游超载时快速熔断
    

---

## 7）大促/活动场景的专项策略

淘天场景一定要有：

-   **预扩容**：活动前提前拉高实例数
    
-   **热身**：模型连接池、缓存预热、热点知识预加载
    
-   **热点隔离**：热门活动单独资源池
    
-   **租户隔离**：避免某商家/场景打爆全局
    
-   **多 AZ / 多机房容灾**
    

---

## 8）一句话落地方案

> **无状态编排 + 分层扩容 + 队列削峰 + 优先级调度 + 多级降级 + 预扩容**

---

# Q11：Agent 系统的核心监控指标有哪些？怎么设计告警策略？

先给结论：

> Agent 监控不能只看“接口 200/500”，必须同时监控 **质量、性能、成本、工具、检索、安全**。

---

## 1）核心监控指标分层

## A. 业务与效果指标

-   请求量、活跃用户数
    
-   任务完成率 / Task Success Rate
    
-   首次解决率
    
-   人工转接率
    
-   用户满意度 / 点踩率
    
-   最终动作成功率（如退款申请成功率）
    

---

## B. Agent 行为指标

-   平均 step 数
    
-   平均工具调用次数
    
-   重规划率
    
-   反思触发率
    
-   fallback 率
    
-   同一工具重复调用率
    
-   循环率 / dead-loop rate
    

这些指标很关键，能直接发现“Agent 在瞎跑”。

---

## C. 模型指标

-   LLM 调用成功率
    
-   P50/P95/P99 延迟
    
-   输入/输出 tokens
    
-   每请求 token 成本
    
-   JSON / Function Call 结构化成功率
    
-   拒答率 / 安全拦截率
    

---

## D. 工具指标

-   每个工具的 QPS、成功率、超时率
    
-   业务错误码分布
    
-   重试率
    
-   幂等冲突率
    
-   写操作成功率
    
-   工具级 P95 延迟
    

---

## E. RAG 指标

-   空召回率
    
-   Recall@K / Hit@K
    
-   Rerank 命中提升率
    
-   检索耗时
    
-   知识新鲜度
    
-   Chunk 命中分布
    
-   引用证据覆盖率
    

---

## F. 系统与资源指标

-   API P95/P99
    
-   队列堆积
    
-   CPU / 内存 / GC
    
-   Redis/DB/QPS
    
-   向量库延迟
    
-   缓存命中率
    
-   Pod 重启 / 冷启动次数
    

---

## G. 安全与合规指标

-   Prompt Injection 命中率
    
-   权限拒绝次数
    
-   敏感信息脱敏失败
    
-   高风险工具调用次数
    
-   人工审核触发率
    

---

## 2）告警策略怎么设计？

## A. 按 SLO 设计，而不是只按阈值

例如：

-   Agent 成功率 < 99%
    
-   P95 延迟 > 3s
    
-   关键工具超时率 > 2%
    
-   空召回率 > 8%
    
-   死循环率 > 1%
    

阈值要按场景定，不是固定值。

---

## B. 分级告警

### P1：立即处理

-   核心链路不可用
    
-   写操作错误飙升
    
-   模型服务大面积超时
    
-   Trace 显示链路断裂
    

### P2：尽快处理

-   P95 延迟持续恶化
    
-   空召回率上升
    
-   rerank 效果显著下降
    
-   fallback 率升高
    

### P3：观察优化

-   token 成本异常上涨
    
-   平均 step 数变多
    
-   缓存命中率下降
    

---

## C. 症状告警 + 原因告警分开

例如：

-   症状：任务完成率下降
    
-   原因：某个工具 timeout 激增
    

这样值班同学不会只看到“结果坏了”，还能快速定位原因。

---

## D. 做抑制与去噪

避免告警风暴：

-   连续 N 个周期异常才告警
    
-   低流量时不触发比例告警
    
-   同一根因做聚合
    
-   发布窗口单独监控
    

---

## E. 线上告警 + 离线评测双轨制

有些质量问题不适合实时告警，比如：

-   回答准确率
    
-   工具选择准确率
    
-   幻觉率
    

这些适合：

-   日级回放评测
    
-   版本对比
    
-   灰度 A/B
    

---

## 3）推荐最小告警集

如果刚开始做，先盯这 8 个：

1.  总成功率
    
2.  P95 延迟
    
3.  模型调用错误率
    
4.  关键工具超时率
    
5.  空召回率
    
6.  fallback 率
    
7.  同工具重复调用率
    
8.  单请求平均 token 成本
    

---

# Q12：如何追踪 Agent 的单次请求完整链路？跨多个服务的 Trace 怎么设计？

这是生产排障的核心能力。

> 最佳实践是：**每个用户 turn 一条根 Trace，所有 LLM 调用、检索、工具调用、异步任务都挂到这条链上。**

---

## 1）建议采用标准

推荐：

-   **OpenTelemetry**
    
-   **W3C Trace Context**：`traceparent` / `tracestate`
    
-   同时把业务上下文放进 `baggage`
    

这样 HTTP、gRPC、MQ 都能统一透传。

---

## 2）Trace ID 怎么设计？

不要只用一个 request\_id，建议至少区分：

-   **trace\_id**：单次请求全链路唯一
    
-   **span\_id**：每个子步骤唯一
    
-   **session\_id**：整段会话唯一
    
-   **turn\_id**：当前轮次唯一
    
-   **tool\_call\_id**：单次工具调用唯一
    
-   **plan\_id**：本轮规划版本
    

注意：

-   **一个用户 turn = 一个 root trace**
    
-   多轮会话不要共用一个 trace，用 `session_id` 关联即可
    

---

## 3）推荐的 Span 层级

```text
agent.request (root)
├─ auth.check
├─ session.load
├─ memory.read
├─ planner.llm
├─ retrieval.pipeline
│  ├─ query.rewrite
│  ├─ vector.search
│  ├─ bm25.search
│  └─ rerank
├─ tool.getRecentOrders
├─ tool.getLogisticsStatus
├─ reflector.llm
├─ answer.llm
└─ memory.write
```

如果有并行步骤，就并列 span。

---

## 4）每个 Span 建议打哪些属性

## 通用属性

-   tenant\_id
    
-   scene
    
-   app\_id
    
-   user\_id\_hash
    
-   session\_id
    
-   turn\_id
    
-   prompt\_version
    
-   workflow\_version
    

## 模型相关

-   model\_name
    
-   temperature
    
-   input\_tokens
    
-   output\_tokens
    
-   latency\_ms
    
-   finish\_reason
    
-   cache\_hit
    

## 检索相关

-   index\_name
    
-   query\_rewrite\_version
    
-   topK
    
-   hit\_count
    
-   rerank\_model
    
-   doc\_ids
    

## 工具相关

-   tool\_name
    
-   tool\_version
    
-   args\_hash
    
-   timeout\_ms
    
-   retry\_count
    
-   error\_code
    
-   side\_effect=true/false
    

---

## 5）跨多个服务怎么透传？

### 同步调用：HTTP / gRPC

把以下 header 透传下去：

-   `traceparent`
    
-   `tracestate`
    
-   `baggage`
    

所有服务统一接入 OTel SDK，中间件自动创建 child span。

### 异步调用：MQ / 事件总线

把 trace context 放在 message headers 里：

-   producer span 记录消息发送
    
-   consumer 收到后继续 trace
    
-   如果是异步分叉任务，可用 **Span Links** 关联，而不是强行父子关系
    

这是很多团队容易漏掉的点。

---

## 6）日志、指标、Trace 要打通

单有 Trace 不够，必须三件套关联：

-   日志里带 `trace_id` / `span_id`
    
-   指标支持 trace exemplar
    
-   回放平台能按 `trace_id` 拉出：
    
    -   prompt 版本
        
    -   检索 doc\_ids
        
    -   工具结果
        
    -   最终回答
        
    -   错误码
        

这样一次投诉/故障能快速复盘。

---

## 7）Agent 特有的追踪重点

Agent 跟普通 RPC 不一样，建议额外记录：

-   规划输出摘要
    
-   当前 step 序号
    
-   是否重规划
    
-   是否触发反思
    
-   是否进入 fallback
    
-   同工具重复调用次数
    
-   最终停止原因<br>（成功 / 超时 / 超预算 / 需人工 / 工具失败）
    

---

## 8）隐私与合规

Trace 里不要直接裸存敏感数据：

-   用户名、手机号、地址做脱敏或 hash
    
-   工具参数记录 hash 或白名单字段
    
-   prompt / 检索文本按分级存储
    
-   高敏数据默认不入 trace 明文
    

---

## 9）采样策略

全量采样往往太贵，建议：

-   正常流量低比例采样
    
-   错误请求 100% 采样
    
-   高延迟请求 100% 采样
    
-   核心业务链路高比例采样
    
-   灰度版本临时提高采样率
    

---

## 10）一句话方案

> **用 OTel 统一 Trace 标准，以“单轮请求”为 root trace，覆盖 LLM、检索、工具、异步任务，并和日志/指标/回放平台打通。**
