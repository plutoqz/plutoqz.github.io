## 2）核心组件职责

### A. 接入层

-   API Gateway：鉴权、限流、灰度、Trace 注入
    
-   Agent Gateway：会话管理、上下文拼装、请求预算控制
    

### B. Agent Runtime

-   **Planner**：把用户目标拆成可执行步骤
    
-   **Executor**：调用工具/检索/子模型
    
-   **Reflector**：检查结果是否满足目标，是否要重试/改计划/追问用户
    
-   **Policy Guardrails**：权限、敏感词、注入攻击、防误操作
    

### C. 模型层

-   大模型不止一个：
    
    -   小模型：意图识别、工具路由、Query Rewrite
        
    -   大模型：复杂规划、最终回答
        
    -   Embedding/Rerank 模型：检索专用
        

### D. 记忆与知识层

-   会话记忆：最近几轮对话
    
-   用户长期记忆：偏好、尺码、价格带
    
-   事件记忆：历史已解决问题
    
-   企业知识：规则、SOP、FAQ、商品知识
    

### E. 工具层

-   商品、订单、物流、优惠、工单、售后、CRM、风控等内部服务
    
-   所有工具都要有：
    
    -   严格 Schema
        
    -   权限
        
    -   超时
        
    -   幂等
        
    -   审计
        

### F. 观测与治理

-   监控、告警、Trace、Prompt 版本、工具版本、回放评测、A/B

## 3）数据流转

一个典型请求的数据流是：

1.  **用户发起请求**
    
    -   例：“我上周买的运动鞋怎么还没到？能不能催发货？”
        
2.  **接入层处理**
    
    -   鉴权、拿 buyer\_id、注入 trace\_id、检查限流
        
3.  **构建上下文**
    
    -   当前问题
        
    -   最近几轮对话摘要
        
    -   用户身份、订单上下文
        
    -   预算：最多调用几个工具、总超时多少
        
4.  **Planner 生成执行计划**
    
    -   先定位订单
        
    -   再查物流
        
    -   判断是否支持催发货
        
    -   最后组织回复
        
5.  **Executor 执行**
    
    -   调 `getRecentOrders`
        
    -   调 `getLogisticsStatus`
        
    -   调 `checkDeliveryExpediteEligibility`
        
6.  **Reflector 反思**
    
    -   订单是否定位成功？
        
    -   物流结果是否足够回答？
        
    -   是否需要追问用户“是哪一单”？
        
7.  **生成最终回复**
    
    -   输出自然语言
        
    -   可附证据/卡片
        
    -   如涉及动作型操作，走确认流程
        
8.  **异步写回**
    
    -   记录会话摘要
        
    -   写入长期记忆（如用户偏好）
        
    -   写日志、Trace、评测样本

# Q2：Agent 的“规划-执行-反思”闭环如何实现？

生产级不要靠“纯 prompt 瞎循环”，而要靠**状态机 / DAG 工作流**来实现。

---

## 1）闭环的本质

### 规划 Plan

回答两个问题：

-   目标是什么？
    
-   用哪些步骤完成？
    

输出尽量是**结构化计划**，不要只是一段自然语言。

示例：

```json
{
  "goal": "回答订单物流问题，并判断是否可催发货",
  "steps": [
    {"id": "s1", "tool": "getRecentOrders", "success": "定位到候选订单"},
    {"id": "s2", "tool": "getLogisticsStatus", "depends_on": ["s1"]},
    {"id": "s3", "tool": "checkDeliveryExpediteEligibility", "depends_on": ["s1"]}
  ],
  "stop_conditions": ["信息足够回答", "需要用户补充", "触发人工兜底"],
  "budget": {"max_steps": 5, "deadline_ms": 4000}
}
```

---

## 2）执行 Execute

Executor 不直接“信模型”，而是：

-   按 plan 调工具
    
-   做参数校验
    
-   做权限校验
    
-   做超时/重试/幂等控制
    
-   记录每一步结果到 working memory
    

---

## 3）反思 Reflect

反思不是让模型“再想想”，而是做**结果评估**：

-   是否完成目标？
    
-   结果是否可信？
    
-   有没有缺失关键字段？
    
-   是否选错工具？
    
-   要不要重规划？
    
-   要不要追问用户？
    

可以用三类方式：

1.  **规则校验**：字段缺失、业务码错误
    
2.  **模型校验**：结果是否回答了原问题
    
3.  **业务校验**：例如退款前必须先校验 eligibility
    

---

## 4）推荐实现方式：有限状态机 FSM

```text
RECEIVE
  ↓
UNDERSTAND
  ↓
PLAN
  ↓
EXECUTE_STEP
  ↓
CHECK_RESULT
  ├─ 成功 → FINISH
  ├─ 信息不足 → ASK_USER
  ├─ 工具失败 → REPLAN
  ├─ 风险高 → HUMAN_REVIEW
  └─ 超预算/超时 → SAFE_FALLBACK
```

---

## 5）伪代码

```python
state = init(request, deadline, max_steps)

while not state.done:
    if state.need_plan:
        state.plan = planner(state.context)

    step = next_executable_step(state.plan, state.results)
    if not step:
        check = critic(state)
        if check.done:
            return finalize(state)
        elif check.need_user:
            return ask_user(check.question)
        else:
            state = replan(state, check.feedback)
            continue

    result = execute_tool(step)

    if not validate(result):
        state.errors.append(result.error)
        if repeated_failure(step):
            ban_tool_for_request(step.tool)
        state = replan(state, "tool_failed_or_invalid")
        continue

    state.results.append(result)

return safe_fallback()
```

---

## 6）生产实践建议

-   简单问题不一定走完整闭环，避免成本过高
    
-   对高风险动作才启用“反思 + 审批”
    
-   设置上限：
    
    -   max\_steps
        
    -   max\_tool\_calls
        
    -   max\_same\_tool\_repeat
        
    -   max\_total\_tokens
        
    -   max\_deadline

# Q3：多轮对话中，记忆如何分级管理？缓存策略怎么设计？

先讲一句最重要的：

> **Memory 是“未来还要用的事实”，Cache 是“为了加速可以丢掉的计算结果”。**

---

## 1）记忆分级

| 层级  | 内容  | 存储  | 生命周期 | 例子  |
| --- | --- | --- | --- | --- |
| L0 Working Memory | 当前请求临时变量、工具结果 | 内存/Redis | 秒~分钟 | 本次命中的订单号 |
| L1 Session Memory | 当前会话摘要、最近几轮槽位 | Redis/DocDB | 分钟~天 | “用户在问物流问题” |
| L2 Profile Memory | 稳定用户偏好 | MySQL/KV/Feature Store | 周~月 | 喜欢耐克、尺码42 |
| L3 Episodic Memory | 过去发生过的事件与解决方案 | Vector DB + 元数据 | 月~年 | 历史售后案例 |
| L4 Semantic Memory | 企业知识库/规则库/商品知识 | Vector DB + ES | 持久化 | 平台规则、活动说明 |

---

## 2）建议的记忆写入策略

不是所有内容都写入长期记忆，要有**准入机制**：

适合写入长期记忆：

-   用户明确表达且相对稳定的偏好
    
-   已验证的结构化事实
    
-   关键事件节点
    
-   常复用的历史处理结果
    

不适合写入：

-   瞬时情绪
    
-   未确认猜测
    
-   高敏感内容
    
-   高频变化事实（如库存、价格）
    

---

## 3）缓存设计

### A. L1 本地缓存

-   进程内 LRU
    
-   缓 prompt 模板、工具 schema、短时热数据
    

### B. L2 Redis

-   会话摘要缓存
    
-   工具结果缓存
    
-   检索结果缓存
    
-   语义缓存
    

### C. L3 持久层

-   MySQL / PostgreSQL / Document DB
    
-   Vector DB
    
-   对象存储
    

---

## 4）典型缓存类型

### ① Prompt Cache

-   相同前缀 prompt 命中
    
-   用于系统提示词、固定模板
    

### ② Embedding Cache

-   文本 hash → embedding
    
-   对重复 chunk 很有效
    

### ③ Retrieval Cache

-   query\_norm + filter → topK docs
    
-   对 FAQ、规则问答很有效
    

### ④ Tool Result Cache

按业务波动性设 TTL：

-   商品属性：30 min ~ 24 h
    
-   店铺信息：10 min ~ 1 h
    
-   优惠资格：30 s ~ 5 min
    
-   物流状态：10 s ~ 1 min
    
-   库存/价格：谨慎，极短 TTL 或不用缓存
    

### ⑤ Semantic Cache

-   “相似问题 → 相似答案”
    
-   适合标准化问答，不适合强个性化/实时场景
    

---

## 5）缓存失效策略

-   **TTL**：最基础
    
-   **事件驱动失效**：库存变更、价格变更、规则发布
    
-   **版本失效**：prompt 版本、模型版本、索引版本更新
    
-   **租户/用户隔离**：不能串数据
    

---

## 6）淘天场景建议

-   **实时事实**（库存、价格、优惠、订单状态）不要只靠记忆/向量库，必须从实时工具取
    
-   **长期偏好**和**历史案例**适合做记忆
    
-   会话摘要建议“每 N 轮压缩一次”，不要全量塞上下文

# Q4：向量数据库选型对比（Milvus / Pinecone / Chroma），淘天场景怎么选？

---

## 1）对比结论先说

-   **Milvus**：更适合大规模、可控、私有化、国产/自建生产环境
    
-   **Pinecone**：更适合快速上线、托管式、运维少
    
-   **Chroma**：更适合本地开发、POC、小规模场景
    

---

## 2）对比表

| 维度  | Milvus | Pinecone | Chroma |
| --- | --- | --- | --- |
| 部署方式 | 自建/私有化/云上自管 | 全托管 SaaS | 本地/轻量 |
| 运维成本 | 中高  | 低   | 很低  |
| 大规模能力 | 强   | 强   | 弱   |
| 可控性 | 很强  | 中   | 中   |
| 合规/数据主权 | 强   | 取决于云区域 | 一般  |
| 成本结构 | 前期运维高，规模化后可控 | 使用方便，但大规模成本可能高 | 低   |
| 适合场景 | 企业生产级、海量知识 | 快速验证、国际化业务 | 开发测试、Demo |

---

## 3）淘天场景怎么选？

### 主结论：

**优先选 Milvus（或同类可私有化向量库）+ Elasticsearch/OpenSearch 做混合检索。**

原因：

1.  **数据规模大**
    
    -   商品、规则、客服知识、商家知识量级都很大
        
2.  **过滤条件复杂**
    
    -   类目、品牌、店铺、区域、时间、租户等 metadata filtering 很重要
        
3.  **合规和成本控制要求高**
    
    -   关键业务更适合自控
        
4.  **需要和现有数据体系深度融合**
    
    -   ES、MQ、数仓、画像系统、业务中台
        

---

## 4）不要把向量库当万能数据库

在淘天场景里：

-   **价格、库存、优惠、订单状态**：走实时服务
    
-   **规则文档、知识文本、历史案例**：适合进向量库
    
-   **精确过滤/关键词召回**：ES/BM25 更强
    
-   **最佳实践**：Hybrid Search = 向量召回 + 关键词召回 + 过滤 + Rerank
    

---

## 5）我的推荐

### 生产环境

-   **Milvus + ES/OpenSearch + Reranker**
    
-   用户个性化小记忆可搭配 Redis / PostgreSQL
    

### 海外或快速 PoC

-   时间紧、团队小：Pinecone
    

### 本地研发

-   Chroma 即可

# Q5：Function Calling 的可靠性如何保障？如果模型一直调用同一个错误工具怎么办？

这是 Agent 落地的核心难点。

---

## 1）可靠性要分层做

### A. 调用前

-   严格 JSON Schema
    
-   参数枚举、范围、必填校验
    
-   权限校验
    
-   工具白名单
    
-   Tool version 固定
    

### B. 调用中

-   超时
    
-   重试（仅可重试错误）
    
-   熔断
    
-   幂等 key
    
-   降级 fallback
    

### C. 调用后

-   返回值校验
    
-   业务码校验
    
-   语义一致性校验
    
-   审计留痕
    

---

## 2）工具返回值要标准化

强烈建议每个工具都返回统一结构：

```json
{
  "status": "SUCCESS | RETRYABLE_ERROR | NON_RETRYABLE_ERROR | VALIDATION_ERROR | PERMISSION_DENIED",
  "data": {},
  "error_code": "",
  "error_message": "",
  "retry_after_ms": 0
}
```

这样 Orchestrator 才知道：

-   是该重试
    
-   该换工具
    
-   还是该问用户
    

---

## 3）如果模型反复调用同一个错误工具，怎么处理？

要做**反循环机制**。

### 检测规则

出现以下任一情况就触发：

-   同一工具 + 同一参数 hash 连续调用 ≥ 2 次
    
-   同一工具 3 次都返回相同错误码
    
-   同一步骤没有信息增量
    
-   调用次数超过预算
    

### 处理策略

1.  **临时封禁该工具（仅本请求）**
    
2.  把错误摘要注入 reflection context
    
3.  强制 replan，并提示：
    
    -   这个工具不能再用
        
    -   可选替代工具有哪些
        
4.  如果仍不确定：
    
    -   追问用户
        
    -   或转人工
        

---

## 4）推荐的防呆机制

-   `max_tool_calls`
    
-   `max_same_tool_repeat`
    
-   `max_steps`
    
-   `deadline_ms`
    
-   `max_side_effect_tools`
    
-   `tool_call_budget_by_scene`
    

---

## 5）高风险动作要“两阶段执行”

比如退款、改地址、发券：

### 阶段1：提议

模型输出：

-   准备调用哪个工具
    
-   参数是什么
    
-   为什么可以调用
    

### 阶段2：确认

-   规则引擎校验
    
-   必要时用户确认 / 人工审批
    
-   再真正执行
    

这样可以大幅降低误操作。

# Q6：工具的 description 怎么写才能让模型准确调用？给出淘天场景示例

一句话原则：

> description 不是写给人看的产品文案，而是写给模型做路由决策的“判别说明书”。

---

## 1）一个好 description 要回答 6 个问题

1.  **这个工具什么时候用**
    
2.  **什么时候不要用**
    
3.  **调用前提是什么**
    
4.  **参数的真实含义是什么**
    
5.  **返回什么结果**
    
6.  **有没有副作用**
    

---

## 2）推荐模板

```text
工具名：
用途：
Use when：
Do NOT use when：
Preconditions：
参数说明：
返回字段：
副作用：
示例：
```

---

## 3）淘天示例 1：订单查询工具

### 差的写法

```text
查询订单信息
```

问题：太泛，模型不知道什么时候该用、能查什么、不能查什么。

### 好的写法

```text
工具名：getBuyerRecentOrders

用途：
查询当前登录买家近90天内的订单列表，用于定位“我买的东西还没到”“我要退款的是哪一单”
这类订单、物流、售后相关问题。

Use when：
- 用户明确提到订单、物流、退款、催发货
- 用户说“我上周买的那件衣服”“最近买的鞋子”

Do NOT use when：
- 商品导购、商品搜索、比价
- 商家后台订单查询
- 用户未登录时

Preconditions：
- 必须有当前登录态 buyer_id
- buyer_id 必须来自鉴权上下文，不能由模型猜测

参数说明：
- buyer_id: 当前登录用户ID，必填
- time_range: 时间范围，可选，如“last_30_days”
- keyword: 商品名关键词，可选，用于缩小订单范围
- status: 订单状态，可选，枚举 [all, paid, shipped, completed, refund]

返回字段：
- order_id
- item_title
- create_time
- order_status
- logistics_status

副作用：
- 无副作用，只读
```

## 4）淘天示例 2：退款申请工具

```text
工具名：createRefundApplication

用途：
为指定订单商品创建退款/退货申请。

Use when：
- 用户明确表达“我要退款”“我要退货”
- 已经明确具体订单 order_id 和商品 item_id
- 已通过 getRefundEligibility 确认可申请

Do NOT use when：
- 用户只是查询退款进度
- 用户只是抱怨，还没明确说要退款
- 未确定订单/商品
- 未做 eligibility 检查

Preconditions：
- 必须先调用 getRefundEligibility
- 必须有用户身份和订单归属校验
- 高风险场景需二次确认

参数说明：
- order_id: 订单ID，必填
- item_id: 子订单或商品项ID，必填
- reason_code: 退款原因，必填，枚举值
- refund_amount: 申请金额，选填
- need_return_goods: 是否退货，必填

返回字段：
- refund_id
- application_status
- next_action

副作用：
- 会创建真实退款申请，属于写操作
```

-   写清前置依赖
